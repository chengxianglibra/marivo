# Marivo 项目 Proposal

## 一句话结论

建议立项 Marivo，建设面向运营、策略、BI、算法等团队的分析智能体基础能力。Marivo 的定位不是“再做一个 BI 工具”，而是成为 **agentic data analysis 的 harness 框架**：把高频、重复、链路长的分析工作流程标准化、自动化，把人从低效取数、拼口径、反复验证中释放出来，提升分析效率，降低单位分析成本。

它的核心设计哲学是：**智能伴随幻觉，约束产生价值**。也就是说，Marivo 不追求放任式生成，而是通过语义层、分析步骤约束、证据链和治理规则，把智能能力放进可控、可复核、可复用的框架里，让结果真正能用于业务决策。

## 为什么要做

当前业务分析普遍存在三个问题：

1. **分析链路长**：一个完整分析任务往往要经过找表、对口径、写查询、反复切片、解释结论多个环节，依赖个人经验，响应慢。
2. **人力消耗高**：运营、策略、BI、算法等团队会反复做相似问题的分析，重复劳动多，资深分析资源容易被低价值工作占满。
3. **结论复用弱**：同类问题经常重新做一遍，分析过程、证据、结论和建议没有被沉淀成可复用资产。

因此，Marivo 的目标不是替代分析师，而是把标准化分析流程产品化：让系统能围绕业务问题自动完成一部分“找数据—做分析—给证据—出结论”的工作，把人力集中到判断和决策上。

更重要的是，这件事适合现在做：

1. **业务复杂度持续上升**，跨团队、跨指标、跨口径的分析需求越来越多，人工模式边际成本持续增加。
2. **数据与基础设施条件已基本具备**，当前更大的瓶颈不是“拿不到数据”，而是“无法高效把数据转成稳定结论”。
3. **大模型与分析编排能力进入可用阶段**，现在具备把部分标准分析流程工程化、产品化的现实条件。

## 为什么可以做

现在推进 Marivo 具备现实基础，原因有三点：

1. **问题真实且高频**：分析效率低已经是运营、策略、BI、算法等团队的共性痛点，需求明确，不需要教育市场。
2. **适合边做边验证**：可以从高频场景切入，用 Demo 和试点快速验证价值，不必一次性押注大而全平台。
3. **技术路径清晰**：Marivo 当前已经具备可演进的雏形，方向明确：
    - 以 **HTTP 服务** 为核心，便于系统集成和后续接入；
    - 以 **语义层 + typed analysis steps** 管理分析动作，而不是简单 text-to-SQL，便于控质量、控口径、控边界；
    - 以 **deterministic evidence packaging** 沉淀证据、结论和建议，便于复盘、复用和治理；
    - 已有 **Demo 级能力基础**，说明从 0 到 1 可行，接下来重点是围绕场景打磨闭环。

从能力上看，Marivo 已经具备作为 harness 框架继续演进的核心骨架：

- **会话化分析流程**：围绕具体业务问题组织分析过程，而不是一次性问答；
- **语义层与指标抽象**：把数据对象、指标、口径和治理规则沉淀为统一分析入口；
- **受约束的分析步骤编排**：通过标准化 step 组合观察、对比、拆解、归因、验证等动作，降低幻觉和随意性；
- **证据化结果输出**：沉淀 findings、assessment、proposal 等结构化结果，支持复核、解释和复用；
- **API + UI 双入口**：便于对接系统，也便于面向业务做 Demo、试点和后续推广。

换句话说，这不是“从零探索一个模糊概念”，而是在真实问题、明确场景和已有技术雏形上，持续把可用性做出来。

同时，Marivo 与现有 BI 或通用 AI 工具的差异也很明确：**它不只是生成图表、SQL 或文字回答，而是面向业务分析闭环，输出可追溯的证据链、判断和建议**，更适合承担组织级分析提效任务。

## 做成什么

Marivo 建议定位为公司的**分析智能体基础平台**，面向典型业务问题，逐步形成以下能力：

1. **理解问题**：将业务问题转成标准化分析意图和步骤，而不是依赖人工逐条拆解。
2. **连接数据**：统一接入数据源、指标语义和治理规则，降低找数、对口径成本。
3. **执行分析**：围绕观察、对比、拆解、归因、验证等分析动作自动运行。
4. **产出证据化结论**：输出的不只是图表或 SQL，而是带证据链、置信度和建议的分析结果。
5. **沉淀复用资产**：把场景、规则、分析路径和结论沉淀成平台能力，形成复利。

## 路径计划

建议按 6 个月完整产品路线分三阶段推进：

| 阶段 | 时间 | 目标 | 关键动作 | 阶段产出 |
| --- | --- | --- | --- | --- |
| 第一阶段：能力打底 | 第 1-2 个月 | 完成基础产品闭环 | 打通会话、语义层、分析步骤、证据沉淀、基础 UI 和 API | 可演示、可跑通的产品雏形 |
| 第二阶段：场景化落地 | 第 3-4 个月 | 在高频分析场景中验证价值 | 选择 2-3 个典型场景，如经营异动分析、指标归因、实验/策略验证，形成稳定 workflow，并与 1 个业务团队联合试点 | 可试点、可被业务使用的场景方案 |
| 第三阶段：平台化扩展 | 第 5-6 个月 | 从点状能力变成通用平台 | 扩展更多数据源、分析模板、治理规则和组织接入方式，形成可复制接入模式 | 可复制、可规模化推广的平台能力 |

推进原则：

- **先解决高频刚需场景，再扩平台边界**；
- **先把“能稳定替代一部分重复分析”做出来，再追求通用智能**；
- **每一阶段都要有明确业务验证，不做只停留在技术演示的系统**。

## 收益目标

Marivo 的核心收益可以归纳为四个方面：

1. **提升分析效率**：缩短从提出问题到拿到初步结论的时间，提高业务响应速度。
2. **降低人力成本**：减少重复取数、重复分析、重复解释，让高价值人才投入更关键的问题。
3. **提高分析一致性**：通过统一语义、流程和证据表达，降低口径不一致和分析质量波动。
4. **形成平台资产**：把一次次分析沉淀为长期可复用的能力，后续边际成本持续下降。

从管理视角看，Marivo 的价值不只是“提高一次分析效率”，而是逐步建立公司在数据分析自动化上的基础设施能力。

建议将收益目标进一步量化为以下三个指标：

1. **被业务接受的智能分析结果交付时间显著缩短**：在高频覆盖场景下，目标将“从提出问题到业务接受初步分析结果”的时间缩短 **50%-80%**。
2. **重复分析工作量下降**：让高频标准问题尽量由系统先完成首轮分析，目标使相关重复人工分析工作量下降 **30%-50%**。
3. **场景复用率提升**：试点阶段沉淀的分析流程、语义定义和证据模板，在后续同类问题中的复用率达到 **50% 以上**。

## 资源与立项建议

建议将 Marivo 作为一个**可分阶段投入、可持续验证收益**的项目推进，而不是一次性重投入的大平台项目。

立项建议如下：

1. **先明确高频试点场景**，以业务价值牵引产品建设；
2. **采用 2-3 人轻量小组推进**，建议由后端/平台、算法或智能体能力、产品或分析侧协同组成，避免项目过重启动；
3. **以“效率提升、成本下降、复用率提升”作为核心评估指标**，持续判断投入产出；
4. **在试点跑通后再扩大覆盖范围**，降低试错成本，提高成功概率。

建议的最小资源配置为：

- **2-3 人核心小组，持续 6 个月推进**；
- **优先与运营或策略团队中的 1 个高频分析团队联合试点**，确保问题定义和效果验证来自真实场景；
- **管理层按阶段评审**，每完成一阶段再决定是否扩大投入。

预算上，建议以**现有数据基础设施复用 + 少量新增模型/API 与算力成本**为主，先用轻量投入验证 ROI，再决定是否追加资源。

## 风险与退出机制

为避免项目变成长期技术探索，建议一开始就明确风险和退出条件：

1. **主要风险**：业务场景不聚焦、数据口径不稳定、结果可信度不足、试点团队参与度不够。
2. **控制方式**：只聚焦高频标准场景；试点阶段必须与业务 owner 共建；结果输出必须强调证据链和可解释性。
3. **退出条件**：若在第二阶段结束后，仍无法在试点场景中显著缩短分析交付时间，或无法获得业务团队持续使用，则应收缩范围或暂停平台化投入。

## 结论

Marivo 值得做，因为它对应的是一个真实、持续、跨团队存在的效率问题；Marivo 也可以做，因为问题场景清晰、技术路径明确、且适合边建设边验证。若推进得当，它将不仅提升单次分析效率，更能沉淀出公司长期可复用的分析智能能力。
