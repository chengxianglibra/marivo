# Factum：面向 Agentic Data Analysis 的 Harness 框架

> 智能伴随幻觉，约束产生价值。

Factum 面向的不是“让模型更自由地访问数据”，而是“让 agent 能更可靠地完成数据分析”。它不是一个 BI 工具，也不是一个 text-to-SQL 包装器，更不是试图替代 agent 的黑盒分析脑。Factum 的定位更接近一个 **analysis runtime / harness**：为 agent 提供稳定语义、受约束动作、结构化证据和治理边界，让分析过程从一次性生成走向可持续推进、可复核、可治理的系统能力。

## 为什么 Agentic Data Analysis 还需要 Harness

今天很多 agentic data analysis 系统已经能回答问题、生成 SQL、访问表、绘制图表，但它们距离可靠的分析系统仍有明显距离。问题通常不在模型本身，而在系统边界太松。

常见路线往往会走向三个方向：要么把问题退化成 text-to-SQL，要么把分析退化成 chat 式问答，要么把一串工具调用包装成“自动分析”。这些路径可以做出不错的 demo，却很难稳定走向生产，因为它们缺少统一的语义入口、有限而明确的分析动作、可持续推进的状态，以及证据优先的结果表达。

分析任务和开放式对话不同。它要求口径稳定、过程可追溯、结论可复核、行为可治理。如果系统把“怎么理解问题”“怎么执行分析”“什么算证据”“哪些动作被允许”全部交给模型即兴决定，结果往往看起来聪明，但难以信任，也难以复用。

## Factum 的核心主张

Factum 的核心主张是：**智能伴随幻觉，约束产生价值**。

这句话的含义并不是否定模型，而是承认模型的价值和边界同时存在。Agent 擅长理解问题、提出假设、决定下一步、解释业务含义；但它并不适合在没有约束的情况下自由定义分析语义、执行边界和证据结构。Factum 的设计目标，就是把这些高风险、高重复、需要稳定性的部分，收束成一个可验证的运行时。

Factum does not replace the agent. It gives the agent a constrained, evidence-first analysis runtime.

## Factum 与 Agent 的分工边界

这层分工如果不说清楚，Factum 很容易被误解成“更复杂的工具箱”，或者“替 agent 做全部分析的系统”。这两种理解都不准确。

Agent 的职责是：

- 理解业务问题，判断当前真正要分析什么。
- 选择分析路径，决定下一步该观察、比较、拆解还是验证。
- 解释结果的业务含义，而不是只停留在数值层。
- 基于当前证据决定是否继续探索、停止，或转向新的假设。

Factum 的职责是：

- 提供稳定的语义入口，让 agent 操作业务对象而不是裸表和临时字段。
- 提供受约束的分析动作，让执行边界清晰，而不是开放式拼装。
- 提供结构化证据表达，让结果能被引用、继承、复核，而不是只返回一段文本总结。
- 提供 policy 和 governance 机制，让分析行为具备生产可用性。

一句话说：

**Agent decides. Factum grounds.**

Agent 负责“想清楚问什么、下一步做什么”，Factum 负责“把这些动作变成受约束、可验证、可治理的分析过程”。

## Factum 如何赋能 Agent 做数据分析

Factum 的价值，不在于替 agent 思考，而在于让 agent 的分析能力从脆弱的通用生成，升级为稳定的系统能力。

第一，Factum 给 agent 提供稳定的分析语法，而不是开放式工具调用。Agent 不需要每次都从 SQL、表结构、临时逻辑重新拼装分析过程，而是可以围绕明确的分析动作工作，把推理预算留给路径选择和结果解释。

第二，Factum 给 agent 提供统一的业务语义，而不是裸数据接口。Agent 操作的是 metric、dimension、entity、time 这样的语义对象，而不是数据库偶然长成的物理形状。这会显著降低口径漂移和上下文负担。

第三，Factum 给 agent 提供可持续推进的状态，而不是一次性问答。真实分析不是一轮对话，而是一个逐步收敛的过程。Factum 让 agent 可以在已有分析状态和证据基础上继续工作，而不必依赖冗长对话来硬撑上下文。

第四，Factum 给 agent 提供证据结构，而不是只返回结果文本。结果可以被后续步骤引用，可以被系统复核，也可以被治理规则约束。这使得分析过程更接近工程系统，而不是一次性的回答生成。

第五，Factum 给 agent 提供生产级治理边界。很多 agent 原型在 demo 阶段表现不错，但一进入真实环境，就会碰到权限、质量、一致性和风险控制问题。Factum 把这些约束前置到运行时，而不是留给人工兜底。

Factum 不是通过增加生成自由度让 agent 更聪明，而是通过提供结构、状态、证据和治理，让 agent 真正具备可用的数据分析能力。

## Factum 如何把约束变成能力

Factum 的约束不是“限制模型”，而是在为可靠分析建立骨架。

`Semantic Layer` 解决的是“分析对象是什么”。它把业务语义从底层物理结构中抽离出来，让分析围绕稳定定义展开，而不是围绕临时 SQL 约定展开。

`Typed Intent` 解决的是“允许做什么分析动作”。它把分析动作收束成有限、明确、可组合的类型化步骤，让 agent 在一个清晰语法中工作，而不是无限制地发明执行路径。

`Evidence Engine` 解决的是“结果如何成为证据”。它要求事实提取和证据组织尽可能确定、可引用、可审计，让后续判断建立在结构化证据上，而不是建立在模型对文本结果的印象上。

`Policy / Governance` 解决的是“系统如何在真实环境中可用”。它把质量、权限、边界和风险控制纳入主路径，使 Factum 不只是分析能力的容器，也是分析纪律的执行框架。

## Analysis Session：Factum 的核心是 Analysis Runtime

Factum 真正重要的，不是这些组件分别存在，而是它们通过 `Analysis Session` 被组织成一个连续工作的运行时。

在 Factum 中，分析不是一次性的 request-response，而是一个 stateful analysis 过程。Session 承载任务目标、分析上下文、阶段性结果和后续动作的连接关系。Semantic Layer 提供稳定入口，Typed Intent 提供动作语法，Evidence Engine 负责沉淀证据，Governance 负责控制边界，而 Session 负责把它们串成一条可以持续推进的链路。

这也是 Factum 和很多“带数据工具的 agent”最本质的差异之一。它不只是让 agent 能访问数据，而是让 agent 能在一个有状态、有证据、有约束的运行时中完成分析。

## Factum 与常见路线的差异

和 text-to-SQL 相比，Factum 关注的不是把自然语言翻译成查询，而是把分析任务变成受约束、可复核的运行过程。SQL 可能是底层实现细节，但不应该是外部契约。

和 BI copilot 相比，Factum 关注的不是辅助查数、解释图表或生成 dashboard，而是完整的分析链路，包括动作边界、状态延续、证据组织和治理控制。

和 chat-style analyst agent 相比，Factum 不把系统能力建立在 prompt 技巧和对话连贯性上，而是建立在稳定语义、类型化动作和证据结构上。前者更灵活，后者更可靠。

## Factum 不是什么

Factum 不是“把 LLM 接到数据仓库上”的薄封装。

它不是一个自然语言生成 SQL 的工具。

它不是一个只服务于人类交互页面的 BI 动作层。

它也不是一个试图替 agent 做所有开放式推理的黑盒 planner。

Factum 更像一个中间层：它不代替 agent 进行判断，也不放任 agent 自由执行一切。它做的是把 agent 的分析能力落到一个可复核、可治理、可复用的框架里。

## 为什么开发者应该关注 Factum

如果你在构建 agentic data analysis 系统，真正困难的通常不是“让模型会回答”，而是“让分析结果能被相信、被复用、被治理”。Factum 提供的价值，正是在这条路径上。

它让分析动作更稳定，让业务语义更统一，让结果表达更接近证据，让分析过程更适合持续推进，也让 agent 更容易从 demo 走向生产。

面向生产的 agentic data analysis，需要的不是放任式生成，而是一个能把智能收束为可靠分析过程的 harness。Factum 的价值就在这里。
