# 规则族设计检查清单

本文档整理规则族设计时必须满足的通用约束，供设计评审、规则注册表扩展与新增评估类型时复用。

状态：working checklist。本文不引入新的规范契约；若与其他文档表述冲突，以 [`evidence-engine/inference-and-gap-engine.md`](../inference-and-gap-engine.md) 为主线基线，并以 [`rule-registry-contract.md`](rule-registry-contract.md) 的 registry metadata、[`assessment-judgment-policy.md`](assessment-judgment-policy.md) 的 family-specific thresholds、[`support-oppose-and-status-resolution.md`](../support-oppose-and-status-resolution.md) 的 executable resolution contract 为补充。

## 目的

[`evidence-engine/inference-and-gap-engine.md`](../inference-and-gap-engine.md) 已经定义了规则族的命名层级、固定评估顺序、职责边界与 `InferenceRecord` 写入约束。

本文只做一件事：

- 把这些分散在契约中的要求整理成可检查的清单

本文不负责：

- 发明新的规则族
- 替代判断策略
- 替代规则注册表
- 定义实现代码组织方式

## 核心硬约束

以下条目属于规范契约的硬约束；违反其中任一条，都不应进入 v1 设计。

### 1. 分组层级必须稳定

- 规则族只能位于 `assessment_type -> rule_family -> rule_id` 三层命名中的第二层。
- 规则族必须是稳定执行阶段或稳定判断目标分组，不得是临时业务主题、实现目录或统计标签。
- 更细的业务分类只能落在规则簇或规则元数据，不能把簇伪装成新的族。

通过标准：

- 新条目可以回答”它为什么是稳定阶段分组”，而不是只回答”它服务某个细分业务”。
- 同一 `rule_id` 的族归属不依赖文件路径、类名或字符串前缀。

### 2. 输入边界必须限制在单命题重算

- 族只能读取当前 target proposition 的规范评估上下文。
- 不允许跨命题直接读取其他命题的评估、缺口或中间状态。
- 不允许把投影摘要、自由文本证据描述或临时评分对象作为族的判断输入。

通过标准：

- 该族的输入可以完全落在事实单元、先前评估、开放缺口及 proposition identity 上。
- 若设计需要跨命题判断，必须先进入独立规范关系或 finding-like 输入，而不是扩大族输入面。

### 3. 族必须能嵌入固定评估顺序

- 族必须明确属于固定顺序中的某一阶段。
- 不允许新增一个族后打破 `gate -> evidence -> resolution -> gap -> confidence -> transition` 的整体顺序。
- 同阶段可以并行实现，但族的可观察语义必须与固定顺序一致。

通过标准：

- 能明确回答该族在候选评估实体化中何时运行。
- 不会出现”先算置信度再改状态”或”先解决转换再决定支持归属”这类逆序依赖。

### 4. 职责边界必须单一且不能越权

- `support_evidence` 只能决定哪些事实单元进入 `supporting_finding_ids`。
- `oppose_evidence` 只能决定哪些事实单元进入 `opposing_finding_ids`。
- `status_resolution` 才能输出最终 `Assessment.status`。
- `gap_management` 才能把判断结果稳定映射为缺口打开 / 保持 / 解决。
- `confidence_shaping` 只能依据结构化依据维度塑形 `confidence_grade`。
- `assessment_transition` 只能解释与先前最新评估的变化，不单独决定支持 / 反驳归属。

通过标准：

- 该族的 `may_emit` 能力与其阶段职责一致。
- 没有任何非 `status_resolution` 族直接写最终状态。
- 没有任何非 `gap_management` 族以 out-of-band 方式解决或关闭缺口。

### 5. 门槛族必须保守

- `precondition_gate`、`quality_gate`、`comparability_gate` 可以打开阻塞性缺口、触发保守降级或压低置信度。
- 门槛族不得在前提未满足时直接产出高置信度的 `supported` 或 `contradicted`。
- 可比性失败应优先表现为缺口或保守降级，而不是被解释成单向强结论。

通过标准：

- 门槛命中失败时，结果仍能解释为 `miss` 或 `partial` 的结构化保守输出。
- 设计里不存在”仅凭门槛通过就建立强判断”的路径。

### 6. 族必须兼容通用映射，不得绕过公共决议面

- 每个 `assessment_type` 只能复用文档中声明的必需族。
- 可选规则簇只是在族之下细分业务规则，不是新增执行阶段。
- 新增评估类型时，必须先声明到现有规则族的映射，再进入规范契约。
- 不允许某个评估类型绕过通用 `status_resolution` 直接写评估。

通过标准：

- 新设计可以说明它复用了哪些必需族。
- 若声称需要新族，必须同时说明为什么簇不够、为什么现有阶段无法承载。

### 7. 族的贡献必须稳定实体化到 `InferenceRecord`

- 所有对当前快照的状态、缺口状态、置信度有贡献的族，都必须产出对应记录。
- 不允许只记录 `hit`，而静默丢弃对当前结论有解释价值的 `miss` 或 `partial`。
- 记录中的 `matched_conditions`、`unmatched_conditions` 必须承载稳定条件 token，不能把主要语义藏进 `notes`。
- 若某族使用 `partial`，必须明确它与 `miss` 的边界，以及下游族消费的是哪些结构化字段，而不是自由文本。
- `partial` / `miss` 的边界应优先落在”当前输入是否仍可作为该判断层的合法输入”，而不是落在文案强弱或定性影响等级上。
- 若某族使用 `partial`，必须明确 `partial` 是否总是要求实体化缺口；若允许无缺口 `partial`，也必须说明下游如何仅凭记录继续消费该警示。
- 同一族下允许多条规则在单次重算中分别产出 `partial`，但读取与审计必须仍能保留逐条 `rule_id` 的贡献边界。

通过标准：

- 该族的正向命中、失败命中、部分命中都能映射到 `hit / miss / partial`。
- 若族会打开或解决缺口，对应缺口 id 一定能在记录字段中追溯。
- 若设计里引入 `partial`，必须能回答”什么条件下是 `partial` 而不是 `miss`””partial` 是否仍是合法输入”以及”后续阶段如何消费这个 `partial`”。

### 8. 族归属必须由注册表显式解引用

- `InferenceRecord` 不新增 `rule_family` 字段。
- `rule_id -> rule_family -> assessment_type` 的归属必须由注册表明确声明。
- 不允许消费者或实现层通过字符串前缀、目录名、类名推断族。

通过标准：

- 任意 `rule_id` 都能唯一解引用到一个 `RuleRegistryEntry`。
- 状态面/上下文面的族分组可以只依赖注册表完成。

### 9. 族设计必须兼容仅变化快照策略

- 族的输出只能服务规范判断输出，不能把额外遥测伪装成快照变化。
- 只有当状态、置信度、支持/反驳归属、缺口归属、应用记录或判断相关子类型载荷变化时，才应触发新快照。
- 若规范结果不变，本次重算产生的候选与记录必须整体丢弃。
- 族设计必须能说明其规范结果至少由哪些判断相关字段组成，以及哪些非判断元数据不参与相等判断。

通过标准：

- 族不依赖”每次执行都写一条记录”来表达存在感。
- 该族的加入不会让相同规范输入稳定地产生不同快照。
- 相同规范输入重算时，不会仅因 token 顺序、`notes` 措辞、展示顺序、时间戳或其他运行时元数据变化而生成新快照。

### 10. 族设计必须允许非单调状态演化

- 族不能假设结论只会增强，不会降级。
- 新证据、质量失败、可比性失败、缺口重开或事实单元不可读，都可能触发降级。
- 状态不变但支持/缺口/置信度改变时，族仍必须支持新快照的形成。

通过标准：

- 该族的设计同时覆盖升级、降级、保持状态但改变支持三类情形。
- 不会把”曾经命中过强结论”当作当前重算的保底前提。

## 设计建议项

以下条目不是新增规范规范，但用于降低实现歧义并提高审计稳定性。

### 1. 优先新增簇，而不是新增族

- 若新增内容只是某类评估的细分判断主题，默认先落到规则簇。
- 只有当它需要新的稳定阶段职责、新的公共输出边界，才考虑新增族。

### 2. 族名称应面向判断职责，而不是算法实现

- 族名称应描述”它负责什么判断职责”，而不是”它用什么算法或指标”。
- 具体方法前提、阈值或业务主题应下沉到 `rule_id` 或规则簇。

### 3. 保持族对评估类型的复用性

- 若一个设计只适用于单一评估类型的局部技巧，优先做成簇。
- 族更适合作为多个评估类型共享的阶段语义，而不是一次性扩展点。

### 4. 让 miss 也具备可审计价值

- 设计规则时，优先让 `miss` 能解释”为什么当前还是 insufficient / 保守降级 / 缺口保持”。
- 不要把 miss 设计成纯调试噪声，否则会破坏上下文面的解释完整性。

### 5. 缺口解决语义应可审计

- 若族会解决缺口，设计里应明确”谁有权解决哪类缺口”，而不是依赖”本次没再提到旧缺口”。
- 解决语义应绑定同一命题与同一要求语义；不要让实现层临时决定是否可以跨族或跨层关闭缺口。
- 若某个缺口被解决但其他前提仍失败，设计应能说明剩余缺口、保守状态和记录如何保留。
- 若同一重算中多条规则可能指向同一缺口标识，设计应明确候选缺口归属如何按要求语义收敛；不要让结果依赖规则执行顺序或”最后一条覆盖前一条”的隐式优先级。

### 6. 稳定 token 需要可演进

- 若族依赖稳定条件 token 或等价结构化标识，应提前声明哪些变更是向后兼容，哪些属于破坏性变更。
- 破坏性变更不应复用旧 token 字面值表达新语义；已持久化记录仍应能通过版本边界被解释。
- 默认不要把 token 演进策略留给实现层或读取层临时猜测。

### 7. 要求键应表达稳定语义，而不是临时实现细节

- 若族依赖 `requirement_key` 或同类稳定要求标识，应优先表达要求语义或消费场景，而不是目录名、类名、调试编号或一次性文案。
- 阈值、算子等可独立结构化承载的参数，默认不应重复编码进 `requirement_key`，除非缺少这些限定就无法区分稳定要求语义。
- 命名格式应在 family-level 契约中给出最小约束与正反例，避免实现层各自发明不兼容写法。
- 若 `requirement_key` 还会嵌入条件 token、缺口标识或其他结构化命名中，设计里应显式约束保留字符；默认不应允许与 token 分隔符冲突的字符混入 key。

### 8. `partial` / `miss` 边界最好给出具体例子

- 若族使用 `partial`，除抽象定义外，建议至少给出 2-3 个能稳定区分 `partial` 与 `miss` 的示例场景。
- 示例应解释”为什么当前还能有限消费”或”为什么已经不能继续作为有效判断输入”，而不是只重复枚举名。
- 这样可以减少实现层把 `partial` 滥用成”弱 hit”或把 `miss` 滥用成纯调试噪声。

### 9. 若输出有序定性等级，应声明等级语义边界

- 若族会输出类似影响、严重性、覆盖之类的有序定性等级，设计中应说明各等级的语义边界与下游消费方式。
- 通用 family-level 契约可以只定义定性边界，不必在没有必要时提前固化跨评估类型的统一数值阈值。
- 若具体阈值明显依赖评估类型、规则簇或方法前提，优先下沉到配置或更细规则文档，而不是挤进通用清单。
- 若这些等级会直接影响保守降级、缺口阻塞或置信度护栏，建议至少给出少量跨等级示例，说明什么情况下是轻微警示、什么情况下已升级为实质限制。

### 10. 不要把上游契约违规伪装成族级 miss

- 若输入格式错误、上游工件违反自身规范 schema、或注册表无法稳定解引用，这类问题默认属于上游 schema / 引擎 / 注册表契约，而不是族级判断 miss。
- 族级契约应说明哪些失败属于”可被当前族正常评估并实体化到记录/缺口”，哪些属于上游契约违规，不应混入 `hit / miss / partial`。
- 不要为了兜底而把本应中止或升级处理的上游错误压成普通 `miss`，否则会污染审计语义。

## 新增规则族审查问题

提出一个新的规则族前，至少先回答以下问题：

1. 它是新的稳定执行阶段，还是只是某类规则簇？
2. 它读取的输入是否完全来自单命题的规范评估上下文？
3. 它在固定评估顺序中的具体位置是什么？
4. 它会直接决定哪些规范输出，又明确不能决定哪些输出？
5. 如果它不命中，是否能稳定产出对当前快照有解释价值的 `miss` 或 `partial` 记录？
6. 如果它使用 `partial` 或缺口解决，是否已经明确这些结果的边界、归属与下游消费方式？
7. 若多条规则指向同一缺口标识，它是否已经明确如何按要求语义收敛，而不是依赖执行顺序？
8. 它是否仍然通过注册表解引用，而不是引入新的隐式命名约定？
9. 它是否会破坏仅变化快照策略或制造额外的假变化？
10. 如果它依赖稳定 token 或要求键，版本演进时是否仍可兼容审计与重放？
11. 它是否已经区分”当前族的正常 miss/partial”与”应由上游 schema / 引擎 / 注册表处理的契约违规”？
12. 如果把它降为规则簇，是否已经足够满足设计目标？

若其中任一问题无法明确回答，默认不应新增族。

## 默认审查结论

对 v1 而言，默认结论应是：

- 优先复用现有族
- 把业务细分放进簇
- 把状态口径留在判断策略
- 把归属关系留在注册表
- 把快照写入边界留在推断引擎契约

只有在现有契约无法表达稳定执行阶段时，才讨论新增规则族。
