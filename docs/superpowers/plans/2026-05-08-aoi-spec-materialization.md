# AOI Spec Materialization Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Materialize the AOI v0.1 public spec directory with a self-contained narrative spec, one canonical JSON Schema contract, minimal valid examples, and release metadata.

**Architecture:** `aoi-spec/` is the extractable public artifact. `aoi-spec/spec.md` is the narrative source, `aoi-spec/schema/aoi.schema.json` is the single validation entry point with inline `$defs`, `aoi-spec/schema/aoi.schema.yaml` is an OSI-style readable contract view, and `aoi-spec/examples/` provides small request/artifact examples that validate against the schema. No runtime, transport, Marivo-private metadata, vendor extension, or conformance-suite files are introduced in v0.1.

**Tech Stack:** Markdown, JSON Schema 2020-12, JSON examples, `rg`, `git diff --check`, `npx --yes ajv-cli@5.0.0` for schema validation, repository git hooks.

```markdown
# AOI v0.1

AOI (Analysis Operation Interface) is a schema-only standard for analysis operation contracts. It defines how callers invoke atomic analysis intents and how implementations return portable artifacts.

## Status

Version: 0.1.0

Status: draft

AOI v0.1 defines foundation primitives, seven atomic intents, artifact contracts, and failure contracts. It does not define transport, runtime sessions, private metadata, derived intents, or a conformance test suite.

## Layout

```text
aoi-spec/
  README.md
  VERSION
  CHANGELOG.md
  spec.md
  schema/
    aoi.schema.json
    aoi.schema.yaml
  examples/
    observe/
    compare/
    decompose/```

`schema/aoi.schema.json` is the canonical validation entry point. It keeps all reusable types under `$defs` so the public schema can be copied, reviewed, and validated without resolving cross-file references. `schema/aoi.schema.yaml` is an OSI-style readable contract view with top-level enumerations and snake_case schema names.

## Validate Examples

From the repository root:

```bash
npx --yes ajv-cli@5.0.0 validate --spec=draft2020 -s aoi-spec/schema/aoi.schema.json -d "aoi-spec/examples/**/*.json"
```

The command validates every JSON example against the AOI v0.1 schema.

## Scope

Included:

- Foundation primitives: `Expression`, `TimeScope`, `TimeGranularity`, `CompareType`, `AnalysisFailure`, `Hypothesis`
- Atomic requests: `observe`, `compare`, `decompose`, `correlate`, `detect`, `test`, `forecast`
- Result schemas: `scalar_observation`, `time_series_observation`, `segmented_observation`, `scalar_delta`, `time_series_delta`, `segmented_delta`, `delta_decomposition`, `anomaly_candidates`, `association_result`, `hypothesis_test_result`, `forecast_series`

Excluded from v0.1:

- Derived intents such as `attribute`, `diagnose`, and `validate`
- Composition or DAG schemas
- Transport bindings
- Runtime sessions, evidence graphs, planning, and caching
- Vendor extensions and private metadata envelopes
- Formal conformance fixtures
```

### Task 3: Create The Canonical JSON Schema

**Files:**
- Create: `aoi-spec/schema/aoi.schema.json`

- [ ] **Step 1: Write the schema header and root union**

Create `aoi-spec/schema/aoi.schema.json` as JSON Schema 2020-12. The root must validate any AOI request or artifact:

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "$id": "https://aoi.dev/schemas/0.1.0/aoi.schema.json",
  "title": "AOI v0.1",
  "description": "Analysis Operation Interface v0.1 canonical schema.",
  "oneOf": [
    { "$ref": "#/$defs/requests/observe" },
    { "$ref": "#/$defs/requests/compare" },
    { "$ref": "#/$defs/requests/decompose" },
    { "$ref": "#/$defs/requests/correlate" },
    { "$ref": "#/$defs/requests/detect" },
    { "$ref": "#/$defs/requests/test" },
    { "$ref": "#/$defs/requests/forecast" },
    { "$ref": "#/$defs/artifacts/Artifact" },
  ],
  "$defs": {
    "primitives": {},
    "requests": {},
    "artifacts": {},
  }
}
```

Subsequent steps fill the four `$defs` objects. Keep `additionalProperties: false` on every object schema except open record maps named in the spec (`DimensionKeyMap` fields).

- [ ] **Step 2: Add primitive definitions**

Inside `$defs.primitives`, define these exact reusable schema names:

- `ISO8601`: string with `format: "date-time"`
- `Expression`: object with required `dialects`, unique dialect entries by convention, and entries `{ dialect?: string default ANSI_SQL, expression: string }`
- `TimeScope`: object with required `start`, `end`, both `$ref` to `ISO8601`
- `TimeGranularity`: enum `hour`, `day`, `week`, `month`, `quarter`, `year`
- `CompareType`: enum `normal`, `yoy`, `mom`, `wow`, `holiday_aligned_yoy`, `weekday_aligned_yoy`, `weekday_aligned_mom`, default `normal`
- `AnalysisFailure`: object with required `code` and `message`
- `Hypothesis`: object with required `family`, `alternative`, `alpha`, `label`; `family` enum `two_sample_mean`, `two_sample_proportion`, `paired_mean`; `alternative` enum `two_sided`, `greater`, `less`; `label` string or null
- `Slice`: object with required `time_scope`, `filter`; `filter` is `Expression` or null
- `NumberOrNull`: `number` or `null`
- `DimensionKeyMap`: object with string values and `additionalProperties: { "type": "string" }`

Use local `$ref` paths such as `#/$defs/primitives/TimeScope`. Do not create any cross-file `$ref`.

- [ ] **Step 3: Add request definitions**

Inside `$defs.requests`, define these seven object schemas with `additionalProperties: false`:

| Schema | Required fields | Key constraints |
|--------|-----------------|-----------------|
| `observe` | `metric`, `time_scope`, `filter` | `oneOf` enforces scalar/time-series/segmented mode: neither `granularity` nor `dimensions`, non-null `granularity` only, or non-empty `dimensions` only |
| `compare` | `left_artifact_id`, `right_artifact_id` | `compare_type` optional `$ref` `CompareType` |
| `decompose` | `compare_artifact_id`, `dimension` | `limit` optional integer minimum 1 |
| `correlate` | `left_artifact_id`, `right_artifact_id` | `method` optional enum `pearson`, `spearman` |
| `detect` | `metric`, `time_scope`, `granularity`, `filter` | `split_by` nullable array of strings; `profile` nullable string; `sensitivity` nullable number minimum 0 maximum 1; `limit` optional integer minimum 1 |
| `test` | `metric`, `left`, `right`, `kind`, `hypothesis` | `kind` enum `numeric`, `rate`; `left` and `right` use `Slice` |
| `forecast` | `source_artifact_id`, `horizon` | `horizon` integer minimum 1; `profile` nullable string |

Every request schema omits `intent` and `spec_version`; the selected request schema already identifies the intent, and AOI v0.1 is followed as a complete standard.

- [ ] **Step 4: Add common artifact envelope definitions**

Inside `$defs.artifacts`, define:

- `Artifact`: one shared envelope with `artifact_id`, `step_id`, exactly one of `result` or `failure`, and `result` as a union of the eleven result schemas.
- Artifact-specific result schemas such as `ScalarObservationResult`, `TimeSeriesDeltaResult`, and `ForecastSeriesResult`.

Do not create per-result artifact wrapper schemas. The externally visible behavior must be: every artifact has `artifact_id`, `step_id`, and exactly one of `result` or `failure`; result shape distinguishes the analytical output.

- [ ] **Step 5: Add observation artifact definitions**

Define observation result schemas matching Section 4.2.3:

- scalar result: `{ "value": number|null }`
- time series result: `{ "points": [{ "bucket_start": ISO8601, "value": number|null }] }`
- segmented result: `{ "rows": [{ "item_id": string, "keys": DimensionKeyMap, "value": number|null }] }`

- [ ] **Step 6: Add compare, decompose, correlate, detect, test, and forecast artifact definitions**

Define compare, decompose, correlate, detect, test, and forecast result schemas matching Section 4.2.3:

- delta results include `matched_time_scope` as `TimeScope` or null.
- compare step metadata includes `left_artifact_id`, `right_artifact_id`, and `compare_type`; delta artifacts do not repeat it.
- decomposition rows include `item_id`, `key`, `contribution`, and `share`.
- anomaly candidates include `item_id`, `bucket_start`, `value`, `score`, and `series_keys`.
- association result includes `coefficient`, `p_value`, `n_pairs`, and `matched_time_scope`.
- hypothesis test result includes `statistic`, `p_value`, `decision.reject_null`, and `assumption_notes`.
- forecast points include `bucket_start`, `value`, `ci_low`, and `ci_high`.

Use `additionalProperties: false` for every nested result row object.
