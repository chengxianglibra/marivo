## 3. `MarivoRuntime` Form

`MarivoRuntime` keeps a flat method API. Every method is a 1–3 line delegation to a use case module. Use case module functions take `runtime: MarivoRuntime` as their first argument (not `ports: RuntimePorts`) — they need both ports and core, and the runtime is their natural context object.

```python
class MarivoRuntime:
    def __init__(self, ports: RuntimePorts, core: CoreEngine) -> None:
        self._ports = ports
        self._core = core

    @property
    def ports(self) -> RuntimePorts:
        return self._ports

    @property
    def core(self) -> CoreEngine:
        return self._core

    # Session lifecycle
    def create_session(self, goal: str, actor: UserId) -> SessionState:
        return session.create_session(self, goal, actor)

    def list_sessions(self, owner: UserId) -> list[SessionState]:
        return session.list_sessions(self, owner)

    def get_session(self, session_id: SessionId) -> SessionState:
        return session.get_session(self, session_id)

    def terminate_session(self, session_id: SessionId, actor: UserId) -> None:
        return session.terminate_session(self, session_id, actor)

    def get_session_runtime_status(self, session_id: SessionId) -> dict:
        return session.get_session_runtime_status(self, session_id)

    # 10 intent dispatchers (delegate to intent_execution)
    def observe(self, session_id: SessionId, params: dict) -> dict:
        return intent_execution.observe(self, session_id, params)
    # ... compare / decompose / detect / correlate / forecast / test / attribute / validate / diagnose

    # Semantic helpers (delegate to semantic_ops)
    def compile_step(self, step_ir: dict, context: dict) -> dict:
        return semantic_ops.compile_step(self, step_ir, context)
    # ... resolve_metric / resolve_metric_execution_context / build_scoped_query / etc.
```

`intent_execution.{name}(runtime, session_id, params)` internally calls the corresponding `app/intents/{name}.run_*_intent(runtime.ports, session_id, params)`. Phase 4 already established these intent runners as the dispatch entry points.

## 4. Port Surface Additions

Three new methods. Two are single-session lookups; one is owner-filtered session enumeration. The port returns projected state directly — adapters are free to fold events on demand or maintain a projection index. Implementations must reuse `core/session/rebuild.rebuild_session_state()` so projection logic does not diverge.

| Port | New method | Signature | Purpose |
|------|-----------|-----------|---------|
| `SessionStore` | `list_sessions` | `(owner: UserId) -> list[SessionState]` | API `GET /sessions` |
| `StepStore` | `list_steps` | `(session_id: SessionId) -> list[Step]` | `get_session_runtime_status` |
| `ArtifactStore` | `list_artifacts` | `(session_id: SessionId) -> list[Artifact]` | `get_artifact_runtime_status` / `get_proposition_runtime_status` |

**Owner indexing invariant** (documented on `SessionEvent` and `SessionStore.list_sessions`): the `actor` of a session's `session_created` event **is** the session owner. Adapters index on `actor` for `session_created` events.

**`load_events` not-found contract** (documented on `SessionStore.load_events`): when the session does not exist, `load_events` raises `NotFoundError(ErrorCode.SESSION_NOT_FOUND, ...)` — it does **not** return an empty list. All adapters comply. 6.3 contract tests verify.

**Local adapter implementations:**

```python
# SqliteSessionStore
def list_sessions(self, owner: UserId) -> list[SessionState]:
    sids = self._db.execute(
        "SELECT DISTINCT session_id FROM session_events "
        "WHERE event_type='session_created' AND actor=:owner",
        {"owner": str(owner)},
    ).fetchall()
    return [
        rebuild_session_state(self.load_events(SessionId(sid)))
        for (sid,) in sids
    ]
```

Naive N+1 fold. Acceptable for local single-user mode; Phase 8's server adapter replaces this with a projection index.

`SqliteStepStore.list_steps` and `FileArtifactStore.list_artifacts` are direct table / directory scans by `session_id`.

## 5. Migration Inventory

The full method-by-method validation runs in the writing-plans phase (`grep` `app/service.py` and `app/session/session_manager.py` to confirm no method is missed). The tables below capture the destinations. Every method in `app/service.py` and `app/session/session_manager.py` must land in this table or be explicitly noted as deleted.

### 5.1 `_svc.run_intent` → 10 intent dispatch

All ten intent methods (`observe`, `compare`, `decompose`, `detect`, `correlate`, `forecast`, `test`, `attribute`, `validate`, `diagnose`) on `MarivoRuntime` change from `self._svc.run_intent(name, ...)` to `intent_execution.{name}(self, session_id, params)`. The `intent_execution.{name}` body is a 1-line call into `app/intents/{name}.run_*_intent(self.ports, session_id, params)`.

### 5.2 `_svc.<helper>` → 15 helpers

| Destination | Methods |
|-------------|---------|
| `runtime/semantic_ops.py` | `resolve_metric_execution_context`, `resolve_metric`, `resolve_metric_table`, `resolve_metric_dimensions`, `compile_step`, `resolve_windowed_query_time_axis`, `build_scoped_query`, `resolve_scope_constraint_column` |
| Direct `ports.artifact_store` | `commit_artifact_with_extraction`, `insert_artifact`, `resolve_artifact_for_ref`, `resolve_artifact_id_for_step` |
| Direct `ports.step_store` | `insert_step` |
| Direct `ports.data_source` | `resolve_engine_for_session`, `resolve_engine`, `discover_catalog` (uses `ports.data_source.schema()`) |

`runtime/semantic_ops.py` is a function module (not a class). Each function takes `runtime: MarivoRuntime` first, loads the model via `runtime.ports.model_store.get(...)`, and delegates to `core/semantic/*` pure functions for the actual computation.

### 5.3 `SessionManager` → `runtime/session.py`

| `SessionManager` method | `runtime/session.py` function | Notes |
|------------------------|-------------------------------|-------|
| `create_session` | `create_session(runtime, goal, actor)` | Append `session_created` event; return `SessionState` |
| `list_sessions` | `list_sessions(runtime, owner)` | One-liner: `runtime.ports.session_store.list_sessions(owner)` |
| `get_session` | `get_session(runtime, sid)` | `load_events` + `rebuild_session_state` |
| `assert_session_exists` | `_assert_exists(runtime, sid)` | Inline in callers; catches `NotFoundError` |
| `assert_session_is_open` | `_assert_open(runtime, sid)` | Rebuild + check `status == 'active'` |
| `terminate_session` | `terminate_session(runtime, sid, actor)` | `ports.authz.check(...)` then append `session_terminated` |
| `get_session_runtime_status` | `get_session_runtime_status(runtime, sid)` | `step_store.list_steps(sid)` aggregated |
| `get_artifact_runtime_status` / `get_proposition_runtime_status` | Same shape, reads `artifact_store.list_artifacts(sid)` |

Ownership and other business rules from `SessionManager` move into the corresponding `runtime/session.py` function and call `ports.authz.check(...)` directly.

## 6. Recommended Work Stages

The spec does not prescribe PR slicing. The implementing agent (writing-plans / executing-plans phase) chooses how to slice. The recommended stages below capture topological order — each stage may be one or several PRs:

```
Stage A: Add 3 port methods (SessionStore.list_sessions, StepStore.list_steps,
         ArtifactStore.list_artifacts) + their local adapter implementations.

Stage B: Build runtime/session.py, runtime/intent_execution.py,
         runtime/semantic_ops.py with the migrated logic in place.

Stage C: Switch each MarivoRuntime method to delegate to the new modules.
         Drop _svc references one method group at a time.

Stage D: Update AppServices in app/api/deps.py to hold a MarivoRuntime
         instead of a SemanticLayerService. Patch every API endpoint
         that dereferences services.service.* to use services.runtime.*.

Stage E: Delete app/service.py and app/session/session_manager.py.
         Add import-linter rule forbidding app.service imports.
         Clean up any residual imports.
```

Tests within each stage update in the same stage. Tests across stages are not required to be green — the gate is at the 6.1 exit, not between stages.

## 7. Testing Strategy

- **No new tests added in 6.1.** The new tests for port contracts and parity belong to 6.3.
- Existing HTTP / MCP / intent E2E tests must pass at the 6.1 exit.
- Direct unit tests of `SemanticLayerService` and `SessionManager` (tests that import these classes by name) are deleted alongside the classes.
- New `import-linter` contract:

```ini
[importlinter:contract:no-app-service]
name = No module may import app.service
type = forbidden
source_modules =
    app
forbidden_modules =
    app.service
```

This is the tripwire that prevents accidental resurrection of the service layer.

## 8. Risks

| Risk | Mitigation |
|------|------------|
| `SemanticLayerService` contains orchestration logic not covered by the migration tables (commit-after-execute hooks, retry, telemetry sites) | writing-plans phase performs a full `app/service.py` method audit; any method not listed in §5 is flagged and decided before implementation starts |
| `SessionManager` business rules are scattered (ownership, status transitions, validation) | Same audit; ownership / AuthZ checks land explicitly in `runtime/session.py` functions |
| `_svc` depends on global state (singletons, request-scoped context) outside of `ports` and `core` | `grep` `app/service.py` for `self._*` fields; anything not sourced from a port or from core gets an explicit decision |
| Frontend or external scripts break when `app/service.py` disappears | Not a Phase 6.1 concern. Frontend already uses MCP/runtime call paths after Phase 5; no external consumers exist (pre-launch) |
