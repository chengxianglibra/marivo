## 1. Scope

Phase 6.3 turns the existing per-port tests in `tests/contracts/` into a shared contract harness and adds a local/server parity skeleton for the query-and-meaning layer.

| Sub-phase | Name | Deliverable |
|-----------|------|-------------|
| 6.1 | Runtime Self-Sufficiency | Runtime no longer depends on `app/service.py` |
| 6.2 | Server Profile Boundary | `resolve_profile()` + `create_server_runtime()` over the existing wrapper adapters |
| **6.3** | **Contract & Parity Tests** | **Shared contract harness + parity skeleton for `DataSource` and `ModelStore`** |

This spec covers **6.3 only**.

### 1.1 In scope

1. A reusable `tests/contracts/` harness that runs the same contract cases against multiple adapter factories.
2. Contract cases for `DataSource` and `ModelStore` only.
3. A parity skeleton that compares local and server adapters on the same named cases and reports contract-level differences.
4. CI wiring that makes local contract tests part of the normal test suite while leaving server parity as observable but non-blocking.

### 1.2 Out of scope

| Item | Lands in |
|------|----------|
| SessionStore / StepStore / ArtifactStore parity | Phase 9 |
| MySQL / testcontainers infrastructure | Phase 9 |
| Native server `SqlSessionStore` | Phase 9 |
| New runtime behavior, profile behavior, or API behavior | Not in 6.3 |
| Server-side mandatory contract gating | Phase 9 |

### 1.3 Functional non-goals

Phase 6.3 does not change product behavior. It does not add new port methods, new adapters, or new execution paths. It only standardizes how adapter behavior is checked and how local/server drift is observed before Phase 9 turns that observation into a hard gate.

## 2. Testing Architecture

The current `tests/contracts/` directory already contains direct adapter tests for `DataSource` and `ModelStore`. Phase 6.3 keeps those logical domains but changes the organization:

1. Contract cases become data, not ad hoc pytest functions.
2. A shared harness runs each case against an adapter factory.
3. A parity layer reuses the same cases to compare local and server implementations.

The goal is to make the contract rules explicit and reusable without forcing all adapter behavior into one lowest-common-denominator shape.

### 2.1 Contract case model

Each port owns a case list. A case describes:

- the adapter input setup
- the operation to invoke
- the expected success result, or the expected exception class and error code
- the assertions that define the stable contract surface

The case model keeps adapter-specific edge behavior isolated. For example, `DataSource` may still support a legacy raw-SQL call shape in local mode, but that remains a local-only case rather than a universal contract requirement.

### 2.2 Shared harness

The harness is the shared executor for all contract cases. It is responsible for:

- constructing the adapter from a factory
- running the case
- collecting result or exception data
- applying the case-specific assertions
- formatting failures consistently

The harness does not know business semantics. It only knows how to run a declared contract case and compare the result to the declared expectation.

### 2.3 Parity skeleton

The parity skeleton uses the same case definitions to compare one local adapter and one server adapter.

Parity is intentionally coarse:

- success vs failure must match
- failure type and `ErrorCode` must match
- stable output fields must match
- implementation details may differ when they are not part of the contract

This keeps the first parity layer useful without demanding full byte-for-byte equivalence from adapters that still differ internally.

## 3. Covered Ports

Phase 6.3 covers two ports only.

### 3.1 `DataSource`

Contract cases must cover:

- `execute(LogicalQuery)` returning `QueryResult`
- SQL syntax errors mapping to `ValidationError(ErrorCode.VALIDATION)`
- `schema(SourceRef)` returning a stable column subset
- missing tables mapping to `NotFoundError(ErrorCode.NOT_FOUND)`

The local DuckDB adapter is the first concrete implementation under test. Legacy raw-SQL support remains local-specific if needed, but it is not a port-wide contract requirement.

### 3.2 `ModelStore`

Contract cases must cover:

- missing selector returning `None`
- `save()` followed by `get()` round-tripping the model by name
- `list()` returning stable summary fields: `name`, `description`, `visibility`, `owner`

Behavior that is specific to one file-backed implementation, such as mtime cache invalidation or atomic temp-file handling, remains in adapter-specific tests and is not elevated into the shared contract.

---

## 4. File Layout

The phase introduces a small test-support layer rather than a new runtime package.

```
tests/contracts/
  contract_cases.py      # contract case definitions for DataSource and ModelStore
  contract_harness.py    # shared executor and assertion helpers
  parity.py              # local/server comparison helpers
  test_data_source.py    # updated to use the shared harness
  test_model_store.py    # updated to use the shared harness
```

The existing `tests/contracts/` package remains the home for contract tests. No new top-level test tree is introduced.

## 5. Contract Semantics

### 5.1 DataSource contract

The shared contract for `DataSource` is defined by the protocol, not by any single adapter. The stable surface is:

- `execute(query)` returns a `QueryResult`
- invalid SQL raises `ValidationError`
- missing schema objects raise `NotFoundError`
- schema output exposes the stable fields needed by runtime consumers

The contract does not require all adapters to accept the same convenience input shapes if those shapes are not part of the protocol. Adapter-specific convenience remains adapter-specific.

### 5.2 ModelStore contract

The shared contract for `ModelStore` is similarly narrow:

- read operations on missing entities are non-fatal and return `None`
- save/get round-trips preserve the semantic model identity
- list returns summaries with stable ownership and visibility fields

Implementation details such as file naming, cache invalidation, and overwrite id behavior stay local to the adapter unless they are part of the protocol surface.

## 6. Parity Rules

Parity is a comparison tool, not a new business contract.

### 6.1 What parity checks

- same case name
- same success or failure shape
- same `ErrorCode` when a failure is expected
- same stable contract fields in successful results

### 6.2 What parity does not check

- internal SQL text
- file layout
- cache strategy
- adapter-specific helper methods
- transport details

### 6.3 Local/server expectation

The local adapter is the baseline. The server adapter must be able to register into the same parity matrix, even if some cases are skipped or marked xfail while Phase 9 work is still incomplete.

Phase 6.3 therefore establishes observability and shared semantics now, while Phase 9 turns the same matrix into a mandatory gate for server adapters.

---

## 7. CI and Acceptance

### 7.1 Local contract tests

Local contract tests are part of the normal test suite. `DataSource` and `ModelStore` contract coverage must run in ordinary CI and must stay green.

### 7.2 Parity skeleton

The parity skeleton must be runnable in CI, but it is not yet a blocking server gate. It must clearly report:

- adapter pair under comparison
- case name
- pass / fail / skip / xfail
- failure reason or mismatch summary

### 7.3 Exit criteria

Phase 6.3 is complete when:

- `tests/contracts/` uses the shared contract harness for `DataSource` and `ModelStore`
- local contract coverage passes in the normal test suite
- parity skeleton can compare local and server adapters on the same case names
- server parity remains non-blocking and clearly annotated until Phase 9

---

## 8. Phase Boundary

Phase 6.3 deliberately stops short of the production server-adapter hardening work.

Phase 9 will:

- extend the same harness to `SessionStore`, `StepStore`, and `ArtifactStore`
- add MySQL-backed adapter coverage
- convert parity from skeleton to mandatory gate
- decide which server failures become hard failures and which remain adapter-specific skips

That separation keeps 6.3 focused on repeatable contract definition and leaves production server enforcement for the phase that owns the production adapters.
