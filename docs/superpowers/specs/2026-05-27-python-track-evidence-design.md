# Python Track Evidence Design

Status: approved design, pending implementation plan.

This document records the accepted design for adding evidence support to the
`marivo.analysis_py` Python library. The source contract is
[`docs/specs/analysis/python-track-evidence-surface.md`](../../specs/analysis/python-track-evidence-surface.md);
this design explains the target architecture, implementation boundaries, and
changes needed to make that contract executable.

## Decision

Use a Python-track native evidence runtime inside `marivo.analysis_py`.

Existing evidence-engine code may be reused only by copying pure rules and
fixtures into the Python track, then reshaping them around `analysis_py` frame,
lineage, and session types. `marivo.analysis_py` must not directly import
`marivo.runtime.*`, `marivo.core.evidence`, `marivo.adapters.*`, or MCP/HTTP
contract code. The old MCP/HTTP evidence chain is legacy and will be removed;
there is no compatibility, dual-write, or data migration requirement.

## Architecture

Add a package dedicated to the Python evidence runtime:

```text
marivo/analysis_py/evidence/
  types.py
  identity.py
  extraction/
  seeding.py
  assessment.py
  followups.py
  knowledge.py
  store.py
  pipeline.py
```

Responsibilities:

- `types.py`: public immutable evidence models: `Subject`, `ArtifactRef`,
  `ConfidenceScope`, `QualitySummary`, `BlockingIssue`, `FollowupAction`,
  `Fact`, `OpenItem`, `Finding`, `Proposition`, `Assessment`, and trace types.
- `identity.py`: deterministic `artifact_id`, `finding_id`, `proposition_id`,
  `followup.action_id`, subject canonical key, and canonical JSON helpers.
- `extraction/`: `analysis_py` frame/meta to canonical finding extractors.
- `seeding.py`: system-seeded proposition templates copied and simplified from
  the existing T1-T6 registry.
- `assessment.py`: latest-snapshot assessment recompute with only the rule
  families in scope for the Python track.
- `followups.py`: result-level C1/C2 followup generation and conformance
  guards.
- `knowledge.py`: `SessionKnowledge` projection from the judgment store.
- `store.py`: session-local SQLite schema, migrations, repository helpers,
  WAL setup, lock handling, and startup recovery.
- `pipeline.py`: step commit side-effect orchestration.

The package depends only on `marivo.analysis_py` modules, standard library, and
existing Python-track dependencies such as Pydantic, pandas, pyarrow, and
SQLite. Import-linter should enforce this boundary.

## Public Surface

The public API follows three layers.

### Surface 1: Result-Bound

Every result frame exposes flat evidence fields directly. The fields should be
declared once on `BaseFrameMeta` so all frame families inherit the same public
contract:

```python
artifact_id: str
subject: Subject
source_refs: list[ArtifactRef]
evidence_status: Literal["complete", "partial", "unavailable"]
confidence_scope: ConfidenceScope
quality: QualitySummary | None
blocking_issues: list[BlockingIssue]
recommended_followups: list[FollowupAction]
```

There is no `result.evidence.*` namespace. Agents read
`result.recommended_followups`, `result.blocking_issues`, `result.quality`, and
`result.evidence_status` directly after each operator call.

`artifact_id` is the canonical replay-stable identity. Existing `ref` remains a
frame-loading alias during the transition, but target-state operators should
make `ref == artifact_id` and stop generating random `frame_*` refs.

### Surface 2: Session-Bound

`Session` gains read and dispatch methods:

```python
knowledge = session.knowledge()
knowledge.facts(kind="change")
knowledge.open_items(kind="anomaly")
knowledge.next_steps(top=5)
result = session.run_followup(action)
```

These methods do not create a separate execution model. `run_followup` dispatches
to the existing operators and writes `triggered_by_followup` into the new step
lineage so `next_steps()` can filter already executed actions.

### Surface 3: Object-Bound

Audit access is available but not part of the default agent workflow:

```python
session.findings(...)
session.propositions(...)
session.assessments(...)
session.evidence.trace(proposition_id)
```

Surface 3 reads `judgment.db` and exposes `Finding`, `Proposition`,
`Assessment`, and `EvidenceTrace` objects.

### Candidate Item Followups

Current `CandidateSet` row-level `recommended_followups_json` remains valid,
but it is distinct from result-level `result.recommended_followups`.

- `result.recommended_followups`: artifact-level C1/C2 followups generated by
  the evidence pipeline.
- `candidate_item.recommended_followups`: candidate row payload read through
  `select(..., field="recommended_followups")`.

Both use the same `FollowupAction` type from `analysis_py.evidence.types`.

## Commit Pipeline

Existing operators keep responsibility for computing DataFrames and basic
semantic metadata. Persistence should move behind one helper:

```python
commit_result(
    session,
    intent="compare",
    frame=result_frame,
    inputs=[...],
    params=...,
    extractor_family="delta_frame",
)
```

`commit_result` performs:

1. Derive deterministic `artifact_id` and `job_id`.
2. Write `frames/<artifact_id>/data.parquet.tmp`, fsync, and rename.
3. Open a SQLite transaction in `judgment.db`.
4. Insert artifact metadata.
5. Extract canonical findings and enforce D4 empty semantics.
6. Compute `confidence_scope` and lightweight `quality_summary`.
7. In a SAVEPOINT, seed propositions, recompute latest assessments, derive
   blocking issues, and generate C1/C2 followups.
8. If the SAVEPOINT fails, roll back only that phase, set
   `evidence_status="partial"`, and insert a `kind="evidence_partial"` issue.
9. Update frame metadata, write the job record, and return the result with
   Surface 1 fields populated.

If `judgment.db` cannot be opened, the operator still returns an in-memory
result with `evidence_status="unavailable"`, intrinsic fields populated, and a
`kind="evidence_store_unavailable"` blocking issue. The result can feed later
operators in the same process but cannot be recovered after restart.

## Storage

Storage stays under the existing session layout:

```text
<project_root>/.marivo/analysis/
  active
  sessions/<session_id>/
    meta.json
    jobs/<job_id>.json
    frames/<artifact_id>/
      data.parquet
      meta.json
    judgment.db
```

SQLite is the source of truth for artifact metadata, findings, propositions,
assessments, blocking issues, and followups. Frame `meta.json` caches Surface 1
fields for `load_frame()` ergonomics; parquet stores raw tabular data only.

Startup recovery should delete `frames/*/*.tmp` and move or delete orphan frame
directories that have no matching artifact row in `judgment.db`.

## Copied-Code Reuse

Allowed sources:

- canonical finding typed fields and identity rules from the existing
  `canonical_finding` code;
- family extractor canonical item key logic;
- T1-T6 seed template registry structure;
- assessment gate constants and status-resolution logic;
- tests and fixtures that encode stable evidence behavior.

All copied code must be transformed into Python-track code: new module paths,
new imports, `AnalysisError` subclasses, `analysis_py` frame/meta inputs, and
session-local SQLite repositories.

Do not copy or preserve:

- server evidence repositories;
- AOI/MCP/HTTP payload adapters;
- action proposal policy engine;
- publish switches;
- runtime replay recovery tied to the old chain;
- transport projections or compatibility shims.

## Error Semantics

New errors subclass `AnalysisError` and follow the repository structured
exception template:

- `EvidenceStoreUnavailable`
- `FindingExtractionFailed`
- `EvidencePartial`
- `FollowupGenerationRuleViolated`
- `SchemaVersionMismatch`
- `MigrationFailed`
- `SessionLockedByAnotherProcess`
- `PropositionNotFound`

Extractor and D4 failures abort the step. Seeding, assessment, followup, and
blocking issue failures downgrade the result to `partial`. Store startup
failure downgrades the result to `unavailable`.

## Test Plan

Required coverage:

- import-linter boundaries for `analysis_py.evidence`;
- frozen model construction, extra-field rejection, and JSON round trips;
- replay-stable IDs and canonical keys;
- SQLite schema initialization, migration, WAL, busy lock, and startup GC;
- SAVEPOINT partial behavior;
- first vertical slice: `observe -> compare -> ChangeFact`;
- copied extractor fixture parity against Python-track frame/meta inputs;
- C1/C2 followup conformance;
- `session.knowledge()` and `session.run_followup()` behavior;
- `make examples-check` after updating `marivo-skill/marivo-py-analysis`.

## Documentation Changes

Update `docs/specs/analysis/python-track-evidence-surface.md` so it is both the
target contract and implementation blueprint.

Update `docs/specs/analysis/python-analysis-operator-design.md` only where
current text conflicts with the unified evidence types, `FollowupAction`
category, result evidence fields, or `artifact_id/ref` target relationship.

Update `marivo-skill/marivo-py-analysis/SKILL.md` and references to show
Surface 1/2 usage only. Keep Surface 3 and `judgment.db` schema in references
or specs, not the main skill entrypoint.

Do not update MCP/HTTP docs for compatibility. If they are touched later, the
only relevant note is that the Python-track evidence API is the target state and
the old transport chain is legacy.

## Implementation Slice

The first implementation plan should be a thin vertical slice:

1. Add evidence types, identity helpers, store schema, and import-linter
   boundaries.
2. Wire `commit_result` for `observe` and `compare`.
3. Extract `delta` findings and seed `change` propositions.
4. Project `ChangeFact` through `session.knowledge()`.
5. Generate only the C1/C2 followups supported by this slice.

Only after that slice is verified should implementation expand across
`decompose`, `discover`, `correlate`, `test`, `forecast`, and Surface 3 audit
coverage.
