## 1. Scope

Phase 6 in the parent spec is decomposed into three sub-phases:

| Sub-phase | Name | Deliverable |
|-----------|------|-------------|
| 6.1 | Runtime Self-Sufficiency | Demolish service layer; `MarivoRuntime.__init__(ports, core)` only |
| **6.2** | **Server Profile Boundary** | **`resolve_profile()` + `create_server_runtime()` over the existing `wrappers.py` adapters; HTTP entry cuts over** |
| 6.3 | Contract & Parity Tests | Port contract test framework + parity skeleton |

This spec covers **6.2 only**. 6.3 is brainstormed when its turn comes.

### 1.1 In scope

1. New module `app/profiles/server.py` exporting `create_server_runtime(config: ServerConfig) -> ServerComposition`. Owns all server-side construction: `MetadataStore`, `AnalyticsEngine`, `DatasourceService`, `QueryRouter`, `SemanticModelV2Service`, the wrapper-based port adapters, and the resulting `MarivoRuntime`.
2. New module `app/profiles/resolver.py` exporting `resolve_profile(entry_point, ...) -> ProfileMode`. Single source of truth for the precedence rules in parent §7. Both stdio and HTTP entrypoints call it as a fail-fast guard.
3. `_build_services` in `app/api/app_factory.py` shrinks to `resolve_profile(...) → create_server_runtime(...) → AppServices(...)`. The legacy `app/runtime/factory.py::create_runtime_from_service` is deleted.
4. `app/profiles/local.py::create_local_runtime` keeps its public signature; its ad-hoc `_check_deployment_guard` is replaced by `resolve_profile(entry_point="local_stdio")`.
### 1.1.1 Hard dependency on 6.1

6.2 cannot start until 6.1's Stage F (delete `app/service.py`, delete `SessionManager`, update `app/runtime/factory.py` to compile without `SemanticLayerService`) has merged to main. Specifically, 6.1's Stage F necessarily owns:

- `SqlSessionStoreAdapter.__init__` dropping its now-orphaned `session_manager` parameter (or 6.1 keeps the param and passes `None`; 6.2 trusts whichever choice 6.1 made).
- `MetadataArtifactStoreAdapter.__init__` dropping its now-orphaned `svc: SemanticLayerService` parameter, AND relocating the live `self._svc.<method>` call site at `app/adapters/server/artifact_store.py:203-205` (the finding-extraction pipeline) into a runtime use-case module — likely `runtime/intent_execution.py` or a new helper in `runtime/session.py`. This is a load-bearing relocation, not a constructor-knob removal.

6.2's writing-plans phase MUST verify these landed before slicing tasks.

### 1.2 Out of scope (deferred)

| Item | Lands in |
|------|----------|
| Native event-sourced `SqlSessionStore` (MySQL `session_events`) | Phase 9 |
| Splitting `app/adapters/server/wrappers.py` into per-port modules | Phase 9 |
| Real `OidcRbacAuthZ` (replaces `NoopAuthZAdapter`) | Phase 8 |
| Real OpenTelemetry, `RedisCacheStore`, `S3EvidenceStore` | Phase 9 |
| `RoutingDataSource` credential / registry redesign | Phase 9 (see parent §13.1) |
| `AlwaysAllowAuthZ` production guard (`MARIVO_ENV=production` fail-closed) | Phase 8 (see parent §13.2) |
| Hiding `DatasourceService` / `QueryRouter` / `MetadataStore` from `AppServices` | Never. These are admin-endpoint dependencies, not port-surface ones. |

### 1.3 Functional non-goal

After 6.2, server profile is **constructible**. The HTTP `/health` round-trip works. Session-dependent endpoints (any path that hits `SessionStore.append_event` / `load_events` / `list_sessions`) continue to raise `NotImplementedError` from `SqlSessionStoreAdapter` until Phase 9. This matches today's behavior; 6.2 does not regress nor improve it.

Operationally this means a session-using HTTP request will surface as a 500 with a bare `NotImplementedError` traceback (whatever today's FastAPI default surfaces). 6.2 does not invest in translating that to a typed `501 Not Implemented` envelope; Phase 9 makes the question moot when the native `SqlSessionStore` lands.

## 2. `ServerConfig` and `ServerComposition`

```python
# app/profiles/server.py

@dataclass
class ServerConfig:
    """Inputs needed to construct a server-profile runtime.

    Carries an already-loaded MarivoConfig plus the few inputs that
    are not in it (analytics path, optional infrastructure overrides
    for tests). Phase 9, when adapters go native, revisits this and
    likely adds direct fields for db_url, S3 config, etc.
    """
    marivo_config: MarivoConfig
    db_path: Path | str | None = None              # analytics path; ":memory:" allowed
    metadata_store: MetadataStore | None = None     # test injection
    analytics_engine: AnalyticsEngine | None = None # test injection


@dataclass
class ServerComposition:
    runtime: MarivoRuntime
    metadata_store: MetadataStore
    analytics_engine: AnalyticsEngine
    datasource_service: DatasourceService
    query_router: QueryRouter
    semantic_v2_service: SemanticModelV2Service
    metrics: MetricsCollector | None
    resolved_analytics_path: Path | str


def create_server_runtime(config: ServerConfig) -> ServerComposition: ...
```

**Why this shape:**

- `marivo_config: MarivoConfig` — reuse the existing config object. Don't invent a parallel schema.
- Test injection of `metadata_store` / `analytics_engine` mirrors today's `create_app(metadata_store=..., analytics_engine=...)`. Existing tests don't have to refactor.
- `ServerComposition` fields map 1:1 onto `AppServices` fields. `_build_services` becomes a one-call adapter. The two structures stay near-redundant through 6.2; Phase 9 collapses them when the wrapper splits make per-port construction simpler.
- `resolved_analytics_path` is preserved on `ServerComposition` because `AppServices.resolved_path` exposes it for tests/diagnostics.

**Stability note:** `ServerConfig` is a transitional shape. Phase 9 will likely add direct fields like `db_url`, `s3_config`, etc., and remove the `metadata_store` / `analytics_engine` test-injection escape hatches once native adapters take their construction inputs directly. Downstream tests that inject `ServerConfig` should expect breaking changes at the Phase 9 boundary.

**Factory body sketch:**

```python
def create_server_runtime(config: ServerConfig) -> ServerComposition:
    setup_logging(level=config.marivo_config.observability.log_level)
    metrics = (
        MetricsCollector()
        if config.marivo_config.observability.metrics_enabled
        else None
    )

    resolved_path, metadata_store, analytics_engine = _resolve_storage(
        config.db_path,
        config.metadata_store,
        config.analytics_engine,
        config.marivo_config,
    )

    datasource_service = DatasourceService(metadata_store)
    query_router = QueryRouter(metadata_store, datasource_service)
    semantic_v2 = SemanticModelV2Service(
        metadata_store, datasource_service=datasource_service
    )

    ports = _build_server_ports(
        metadata_store=metadata_store,
        analytics_engine=analytics_engine,
        datasource_service=datasource_service,
        query_router=query_router,
        semantic_v2_service=semantic_v2,
        marivo_config=config.marivo_config,
    )
    runtime = MarivoRuntime(ports, CoreEngine())

    return ServerComposition(
        runtime=runtime,
        metadata_store=metadata_store,
        analytics_engine=analytics_engine,
        datasource_service=datasource_service,
        query_router=query_router,
        semantic_v2_service=semantic_v2,
        metrics=metrics,
        resolved_analytics_path=resolved_path,
    )
```

`_resolve_storage` is lifted from `app/api/app_factory.py` into `app/profiles/server.py` as a private helper. `app_factory.py` imports it back if it still needs it. No duplication.

`_build_server_ports` is a private function in `app/profiles/server.py` that constructs each port adapter from `app/adapters/server/wrappers.py` (and `app/adapters/server/artifact_store.py`). The post-6.1 wrapper signatures (`MetadataArtifactStoreAdapter` and `SqlSessionStoreAdapter` without their orphaned constructor params, and the relocated artifact-extraction call site) are 6.1's deliverable per §1.1.1; 6.2 just calls them.

## 3. `resolve_profile()` and `ProfileMode`

```python
# app/profiles/resolver.py
from __future__ import annotations

import logging
import os
import tomllib
from pathlib import Path
from typing import Literal, Mapping

from app.config import MarivoConfig
from app.contracts.errors import DomainError, ErrorCode

logger = logging.getLogger(__name__)

ProfileMode = Literal["local", "server"]
EntryPoint = Literal["local_stdio", "server_http"]


class ProfileResolutionError(DomainError):
    """Entry point cannot resolve a valid profile, or the resolved
    profile is incompatible with the entry point."""


def resolve_profile(
    *,
    entry_point: EntryPoint,
    explicit: str | None = None,
    env: Mapping[str, str] | None = None,
    workspace_config_path: Path | None = None,
    service_config: MarivoConfig | None = None,
) -> ProfileMode:
    env_map = env if env is not None else os.environ

    # Normalize empty strings to None: MARIVO_PROFILE="" is missing, not "".
    explicit = explicit or None
    env_value = env_map.get("MARIVO_PROFILE") or None

    candidate: str | None = explicit or env_value
    source = "explicit" if explicit else ("env" if env_value else None)

    # Service config file (loaded MarivoConfig) — consulted at server entry
    # per parent §7 precedence (explicit > env > service config file).
    if candidate is None and service_config is not None:
        cfg_value = getattr(service_config, "profile", None) or None
        if cfg_value:
            candidate = str(cfg_value)
            source = "service_config"

    # Workspace .marivo/marivo.toml — consulted only at local stdio entry
    # per parent §7 precedence. Server HTTP entry ignores the workspace.
    if (
        candidate is None
        and entry_point == "local_stdio"
        and workspace_config_path is not None
    ):
        candidate = _read_profile_from_toml(workspace_config_path)
        if candidate:
            source = "workspace_toml"

    if candidate is None:
        candidate = _DEFAULT_BY_ENTRY[entry_point]
        source = "default"

    if candidate not in ("local", "server"):
        logger.error(
            "profile.unknown entry_point=%s source=%s candidate=%r",
            entry_point, source, candidate,
        )
        raise ProfileResolutionError(
            ErrorCode.VALIDATION,
            f"Unknown profile {candidate!r}; expected 'local' or 'server' "
            f"(source={source}, entry_point={entry_point})",
        )

    if candidate not in _ALLOWED_BY_ENTRY[entry_point]:
        logger.error(
            "profile.incompatible entry_point=%s source=%s candidate=%s",
            entry_point, source, candidate,
        )
        raise ProfileResolutionError(
            ErrorCode.VALIDATION,
            f"Profile {candidate!r} (source={source}) is not allowed at "
            f"entry point {entry_point!r}",
        )

    return candidate


def _read_profile_from_toml(path: Path) -> str | None:
    """Read top-level `profile` from .marivo/marivo.toml, or None if absent.

    Wraps malformed-TOML errors as ProfileResolutionError so stdio start
    fails cleanly instead of crashing with a bare TOMLDecodeError.
    """
    if not path.is_file():
        return None
    try:
        with path.open("rb") as fh:
            data = tomllib.load(fh)
    except tomllib.TOMLDecodeError as exc:
        raise ProfileResolutionError(
            ErrorCode.VALIDATION,
            f"Malformed TOML in {path}: {exc}",
        ) from exc
    value = data.get("profile")
    return str(value) if value else None


_DEFAULT_BY_ENTRY: dict[EntryPoint, ProfileMode] = {
    "local_stdio": "local",
    "server_http": "server",
}

_ALLOWED_BY_ENTRY: dict[EntryPoint, frozenset[ProfileMode]] = {
    "local_stdio": frozenset({"local"}),
    "server_http": frozenset({"server"}),
}
```

**Why this shape:**

- Two concerns separated: precedence (explicit → env → service config → workspace toml → default) and entry-point compatibility (which profiles each entry can serve). Mixing them in one branch tree is what made parent §7's table hard to translate to code.
- Per parent §7, server HTTP entry consults `(explicit, env, service config file)` and ignores workspace `.marivo/marivo.toml`. Local stdio entry consults `(explicit, env, workspace toml)`. The resolver makes both source sets data-driven so a future entry point only changes its source ordering.
- The "service config file" at server entry is the loaded `MarivoConfig` — already resolved by `create_app()` via `resolve_config_path()` (which honors `MARIVO_CONFIG`). The resolver reads `marivo_config.profile`. The TOML schema for this field is documented in §3.1.
- `_ALLOWED_BY_ENTRY` is currently 1-to-1, so any explicit override that picks the "wrong" profile for the entry point fails closed. This is the parent §7 guarantee: profile auto-detection must never let an enterprise server bind itself from a nearby developer workspace config by accident. A future entry point that legitimately serves multiple profiles changes only the table.
- `ProfileMode` is `Literal["local", "server"]` (no `client` — parent §7 abolished the client profile). Lightweight, JSON-serializable for telemetry. Promote to enum later if a real consumer needs it.
- `MARIVO_DEPLOYMENT` env var (used by today's ad-hoc `_check_deployment_guard`) is **dropped**. It was a one-way warning that overlapped `MARIVO_PROFILE`. After 6.2, `MARIVO_PROFILE=local|server` is the single bidirectional control. Parent §7's example "(e.g. `MARIVO_DEPLOYMENT=server` env var, or presence of a `server.toml` config)" is updated to reference `MARIVO_PROFILE` and the loaded `marivo_config.profile` field; `server.toml` was always aspirational and is dropped.
- Empty-string env vars (`MARIVO_PROFILE=""`) are normalized to missing rather than treated as the literal string `""`. This is a common shell-export footgun and a fail-loud surprise is worse than fail-through.
- Errors log structured context (`entry_point`, `source`, `candidate`) before raising so operators can debug a refusing-to-start production server from logs alone.

### 3.1 Profile schema (two file formats)

Marivo today carries two parallel config files: `marivo.yaml` (server, loaded by `app/config.py::load_config()` into a `MarivoConfig` Pydantic model) and `.marivo/marivo.toml` (local stdio workspace, exposed via `TomlRuntimeConfig` to the `RuntimeConfig` port). 6.2 does not unify them — that's a deeper refactor outside this phase. Each format gets its own `profile` field:

```yaml
# marivo.yaml (server) — read by load_config() → MarivoConfig
profile: server      # or "local"
```

```toml
# .marivo/marivo.toml (local workspace)
profile = "local"    # or "server"
```

The field is optional in both formats. Absence falls through to the next precedence source. Any value other than `"local"` or `"server"` raises `ProfileResolutionError` with a descriptive message. Malformed TOML at the local entry raises `ProfileResolutionError` rather than a bare `TOMLDecodeError`.

Implementation:
- Server entry: resolver reads `marivo_config.profile` (Pydantic field added to `MarivoConfig` in this phase).
- Local entry: resolver reads top-level `profile` key from `.marivo/marivo.toml` via `_read_profile_from_toml`.

The field lives at the top of each file rather than under a `[profile]` section because it carries a single scalar value.

**Wiring at the two entry points:**

```python
# app/profiles/local.py
def create_local_runtime(
    config: LocalConfig,
    *,
    explicit: str | None = None,
) -> MarivoRuntime:
    resolve_profile(
        entry_point="local_stdio",
        explicit=explicit,
        workspace_config_path=config.workspace_root / ".marivo" / "marivo.toml",
    )
    # ... existing construction ...
```

```python
# app/api/app_factory.py
def _build_services(*, marivo_config, db_path, metadata_store, analytics_engine) -> AppServices:
    resolve_profile(
        entry_point="server_http",
        service_config=marivo_config,  # parent §7: server entry consults service config file
    )  # raises ProfileResolutionError if env mis-set

    composition = create_server_runtime(
        ServerConfig(
            marivo_config=marivo_config,
            db_path=db_path,
            metadata_store=metadata_store,
            analytics_engine=analytics_engine,
        )
    )

    return AppServices(
        resolved_path=composition.resolved_analytics_path,
        config=marivo_config,
        runtime=composition.runtime,
        datasource_service=composition.datasource_service,
        query_router=composition.query_router,
        metadata_store=composition.metadata_store,
        analytics_engine=composition.analytics_engine,
        metrics=composition.metrics,
        semantic_v2_service=composition.semantic_v2_service,
    )
```

Exact field shape of `AppServices` follows whatever 6.1's Stage E left it at — 6.2 does not restructure it.

## 5. Error Handling

- `ProfileResolutionError` is a `DomainError` subclass with `ErrorCode.VALIDATION`. HTTP and CLI surfaces translate via existing error envelopes — no new error code is invented.
- Every `ProfileResolutionError` raise site logs structured context (`entry_point`, `source`, `candidate`) at `ERROR` level *before* raising. Operators debugging a refusing-to-start production server should be able to see, from logs alone: which entry point asked, which source produced the rejected value, and what that value was.
- `_read_profile_from_toml` wraps `tomllib.TOMLDecodeError` to `ProfileResolutionError(VALIDATION, ...)` so a malformed `marivo.toml` fails cleanly at stdio start instead of crashing with a bare parser exception.
- `create_server_runtime` does not catch infrastructure construction errors. A bad config or unreachable database fails at `metadata_store.initialize()` time and propagates. Today's `_build_services` behaves the same way.
- `_build_services` calls `resolve_profile()` *before* `create_server_runtime()` so misconfiguration surfaces with the cheaper error first.

## 6. Testing Strategy

No port contract framework yet — that ships in 6.3.

| Test file | Coverage |
|-----------|---------|
| `tests/profiles/test_resolver.py` | Precedence: explicit > env > service config > workspace toml > default. Entry-point compatibility: local entry rejects `MARIVO_PROFILE=server`; server entry rejects workspace toml saying `local`. Unknown profile string raises. Empty-string env var (`MARIVO_PROFILE=""`) treated as missing. Malformed TOML raises `ProfileResolutionError`, not bare `TOMLDecodeError`. Logger emits structured `entry_point` / `source` / `candidate` on every raise. |
| `tests/profiles/test_server_factory.py` | `create_server_runtime(ServerConfig)` returns a `ServerComposition` whose `runtime` is a `MarivoRuntime` over wrapper adapters. Test injection of `metadata_store` / `analytics_engine` works. (The "no `app.service` / `app.runtime.factory` import" property is enforced by import-linter, not by a unit test.) |
| `tests/api/test_app_factory.py` (existing) | Extend: `_build_services` end-to-end smoke routes through `create_server_runtime`. `/health` succeeds. |
| `tests/local/test_local_runtime_factory.py` (existing) | Add: `MARIVO_PROFILE=server` at local stdio entry raises `ProfileResolutionError`. Remove the obsolete `MARIVO_DEPLOYMENT=server` cases. |

**Test-environment isolation rule:** every resolver test passes `env=` explicitly (`env={}` or a tightly-scoped `env={"MARIVO_PROFILE": ...}`). No resolver test relies on `os.environ` defaults. CI machines and developer shells regularly carry stray `MARIVO_*` env vars; relying on defaults would make the suite flaky.

**Import-linter additions:**

```ini
[importlinter:contract:profiles-do-not-import-runtime-factory]
name = profiles must not import the deleted runtime factory
type = forbidden
source_modules =
    app.profiles
forbidden_modules =
    app.runtime.factory
```

The file deletion is itself a tripwire; the contract is belt-and-suspenders for the rare case someone reintroduces the module. The `no-app-service` rule already added in 6.1 continues to forbid any module from importing `app.service` and is not duplicated here.

## 8. Acceptance Criteria

0. **Prerequisite (verified before 6.2 PR opens):** 6.1's Stage F has merged to main. `app/service.py` and `app/session/session_manager.py` no longer exist. `MetadataArtifactStoreAdapter._svc` and `SqlSessionStoreAdapter._session_manager` no longer carry live call sites.

6. HTTP `/health` smoke test passes against `create_app()`. All existing tests that previously passed still pass; tests for the dropped `MARIVO_DEPLOYMENT` env var are removed.
7. Import-linter rule `profiles-do-not-import-runtime-factory` passes.
8. Resolver tests pass `env=` explicitly; suite remains stable when CI machines or developer shells carry stray `MARIVO_*` env vars.
9. Server profile remains functionally limited per §1.3 — session-dependent endpoints still raise `NotImplementedError`. This is expected and gated to Phase 9.
