# Marivo：面向智能体数据分析的约束式运行框架

> 智能伴随幻觉，约束产生价值。

Marivo 面向的不是“让模型更自由地访问数据”，而是“让 agent 能更可靠地完成数据分析”。它不是一个 BI 工具，也不是一个 text-to-SQL 包装器，更不是试图替代 agent 的黑盒分析脑。Marivo 的定位更接近一个 **分析运行时 / 约束框架**：为 agent 提供稳定语义、受约束动作、结构化证据和治理边界，让分析过程从一次性生成走向可持续推进、可复核、可治理的系统能力。Marivo 支持双模部署——本地 agent 分析模式通过 MCP stdio 嵌入式运行，无需守护进程；企业智能分析平台模式通过 HTTP API 与 HTTP MCP 提供集中式治理——同一套核心引擎，适应不同落地场景。

## 为什么智能体数据分析还需要约束框架

今天很多智能体数据分析系统已经能回答问题、生成 SQL、访问表、绘制图表，但它们距离可靠的分析系统仍有明显距离。问题通常不在模型本身，而在系统边界太松。

如果把真实业务分析拆开看，会发现它并不只是“把一句自然语言翻译成一条查询”。它还要求：

- 口径稳定，而不是每轮都临时解释 metric 的含义。
- 分析动作稳定，而不是每轮都重新发明执行路径。
- 状态可持续推进，而不是靠对话上下文硬记住前面查过什么。
- 证据可复核，而不是最后只剩一段看起来合理的文本总结。
- 行为可治理，而不是把权限、质量、边界问题都留给人工兜底。

如果系统把“怎么理解问题”“怎么执行分析”“什么算证据”“哪些动作被允许”全部交给模型即兴决定，结果往往看起来聪明，但难以信任，也难以复用。

## 为什么 RAG / 知识库增强的 Text2SQL 不是可靠的分析运行时

在今天的工程现实里，最常见的一条路线，是给 text-to-SQL 叠加 schema 文档、指标词典、表说明、样例 SQL、常见问答，甚至向量检索和重排，希望模型先“查知识库”，再“写出更准的 SQL”。

这条路线不是没有价值。它通常能改善几件事：

- 更容易找到相关表和字段。
- 更容易把业务别名映射到物理列名。
- 更容易复用已有 SQL 模板和历史查询经验。
- 在简单查数问题上，首轮命中率会明显提升。

但这条路线本质上仍然是在优化“如何让模型写出一条更像样的 SQL”。它增强的是查询生成，不是分析运行时。因此它很难稳定支撑真实业务分析场景。

### 它的问题不只是 SQL 不够准，而是系统边界错了

第一，知识库能补充描述，但不能替代稳定语义契约。
RAG 可以把“watch time 大概对应哪些表和字段”塞给模型，却不能天然保证 `metric.watch_time` 的定义、时间对齐方式、粒度边界、去重口径、实体归属在多轮分析中始终一致。知识文本越多，模型越容易“看起来理解了”，但并不意味着系统得到了一个稳定可执行的语义对象。

第二，外部动作仍然是 SQL，而不是分析动作。
当系统的外部契约仍然是一条查询时，agent 面对的其实是开放式执行空间。它可以写聚合、写 join、写窗口、写过滤，但系统本身不知道这一步到底是在 `observe`、`compare`、`decompose` 还是 `test`。没有类型化动作边界，就很难做执行前检查、兼容性检查、结果复用和错误归因。

第三，它通常是一问一查，而不是有状态分析。
RAG/Text2SQL 的基本单位是“问题 -> 检索 -> SQL -> 结果”。一旦要继续追问，系统常常只能依赖长上下文对话，或者让模型重新描述已经查过的条件和结论。前面算过什么、哪些结果可引用、哪些假设已被支持、哪些缺口尚未关闭，通常没有一个正式的状态容器来承接。

第四，它返回的是查询结果，不是规范化证据。
SQL 返回的是结果表，模型再把结果表压缩成结论文本。这个链路可以生成“答案”，但很难形成 `artifact -> finding -> proposition -> assessment` 这样的结构化证据链。缺少证据对象，也就缺少可引用性、可继承性和可审计性。

第五，它很难把治理前置进主路径。
真实业务分析不只是“能跑出来就行”。系统还需要处理可比性、前置条件、质量闸门、就绪状态、权限边界、执行预算、执行引擎路由等约束。RAG/Text2SQL 往往把这些问题放在 SQL 执行之后暴露出来，或者干脆不建模，导致系统在演示阶段能跑，在生产阶段不稳。

### 它无法支持真实业务分析的根因

根因不在于“模型还不够聪明”，而在于它试图把分析问题退化为查询生成问题。

真实业务分析要解决的是：

- 围绕稳定语义定义分析对象。
- 围绕有限动作语法推进分析过程。
- 围绕 session 状态持续收敛，而不是单轮问答。
- 围绕证据链形成结论，而不是围绕 SQL 结果形成印象。

RAG / 知识库增强的 Text2SQL 优化的是“把自然语言翻译成查询”。Marivo 试图解决的是“把分析任务变成受约束、可复核、可治理的运行过程”。这两者不是同一层问题。

## Marivo 的核心主张

Marivo 的核心主张是：**智能伴随幻觉，约束产生价值**。

这句话的含义并不是否定模型，而是承认模型的价值和边界同时存在。Agent 擅长理解问题、提出假设、决定下一步、解释业务含义；但它并不适合在没有约束的情况下自由定义分析语义、执行边界和证据结构。Marivo 的设计目标，就是把这些高风险、高重复、需要稳定性的部分，收束成一个可验证的运行时。

Marivo 不替代 agent。它为 agent 提供的是一个受约束、证据优先的分析运行时。

## Marivo 与 Agent 的分工边界

这层分工如果不说清楚，Marivo 很容易被误解成“更复杂的工具箱”，或者“替 agent 做全部分析的系统”。这两种理解都不准确。

Agent 的职责是：

- 理解业务问题，判断当前真正要分析什么。
- 选择分析路径，决定下一步该观察、比较、拆解还是验证。
- 解释结果的业务含义，而不是只停留在数值层。
- 基于当前证据决定是否继续探索、停止，或转向新的假设。

Marivo 的职责是：

- 提供稳定的语义入口，让 agent 操作业务对象而不是裸表和临时字段。
- 提供受约束的分析动作，让执行边界清晰，而不是开放式拼装。
- 提供结构化证据表达，让结果能被引用、继承、复核，而不是只返回一段文本总结。
- 提供策略与治理机制，让分析行为具备生产可用性。

一句话说：

**Agent 负责决策，Marivo 负责落地与约束。**

Agent 负责“想清楚问什么、下一步做什么”，Marivo 负责“把这些动作变成受约束、可验证、可治理的分析过程”。

## Marivo 的三层核心能力

Marivo 的价值，不在于替 agent 思考，而在于让 agent 的分析能力从脆弱的通用生成，升级为稳定的系统能力。

### 1. Semantic Layer：稳定业务对象，而不是临时数据入口

`semantic layer` 解决的是“分析对象是什么”。

在 Marivo 中，agent 不应直接围绕裸表、裸列、临时 join 和历史 SQL 模板工作，而应围绕稳定语义对象工作。典型对象包括：

- `metric`：业务度量的稳定定义，例如 `metric.watch_time`、`metric.gmv`、`metric.dau`。
- `dimension`：共享分析轴，例如渠道、地区、品类、端类型、用户段。
- `entity`：业务主体或稳定归属边界，让“这是谁的属性”这件事不再漂移。
- `time`：统一时间语义、时间轴和可执行的时间绑定方式。
- `binding`：把语义对象和底层数据源、执行引擎、物理载体连接起来。

这层能力带来的不是“更方便找字段”，而是更基础的几件事：

- 让 metric、dimension、entity、time 有稳定引用，而不是每轮临时命名。
- 把业务口径从物理存储形状中抽离出来，降低底层 schema 变化对分析动作的影响。
- 在运行时里显式区分 `lifecycle_status` 和 `readiness_status`，避免“已发布”被误当成“可执行”。
- 让系统在执行前就知道一个语义对象是否具备 bindings、兼容性和时间语义，而不是让错误在 SQL 运行阶段才暴露。

在 Marivo 里，SQL 可以是下沉后的实现结果，但不能是外部契约。外部契约应该是稳定的 semantic refs。

### 2. Typed Intent：稳定分析动作，而不是开放式拼装

`typed intent` 解决的是“允许做什么分析动作”。

Marivo 不把分析系统设计成“给 agent 一组工具，让它自己自由拼装任意执行路径”。它更接近一个类型化分析动作面，让 agent 在有限、明确、可组合的语法中工作。

原子意图覆盖最小分析问题，派生意图封装高频、确定性展开的分析套路。两类 intent 一起构成了 agent 的稳定分析语法。

### 原子意图

| Intent | 它回答什么问题 | 典型输入 | 在分析链路中的作用 |
|--------|----------------|----------|--------------------|
| `observe` | “当前值、分布、时间序列或样本情况是什么？” | `metric`、`scope`、`time_scope`、可选 `dimensions` | 作为最基础的观测动作，为后续比较、检测、验证提供事实输入 |
| `compare` | “两个观测之间差了多少，差异方向和幅度是什么？” | `left_ref`、`right_ref` | 把两个已存在观测对齐成类型化差异结果，是归因和验证前的关键桥梁 |
| `decompose` | “一个已知变化主要由哪些维度成员贡献？” | `compare_ref`、`dimension` | 把总变化拆成结构化贡献项，支持渠道、地区、品类、端类型等归因分析 |
| `correlate` | “两个序列是否存在统计关联？” | `left_ref`、`right_ref` | 用于发现共变关系，帮助筛选候选解释，但不直接等价于因果结论 |
| `detect` | “在给定时间范围内，哪里出现了异常？” | `metric`、`time_scope`、`strategy`、可选 `scope`、`dimension` | 先定位异常窗口或异常切片，为后续 `compare` / `diagnose` 固定分析起点 |
| `test` | “这个明确假设在统计上是否成立？” | `metric`、`left`、`right`、`kind: "numeric"`、`hypothesis` | 把候选解释从直觉判断推进到可检验结论，减少把波动误当成规律 |
| `forecast` | “基于已有序列，未来一段时间可能如何演化？” | `source_ref`、`horizon` | 用于趋势外推、基线预估或后续偏离识别，不替代因果诊断 |

### 派生意图

| Intent | 它封装的完整分析动作 | 典型展开方式 | 在分析链路中的作用 |
|--------|----------------------|--------------|--------------------|
| `attribute` | “解释一次已知变化主要由谁贡献” | `observe -> compare -> decompose` | 把高频归因套路固化下来，减少 agent 手工拼分析步骤的成本 |
| `diagnose` | “围绕一次异常做系统化排查” | `detect -> compare -> decompose`，必要时追加后续步骤 | 适合从异常发现直接进入诊断，形成更完整的排查链路 |

这组设计有几个直接好处：

- `observe`、`compare`、`decompose` 让 agent 能围绕“看事实、量变化、拆贡献”稳定推进，而不是反复生成临时 SQL。
- `detect`、`correlate`、`test`、`forecast` 让系统具备异常识别、关联判断、统计验证和趋势外推等能力，但每一类动作都有清晰边界。
- `attribute`、`diagnose` 把高频分析套路固化为可审计的宏动作，减少流程拼装成本，而不是减少严肃分析所需的证据步骤。

这层设计带来几个直接能力：

- Agent 可以把自然语言业务问题翻译成低歧义的类型化分析请求。
- 上下游步骤可以通过 typed refs 相互引用，而不是每一轮都重复描述 scope 和语义。
- 系统可以在执行前做预检、兼容性、治理和质量闸门检查。
- 结果可以按意图类型稳定组织，便于下游证据抽取和 session state 更新。

Typed intent 的目标不是减少所有步骤，而是减少无意义的流程拼装成本，把模型预算留给路径选择和业务解释。

### 3. Evidence Engine：结构化证据，而不是结果文字

`evidence engine` 解决的是“结果如何成为证据”。

Marivo 不满足于“跑出一张结果表，再让模型总结一下”。它要求每一步分析结果都能沉淀为结构化证据，并进入一条明确的规范证据链：

`artifact -> finding -> proposition -> assessment -> action proposal`

这条链路里，各层职责不同：

- `artifact`：typed intent 的完整执行结果，是回放、审计、来源追踪的权威输入。
- `finding`：从 artifact 中由代码确定性抽取的事实单元。
- `proposition`：系统当前要判断什么。
- `assessment`：当前判断进展到了什么程度，哪些证据支持，哪些缺口未补。
- `action proposal`：基于最新 assessment 生成的下一步建议，但不替代判断层。

Evidence engine 的关键能力包括：

- 事实抽取由代码确定性完成，避免把证据结构交给模型即兴生成。
- 让 findings、propositions、assessments 可以被后续 typed intents 和读取面直接引用。
- 通过规则族、证据缺口、判断策略维持“哪些结论已被支持，哪些仍不成立”。
- 让 session state 和 proposition context 成为机器可读的决策面，而不是只依赖叙述性摘要。

这使得 Marivo 输出的不是一段“像分析”的文本，而是一组可持续推进的证据对象。

## Analysis Session：把三层能力组织成真正的分析运行时

Marivo 真正重要的，不是这些组件分别存在，而是它们通过 `analysis session` 被组织成一个连续工作的运行时。

在 Marivo 中，分析不是一次性的请求-响应，而是一个有状态的分析过程。Session 是分析任务的根容器，它承载：

- 当前任务目标和问题上下文。
- 治理边界和执行预算。
- typed intent 的写入归属。
- 规范化证据的归属关系。
- 持续推进的 session state。

这意味着 agent 不需要靠长对话硬记“前面查了什么”。它可以在 session 中工作：

- 通过 semantic layer 选择稳定业务对象。
- 通过 typed intents 执行受约束分析动作。
- 通过 evidence engine 沉淀 findings、propositions、assessments。
- 通过 state surface / context surface 消费已有证据，再决定下一步。

这也是 Marivo 和很多“带数据工具的 agent”最本质的差异之一。它不只是让 agent 能访问数据，而是让 agent 能在一个有状态、有证据、有约束的运行时中完成分析。

## 示例案例一：观看时长下降分析

设想一个内容平台场景：业务方提出问题，“过去一周观看时长明显下降，主要问题出在哪里？”

如果用 RAG/Text2SQL 路线，系统通常会尝试检索“观看时长对应哪些表”，生成几条 SQL，先查总体走势，再查渠道，再查内容类型。结果也许能查出来，但整个过程本质上还是一连串离散查询。系统并不知道哪些结果构成了同一个分析任务，哪些比较口径已经对齐，哪些候选解释只是印象，哪些已经形成了正式证据。

在 Marivo 里，这个问题会被组织成一个 analysis session。

### 第一步：建立 session，固定任务边界

Agent 先创建 session，把目标定义为“排查观看时长下降”。从这一刻开始，后续 typed intents、artifacts、findings 都属于同一个分析上下文。

这里重要的不是“开了一个会话”，而是系统获得了一个正式的分析根对象。后续每一步都不再是孤立请求，而是 session 内持续推进的状态更新。

### 第二步：通过 semantic layer 固定分析对象

Agent 不需要先猜表和字段，而是直接围绕语义对象工作：

- 度量是 `metric.watch_time`
- 时间范围是异常周及其基线周期
- 候选拆解轴可能是渠道、内容类型、地区、端类型等 `dimension.*`

这时 semantic layer 提供的价值，不是“帮助写 SQL”，而是：

- 保证 `watch_time` 的业务口径稳定。
- 保证时间绑定方式可执行。
- 保证拆解维度是运行时可消费的语义轴。
- 在执行前就能发现 readiness 或 binding 缺口。

### 第三步：用 typed intents 推进分析链路

Agent 可以先执行 `detect`，让系统识别观看时长的异常窗口。接着执行 `compare`，把异常窗口和基线周期对齐比较，确认下滑幅度。然后执行 `decompose`，查看各渠道、内容类型或地区对总差异的贡献。必要时再用 `attribute` 或 `diagnose` 把高频分析套路展开成稳定的组合动作。

这里的关键不是“系统提供了几个 API”，而是：

- 每一步都是明确分析动作，不是任意查询。
- 每一步都有类型化输入输出。
- 后续动作优先引用上游结果，而不是重复描述相同分析条件。

### 第四步：用 evidence engine 把结果变成可复核证据

每次 typed intent 执行后，结果不会只停留在结果表里。系统会把执行结果沉淀为 artifact，并进一步确定性抽取 findings，例如：

- watch time 在异常周相对基线周期下降了多少。
- 哪个渠道贡献了最大负向差异。
- 哪个内容类型的变化只占局部，不足以解释整体下滑。

这些 findings 会继续 seed propositions 和 assessments，例如：

- “主要下降来自推荐流量收缩”是否已被支持。
- “某类内容供给下降是主因”是否仍存在 gap。

最终 agent 读取的不是一段模糊总结，而是一个有状态的证据面：哪些解释已经被支持，哪些解释被削弱，哪些问题还需要继续查。

### 第五步：得到的是可靠分析结论，而不是一次性答案

因此，Marivo 在这个案例里交付的不是“模型猜到了根因”，而是一个更强的东西：

- 稳定的业务对象。
- 稳定的分析动作。
- 连续的 session 状态。
- 可继承、可复核的证据链。

这让观看时长下降分析从“一串 SQL + 一段总结”，升级为“一个可持续推进的分析运行时”。

## 示例案例二：GMV / 营收波动分析

再看一个更典型的经营分析问题：“本月 GMV 明显低于上月，主要由哪些区域和品类驱动？某次促销流量变化是否是关键原因？”

这种问题在业务里非常常见，也最容易暴露 Text2SQL 路线的短板。因为这里不只是查一个数，而是要同时处理口径、对比、归因、假设验证和后续追查。

### 第一步：session 固定的是一个经营问题，而不是一条查询

Agent 创建 session，把问题定义为解释某周期 `metric.gmv` 的下滑。这样后续所有动作都围绕同一个经营分析任务展开，而不是每次重新发问“再查一下 region”“再查一下 category”。

### 第二步：semantic layer 保证经营口径稳定

在 GMV 场景里，语义层尤其重要。因为 GMV、订单数、支付用户数、region、category、channel 往往跨多个物理表和历史口径。

Marivo 通过 semantic layer 提供的是：

- GMV 的稳定度量契约。
- 区域、品类、渠道等共享分析轴。
- 与实体归属一致的拆解边界。
- 可执行的 bindings 和 readiness gating。

这避免了 agent 在多轮分析中不断切换“到底按支付口径还是下单口径”“地区归属按用户侧还是订单侧”的问题。

### 第三步：typed intents 串联归因与验证

一个典型链路可能是：

- `observe` 当前周期和上周期的 `metric.gmv`
- `compare` 确认差异幅度
- `decompose` 按 `dimension.region` 和 `dimension.category` 看贡献结构
- `test` 检查“促销流量下降是否显著影响 GMV”这样的候选假设

这里的区别在于，系统不是让 agent 用自然语言说“帮我看看是不是促销导致的”，然后临时生成一条 SQL 试试，而是把“观察”“比较”“拆解”“验证”都放进明确的类型化分析契约。

### 第四步：evidence engine 区分观察与判断

在 GMV 案例里，最容易混淆的是“观察到变化”和“确认了原因”。

Marivo 的 evidence engine 会显式区分：

- 哪些 finding 只是说明某区域、某品类、某渠道出现了负向变化。
- 哪些 proposition 已经被足够证据支持。
- 哪些 assessment 仍处于 gap 未闭合状态。

于是最终输出不会把“region A 的下滑贡献最大”直接偷换成“region A 的市场策略失效就是主因”。前者是观察，后者是判断。系统会把这两层严格分开。

### 第五步：经营分析结果因此更可信

这个案例展示的是，Marivo 并不是“更会写经营 SQL”，而是更适合承接经营分析所需的结构：

- 统一口径
- 类型化动作
- 假设验证
- 证据分层
- 持续推进的 session state

这正是经营数据分析从演示走向生产时真正需要的能力。

## 示例案例三：DAU / 留存异常分析

第三类问题更接近增长团队的日常：“某一时间段 DAU 或留存出现异常，和版本发布、渠道结构、端侧变化之间是否有关？”

这类问题的难点在于，它通常不是一次观察就能结束，而是需要多轮排查和多条候选路径并行推进。如果系统没有 session 和证据链，很容易在第三轮之后失去上下文控制。

### 第一步：从异常识别开始，而不是先写假设 SQL

Agent 创建 session 后，可以先对 `metric.dau` 或相关 retention metric 执行 `detect`，识别异常时间窗口。这个动作的价值在于，系统先把“异常发生在哪”固定下来，再进入后续解释阶段。

如果直接走 Text2SQL，模型通常会先假设一个时间段，或者用一条自写 SQL 去扫趋势；即使结果大体正确，也很难保证后续所有比较都围绕同一异常窗口展开。

### 第二步：typed intents 支持多轮关联和拆解

在异常窗口固定之后，agent 可以继续：

- 用 `correlate` 看 DAU 与版本发布后某行为指标、某渠道流量、某端类型指标之间是否存在统计关联。
- 用 `decompose` 查看异常主要集中在哪些用户段、渠道或端类型。
- 用 `test` 对候选解释做进一步检验。

这一串动作的价值，不是“流程更多”，而是“每一步都留下正式可引用的中间结果”。当 agent 进入第四轮第五轮时，不必重新靠提示词记忆“之前哪个渠道已经看过”“哪个相关性只是弱关联”“哪个假设还缺前置证据”。

### 第三步：evidence engine 让排查过程可积累，而不是可遗忘

增长分析里最常见的问题，是对话型系统每往前推进一步，就把前面已知的证据压缩成更短的一段话，最后只剩一份高层叙述。这样一来，系统越查越像在“重新理解问题”，而不是在“积累证据”。

Marivo 反过来做：

- 每次 intent 执行先留下 artifact。
- findings 由代码确定性抽取。
- propositions 和 assessments 持续更新。
- gaps 会显式保留，提示哪些解释仍缺证据。

于是 agent 消费的是一个不断增长、不断收敛的分析状态，而不是一段越来越模糊的聊天摘要。

### 第四步：增长场景因此更适合用 session 驱动分析

这个案例特别能说明 analysis session 的价值。因为增长分析不是“查一个答案”，而是“在多个候选解释之间持续缩小空间”。没有 session，系统只能做离散问答；有了 session，系统才真正具备排查和诊断能力。

## Marivo 不是什么

Marivo 不是“把 LLM 接到数据仓库上”的薄封装。

它不是一个自然语言生成 SQL 的工具。

它不是一个基于 RAG / 知识库的 Text2SQL 增强层。

它不是一个只服务于人类交互页面的 BI 动作层。

它也不是一个试图替 agent 做所有开放式推理的黑盒规划器。

Marivo 更像一个中间层：它不代替 agent 进行判断，也不放任 agent 自由执行一切。它做的是把 agent 的分析能力落到一个可复核、可治理、可复用的框架里。

## 架构：从单一服务到双模运行时

上述能力之所以能在不同场景落地，依赖于 Marivo 的五层架构设计：

**Surfaces → Runtime → Core Engine → Ports → Adapters/Profiles**

- **Surfaces** 是 agent 与 Marivo 交互的入口。本地模式通过 MCP stdio 直连，企业模式通过 HTTP API / HTTP MCP 接入。
- **Runtime** 统一协调 session 生命周期、intent 调度和证据流转。
- **Core Engine** 是纯域逻辑层，封装语义解析、intent 执行、证据抽取等核心能力，零 I/O 依赖，可独立测试和演进。
- **Ports** 定义 Core 与外部世界的契约边界。
- **Adapters/Profiles** 把 Ports 对接到具体基础设施（SQLite / MySQL、DuckDB、本地文件 / 对象存储等）。

**Profile 系统**让同一个 Runtime 可以搭配不同的 Adapter 组合：本地模式使用 SQLite + DuckDB + `.marivo/` 工作区，企业模式可切换到 MySQL 元数据 + 集中式存储，而 Core Engine 的分析逻辑完全不变。

这意味着开发者可以在本地用轻量配置快速验证分析链路，再无缝切换到企业部署，而不需要重写任何分析逻辑。

## 为什么开发者应该关注 Marivo

如果你在构建智能体数据分析系统，真正困难的通常不是“让模型会回答”，而是“让分析结果能被相信、被复用、被治理”。Marivo 提供的价值，正是在这条路径上。

它让分析动作更稳定，让业务语义更统一，让结果表达更接近证据，让分析过程更适合持续推进，也让 agent 更容易从演示走向生产。双模部署架构让 Marivo 既能嵌入本地 agent 工作流做快速分析，也能作为企业级分析平台支撑集中治理，而核心分析能力始终一致。

面向生产的智能体数据分析，需要的不是放任式生成，也不是一层又一层叠在 Text2SQL 外面的知识检索，而是一个能把智能收束为可靠分析过程的约束框架。Marivo 的价值就在这里。
