description: Ask an agent to draft a first semantic layer, review the business contract, then continue to analysis.
---

## Before you ask the agent

Create or initialize a project with `marivo init` (see
[Installation](/en/latest/installation/#initialize-the-project)). A minimal
project has a manifest plus datasource and semantic declaration directories:

```text
your-project/
  marivo.toml
  models/
    datasources/
    semantic/
```

Ask an agent that can work in this project to use the packaged
`marivo-semantic` skill. The agent follows an evidence-bounded authoring route;
you remain responsible for approving the business meaning it encodes.

## What you need to decide

Before a metric is used, you must confirm its business contract. For completed-order
revenue, that includes:

- what revenue means and does not mean;
- which rows are in the population and which filters select them;
- how cancellations, refunds, test orders, and other exclusions are treated;
- which time axis and timezone apply;
- the currency or other unit, additivity, and valid breakdowns;
- known limitations and the accountable owner.

Column names and sampled values can support a question, but they cannot approve any
of these decisions for you. It is fine to start with unknowns: the agent should ask
one evidence-grounded business question when a required decision is missing.

## What to ask the agent to do

Use a request that names the business goal, constrains evidence collection, and sets
a review stop:

```text
Use the marivo-semantic skill to model the orders table in the sales domain.

The business goal is completed-order revenue analysis. Start by inspecting the
source and choose an explicitly bounded sample. Draft one semantic object at a
time. Do not infer the revenue population, refund treatment, currency, time
axis, additivity, or allowed breakdowns from column names or sampled values.
Ask me one evidence-grounded business question whenever a required decision is
missing. After each object, verify it, preview it with the same snapshot, and
show readiness. Stop before analysis so I can review the proposed metric and
guardrails.
```

## How the agent builds one object at a time

The `marivo-semantic` route is:

```text
help/browse -> inspect -> explicit scope -> sample once -> project evidence -> settle/grill -> author one Python object -> load typed object -> static verify -> scoped preview -> readiness -> analysis
```

The agent first reads `md.help("authoring")` and `ms.help("authoring")`, browses
existing typed catalog objects, and uses `md.inspect(...)` to learn the source's
physical extent, partitions, schema, and execution capabilities. It then chooses
`md.partition(...)` or explicitly acknowledges a broad read with `md.unpruned(...)`;
both have positive row and timeout guards.

One call to `inspection.sample(...)` creates the snapshot for the active authoring
batch. Query-free snapshot projections provide entity, dimension, value, time, and
measure evidence. The agent settles only what help, project facts, source provenance,
prior decisions, and that evidence support. If a semantic choice is still missing,
it asks exactly one grounded question and stops. It writes one explicit Python object,
reloads its typed `CatalogObject`, then verifies, previews, and checks readiness before
moving to the next object.

## What the agent writes to the project

After you answer the required business questions, the agent writes explicit Python
declarations under `models/`. For this illustrative example only, suppose the user has
reviewed and approved every modeling choice below. This does not describe approval by
a real business owner, and Mina Zhang is a fictional owner placeholder:

- `order_id` is a unique, non-null primary key, so `orders` has one row per completed
  order;
- cancelled orders are excluded, refunded orders remain, and `amount` is their CNY
  value net of refunds;
- missing `region` values are not a valid region;
- `order_date` at day granularity is the default reporting axis;
- `amount` is additive across the approved time axis and breakdown; and
- `region` is the only approved breakdown for this example; no other breakdown is
  implied.

```python
# models/datasources/warehouse.py
import marivo.datasource as md
import marivo.semantic as ms

md.duckdb(
    name="warehouse",
    path="warehouse.duckdb",
    ai_context=ms.ai_context(
        business_definition="Local DuckDB warehouse for approved sales analysis.",
        guardrails=["Use only for development or approved local analysis."],
    ),
)
```

```python
# models/semantic/sales/_domain.py
import marivo.datasource as md
import marivo.semantic as ms

ms.domain(
    name="sales",
    owner="Mina Zhang",
    ai_context=ms.ai_context(
        business_definition="Completed-order sales analysis domain.",
        guardrails=["Use only the user-approved completed-order population."],
    ),
)

orders = ms.entity(
    name="orders",
    datasource=md.ref("datasource.warehouse"),
    source=md.table("orders"),
    primary_key=["order_id"],
    ai_context=ms.ai_context(
        business_definition="One row per completed sales order.",
        guardrails=["Cancelled orders are excluded; refunds remain reflected in amount."],
    ),
)

region = ms.dimension_column(
    name="region",
    entity=orders,
    column="region",
    ai_context=ms.ai_context(
        business_definition="Sales region assigned to the completed order.",
        guardrails=["Do not treat missing region as a real region."],
    ),
)

order_date = ms.time_dimension_column(
    name="order_date",
    entity=orders,
    column="order_date",
    granularity="day",
    is_default=True,
    ai_context=ms.ai_context(
        business_definition="Approved reporting date for completed-order revenue.",
        guardrails=["Use this as the default time axis for this revenue metric."],
    ),
)

amount = ms.measure_column(
    name="amount",
    entity=orders,
    column="amount",
    additivity="additive",
    unit="CNY",
    ai_context=ms.ai_context(
        business_definition="Completed-order amount in CNY, net of cancellations and refunds.",
        guardrails=["Do not use for gross bookings or pre-refund revenue."],
    ),
)

revenue = ms.aggregate(
    name="revenue",
    measure=amount,
    agg="sum",
    ai_context=ms.ai_context(
        business_definition="Total completed-order revenue in CNY.",
        guardrails=["For this example, break down only by sales region."],
    ),
)
```

The agent also runs the evidence handoff against the same bounded snapshot. This is
review evidence, not another source of business approval:

```python
import marivo.datasource as md
import marivo.semantic as ms

warehouse = md.ref("datasource.warehouse")
inspection = md.inspect(warehouse, md.table("orders"))
snapshot = inspection.sample(
    scope=md.unpruned(max_rows=1000, timeout_seconds=30),
    columns=("order_id", "region", "order_date", "amount"),
)

catalog = ms.load()
sales = catalog.domains.get("sales")
orders = sales.entities.get("orders")
revenue = orders.metrics.get("revenue")

catalog.verify_object(revenue).show()
catalog.preview(revenue, using=snapshot).show()
report = catalog.readiness(refs=[revenue])
report.show()
```

## What to review before continuing

Review the proposed declarations together with the verification, preview, and readiness
output. Confirm that:

- the definition and non-definition match the intended decision;
- the row population, filters, exclusions, cancellations, and refunds are explicit;
- the time axis, timezone, unit, and additivity are correct;
- every allowed breakdown is valid and unsupported breakdowns are constrained;
- limitations and guardrails are visible, and the named owner accepts them;
- the preview used the same explicitly bounded snapshot; and
- scoped readiness has no blockers for the metric's dependency closure.

A ready report is a technical handoff result. It does not mean a business owner has
approved the metric definition, filters, exclusions, units, or intended use.

## Continue to analysis

After the business owner approves the contract and scoped readiness is not blocked,
hand the ready metric to the `marivo-analysis` skill. See
[Analysis Workflow](/en/latest/concepts/analysis-workflow/) for the session and
observation route.

For deeper reference material, read [Semantic Layer](/en/latest/concepts/semantic-layer/),
[Readiness](/en/latest/concepts/readiness/), and
[Project Configuration](/en/latest/reference/project-configuration/).
