## When the agent runs readiness

Readiness is Marivo's explicit certification and diagnostic check for selected
semantic objects. Run it after authoring or changing an object, or when a
workflow asks for technical certification. It identifies incomplete
objects, unknown references, and missing current validation or preview evidence.

A blocked report does not mean the analysis question is invalid. It means the
selected objects have not completed this certification. Repair the named gap
before declaring the semantic change complete.

Analysis APIs do not invoke readiness automatically. Routine analysis of
unchanged objects relies on live catalog loading, semantic input resolution,
planning, and runtime execution checks instead of refreshing authoring snapshots
before every call.

## How to respond

Most readiness issues belong to the agent: it can repair declarations, acquire
the relevant evidence, and rerun the scoped check through `marivo-semantic`.
You only need to step in when the fix requires a business decision, such as the
meaning of a metric, an inclusion rule, or the correct time axis.

Confirm or correct that business choice, then let the agent continue. You do
not need to request validation, preview, dependency traversal, or report fields
individually.

## What ready means

`ready` means the selected semantic objects have current technical certification
evidence for the requested scope. It is not a runtime authorization token: it
does not approve the business definition, guarantee every future query, or
decide whether a conclusion is suitable for a business decision.

The sections below document the report and API behavior for readers who need
to inspect the technical contract.

## Zero-query closeout after scoped runtime evidence

`catalog.readiness(refs=[...])` is a **pure in-memory check**. It never opens a datasource
connection — it reasons about the declarations themselves: missing definitions,
unknown refs, fan-out across datasources, fragile string references, and SQL
parity status. It does not inspect GitHub/GitLab approval state and does not
make audit-approval judgments.

Backend-touching validation is separate and runs on demand:

| Check | What it does | Touches backend? |
| --- | --- | --- |
| `catalog.verify_object(obj)` | Static object validation | No |
| `catalog.preview(obj, using=snapshot)` | Executes the scoped runtime smoke check and records metadata, never rows | Yes |
| `catalog.readiness(refs=[revenue])` | Consumes matching verification/preview evidence | No |
| `ms.parity_check(ref)` | Optional, potentially unbounded metric/provenance SQL diagnostic | Yes |
| `ms.richness()` | Advisory coverage/depth report; never blocks | No |

:::caution[Readiness does not prove the query runs]
A `ready` report means the object passed static validation and has matching
scoped preview evidence. Snapshot and preview timestamps remain reference
metadata; age alone does not block readiness or trigger reacquisition. It does
not prove every future analysis query or full-source
truth. Parity is never required for readiness.
:::

## The report

`catalog.readiness(refs=[...])` returns a frozen `ReadinessReport`:

| Field | Type | Meaning |
| --- | --- | --- |
| `status` | `"ready" \| "ready_with_warnings" \| "blocked"` | Overall verdict (see below). |
| `blockers` | `tuple[ReadinessIssue, ...]` | Issues that prevent the selected refs from completing certification. |
| `warnings` | `tuple[ReadinessIssue, ...]` | Advisory issues that do not block. |
| `analysis_ready_refs` | `tuple[str, ...]` | Refs that passed the current certification contract. |
| `input_summary` | `ReadinessInputSummary` | What was checked: datasources, refs, and tables. |
| `checked_at` | `str` | ISO-8601 timestamp of the check. |
| `analysis_handoff` | `SemanticToAnalysisHandoff \| None` | Typed handoff to the analysis surface; `None` when blocked or no ref is ready. |

`status` is derived from severity: any blocker ⇒ `blocked`; warnings only ⇒
`ready_with_warnings`; otherwise `ready`.

Each `ReadinessIssue` carries a typed `repair` instead of free-form next-step
text, so the next action is machine-readable and points at the owning live help
target:

| Attribute | Meaning |
| --- | --- |
| `kind` | The issue kind (see table below). |
| `severity` | `"blocker"` or `"warning"`. |
| `refs` | The semantic refs the issue applies to. |
| `message` | What was expected and what was found. |
| `repair` | A typed `AuthoringRepair \| None` carrying `action`, `help_target`, optional `snippet`, and `candidates` — the concrete next step and where to read its contract. |

`report.show()` prints a bounded card; `report.to_dict()` returns the full machine
list for programmatic handling; `report.contract()` returns the mechanical
continuation contract, including the `analysis_handoff` transition when refs
passed certification.

Optional `guardrails` enrichment is reported by readiness and richness; it does
not block certification.

## Issue kinds

| Kind | Severity | Trigger | Next step |
| --- | --- | --- | --- |
| `load_error` | blocker | A declaration failed to load. | Fix the semantic load errors and reload. |
| `unknown_ref` | blocker | A requested ref is not in the registry. | Fix the ref or remove it from the readiness scope. |
| `missing_business_definition` | blocker | A checked object has no `ai_context.business_definition`. | Add a `business_definition`. |
| `cross_datasource_unfederated` | blocker | A metric spans more than one datasource without federation. | Move integration upstream, enable a federated backend, or split the metric. |
| `missing_guardrails` | warning | A checked metric, entity, dimension, measure, or time dimension has no `ai_context.guardrails`. Guardrails are recommended enrichment, not a certification requirement. | Add `guardrails` when explicit usage constraints would help. |
| `sql_parity_unverified` | warning | A metric has `provenance` SQL but parity is unconfirmed. | Run `ms.parity_check(ref)`. |
| `fragile_string_ref` | warning | An object uses a fragile string reference. | Replace it with a stable object ref. |
| `time_dimension_pushdown_advisory` | warning | A time dimension may not push down to the partition. | Keep the raw partition column and declare its parse if the business axis matches it. |
| `snapshot_missing` | blocker | No structurally matching datasource snapshot metadata exists for an executable ref. Expiration alone never triggers this issue. | Acquire a matching scoped snapshot. |
| `runtime_preview_missing` | blocker | No preview check matches the current semantic definition, dependencies, snapshot ids, backend, and scopes. Expiration alone never triggers this issue. | Run the scoped preview with the matching snapshot bindings. |

## Scope the check

Pass typed catalog objects through `refs=` to scope readiness to the objects
being authored, changed, or explicitly certified. Marivo resolves the full dependency closure (the
metric, its measures, dimensions, entities, and datasources) for you:

```python
revenue = catalog.get("metric.sales.revenue")
report = catalog.readiness(refs=[revenue])
for issue in report.blockers:
    print(issue.kind, issue.refs, "→", issue.repair.action if issue.repair else "")
```

Use this report to close out semantic changes or satisfy a stricter workflow's
certification requirement. Routine analysis does not call it automatically. For
how analysis records what it concludes, see [Evidence](/en/latest/concepts/evidence/).

## Typed handoff to analysis

When readiness is not blocked and at least one requested ref is analysis-ready,
the report carries a typed `SemanticToAnalysisHandoff` on `analysis_handoff`.
Its project, catalog, and environment fingerprints are computed from the same
index and workspace the analysis-side validator checks, so the handoff proves
the mechanical facts were current at readiness time without carrying any query
result.

The analysis surface has exactly one receiver for this handoff:
`Session.validate_semantic_handoff(handoff)`. It performs no datasource query,
opens no connection, and mutates no state — it re-verifies environment identity,
fingerprints, ref existence and kind, current readiness, warning consistency,
and preview-evidence existence and ownership, then returns an in-memory
`SemanticHandoffReceipt`. A stale fact raises a typed `AnalysisError` with
semantic repair pointing back at the semantic surface; it never silently
falls back.

```python
import marivo.semantic as ms

catalog = ms.load()
revenue = catalog.get("metric.sales.revenue")
report = catalog.readiness(refs=[revenue.ref])
handoff = report.analysis_handoff  # SemanticToAnalysisHandoff | None
# route to analysis: session.validate_semantic_handoff(handoff) -> SemanticHandoffReceipt
```

`report.contract()` exposes the `analysis_handoff` transition as a
mechanically valid next action when refs passed certification, so an agent can
route from readiness to analysis without free-form suggestions.
