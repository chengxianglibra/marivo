接受结论的检查过程应从业务判断开始，再沿着它的假设、artifact 和证据记录向后追溯。
最终数字可以回溯是必要条件；如果指标契约、分析范围、比较基线或拆分维度不适合当前决策，
仅有可回溯的数字仍然不够。

## 检查结论，而不只是最终数字

让智能体先说明结论、适用范围和比较基线，再展示支持细节。你负责判断结论是否回答了经过
确认的业务问题，以及其中的假设是否可以接受。

证据说明 Marivo 如何得到结果，但不会批准指标的业务含义，也不会替你决定结论是否适合
当前用途。

## 你需要决定什么

接受结论前，请判断：

- 指标定义是否代表你关心的业务结果；
- 分析群体、排除项、窗口和粒度是否与决策一致；
- 比较基线是否能让报告的变化得到正确解释；
- 拆分维度是否有效、有用，并且符合已编写的使用约束；
- 质量问题或阻塞项是否仍允许结论获得足够支持；
- 已说明的注意事项是否把剩余不确定性控制在你可以接受的范围内。

## 可复制的智能体请求

```text
请使用 marivo-analysis skill 回答这个问题。先给出结论，再展示实际使用的指标契约、分析范围和窗口、比较基线、每个重要拆分维度、来源 artifact、证据状态、阻塞项、注意事项，以及支撑每项重要判断的 session.evidence trace。如果发现语义缺口，请停止分析并把问题交给 marivo-semantic，不得改写契约或替换指标。
```

## 让智能体展示什么

要求智能体先给出结论，并展示：

- 实际使用的指标定义和已编写的使用约束；
- 当前期与基线期的分析范围、窗口、粒度和比较基线；
- 每个重要拆分维度，以及选择它的理由；
- 来源 artifact，包括相关的 `artifact.show()` 输出；
- 证据不完整、不可用或受质量限制时的 `artifact.meta.evidence_status`、
  `artifact.meta.blocking_issues`、`artifact.meta.confidence_scope` 和
  `artifact.meta.quality_summary`；
- 需要跨步骤综合时的 `session.knowledge()`，以及需要完整审计时的
  `session.evidence` trace；
- 假设、排除项、注意事项、未解决问题和被放弃的分支。

## 接受结论前需要检查什么

按下面的顺序检查证据：

1. **指标契约：**业务定义、单位、聚合行为和使用约束与你的预期一致。
2. **范围与窗口：**分析群体、排除项、当前期、粒度和时间边界正确。
3. **比较基线：**基线与对齐方式能够让差异得到正确解释。
4. **拆分维度：**维度和选中的分组有效，不会把局部规律表述成普遍结论。
5. **质量与阻塞项：**结论的置信范围和措辞反映了不完整的证据、数据质量限制或被阻塞的
   后续操作。
6. **来源与注意事项：**每项重要判断都指向来源 artifact 和审计记录，并明确说明假设和剩余
   不确定性。

任何一项检查不通过时，都应拒绝或缩小结论。只有新的分析分支能够解决这个明确缺口时，才
要求智能体继续调查。

## 业务口径有误时让智能体返回修正

如果指标定义、维度、关系、时间维度、单位、使用约束或其他语义契约有误或不完整，请停止
调查，明确指出缺口，并交给 `marivo-semantic` 起草或完成技术验证。分析恢复前，仍须由你
检查并确认其业务含义。

`marivo-analysis` 不得为了挽救结论而改写语义、编造替代指标，或重新解释物理字段。使用错误
的契约重复运行 operator，只会为错误的问题产生更多证据。

## 证据模型与审计参考

每个 analysis operator 都会把 evidence 写入 session。这些 evidence 让 agent 不只说明*发生了什么*，
也能说明它*如何*得出结论；人工审阅者可以沿着这条链追溯到产生结果的输入。

这些记录是 append-only 的，并持久化在 session 的 judgment 存储中（位于 project 本地的
`.marivo/`）。因此，session 结束后，结论仍然可审计、可重放。

当 operator 产生 evidence 时，`artifact.show()` 会在结果 preview 之前直接展示有界的
commit-time evidence 摘要。理解当前步骤只需一次 `show()`；回顾或综合多个步骤时使用
`session.knowledge()`，需要完整 typed records 与审计链时使用 `session.evidence`。

## 三个层次

证据链分三个层次记录，从原始观察到带版本的判断：

| 层次 | 它是什么 | 关键字段 |
| --- | --- | --- |
| **Finding** | 从单个 artifact 中提取的原子、类型化观察（例如“West 区域 revenue 环比下降 12%”）。 | `finding_type`、`artifact_id`、`subject`、`observed_window`、`quality_status` |
| **Proposition** | 由系统播种、关于某个 subject 的论断，从 findings 派生（例如“Q4 的 revenue 发生了变化”）。 | `proposition_type`、`subject_key`、`seed_finding_refs` |
| **Assessment** | 对某个 proposition 的、带版本的判断：它的状态与置信度。 | `status`、`confidence`、`confidence_basis`、`supersedes_id`、`is_latest` |

Findings 与 propositions 是不可变的。重新运行 operator 不会原地修改判断，而是写入一条
**新的** assessment 来取代上一条（`supersedes_id`），并把最新的一条标记为 `is_latest`。
完整的判断历史会被保留下来。

## 读取 session 已知的事实

`session.knowledge()` 返回一个不可变的 `SessionKnowledge` 快照。这是 agent 用来决定下一步的
读取视图，承载**观察摘要**、**已确定的 facts** 与**未决的 open items**：

```python
knowledge = session.knowledge()

# 观察到了什么：每次 observe / derive_metric_frame commit 对应一条有界 digest。
for obs in knowledge.observations():
    print(obs.subject.metric, obs.semantic_kind, obs.analysis_purpose, obs.digest)

# 已确定的 facts：带有当前 assessment 的 propositions。
for fact in knowledge.facts(kind="driver"):
    print(fact.dimension, fact.contribution_share, fact.status)

# Open items：等待跟进的未决 anomalies 与 questions。
for item in knowledge.open_items(kind="anomaly"):
    print(item.subject, item.confidence)

# 建议的下一步 operators，已排序。
for action in knowledge.next_steps(top=3):
    print(action)
```

`observations()` 按 commit 顺序返回有界 digest：按 shape 分派的取值摘要（scalar 的值；
时间序列的首末/极值/方向；segmented 按绝对值排序的 top segments），外加窗口、行数和
`analysis_purpose`。合计与占比属于 composition 语义，仅对可加（additive）指标填充；
非可加指标（ratio、percentile、sampled fold 等）保留逐段取值但不给出合计/占比，
非可加的 panel 也不做跨桶的 segment 排名。观察是 ground truth 而不是待检验主张，
因此不携带 status 或 confidence。

Facts 分为五种，每种由一个 operator 产生：

| Fact 种类 | 由谁产生 | 携带 |
| --- | --- | --- |
| `change` | `compare` | 方向、幅度、对比窗口 |
| `driver` | `attribute` | dimension、贡献占比、贡献角色 |
| `tested_hypothesis` | `hypothesis_test` | 方法、alpha、p 值、是否拒绝原假设 |
| `forecast` | `forecast` | 预测窗口、horizon、预测区间 |
| `association` | `correlate` | 相关系数、lag、join 依据 |

`SessionKnowledge` 还提供 `open_items(kind="question")`、`blocked_followups()`（在某个问题
解决前无法运行的 follow-ups），以及 `for_subject(subject)` 将整个快照过滤到单个 subject。
`evidence_completeness` 报告底层 evidence 是 `complete`、`partial` 还是 `unavailable`。

## 审计一个结论

要获取原始记录与完整审计链，使用 `session.evidence` 命名空间：

| 方法 | 返回 |
| --- | --- |
| `session.evidence.findings(...)` | Findings，可按 `artifact_id`、`finding_type`、`subject` 过滤。 |
| `session.evidence.propositions(...)` | Propositions，可按 `proposition_type`、`subject`、`status` 过滤。 |
| `session.evidence.assessments(...)` | Assessments，默认仅返回最新的。 |
| `session.evidence.proposition(id)` | 按 id 取单个 proposition。 |
| `session.evidence.latest_assessment(id)` | 当前 assessment；若从未评估则返回 `None`。 |
| `session.evidence.trace(id)` | 某个 proposition 的完整 `EvidenceTrace`。 |

`trace(id)` 是审计入口：`EvidenceTrace` 会把 proposition 连接到它的 `latest_assessment`、
`seed_findings`、判断背后的 `support_findings` 与 `oppose_findings`，以及产生它的
`source_artifacts` 与 `source_steps`。

```python
props = list(session.evidence.propositions(proposition_type="change"))
trace = session.evidence.trace(props[0].proposition_id)

print(trace.proposition.subject_key, "→", trace.latest_assessment.status)
for finding in trace.seed_findings:
    print(finding.finding_type, "from", finding.artifact_id)
```

人工审阅者可以沿着这条链，从一个结论回溯到支持它的具体观察与分析步骤。这也是 Marivo 结论
“可审计”而不只是“看起来合理”的原因。关于写入这些 evidence 的 operators，参见
[分析流程](/zh-cn/latest/concepts/analysis-workflow/)。
