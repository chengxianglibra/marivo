title: 就绪检查
description: 对新建或修改的语义对象进行显式技术认证。
---

## 智能体什么时候运行 readiness

就绪检查是针对所选 semantic objects 的显式技术认证和诊断。新建或修改 object 后，或者
workflow 要求 technical certification 时运行它。它会识别不完整对象、未知引用，以及
缺少当前 validation 或 preview evidence 的情况。

Blocked report 不代表业务问题不能分析，而是说明所选 objects 存在语义或技术 blocker。
缺失或不匹配的 runtime preview certification 改为 warning：没有其他 blocker 时，披露后可继续
日常分析，但在声明新建或修改的 semantic object 已完成前仍需修复。

Analysis APIs 不会自动调用 readiness。日常分析复用未变化的 objects 时，依赖 live catalog
loading、semantic input resolution、planning 和 runtime execution checks，不会在每次调用前刷新
authoring snapshots。

## 用户怎么处理

大部分就绪问题都由智能体通过 `marivo-semantic` 处理，例如修复声明、取得相关证据并重新
检查。只有修复过程涉及业务判断时，才需要用户参与，例如确认指标含义、纳入规则或正确的
时间轴。

确认或纠正这项业务选择后，让智能体继续即可。用户不需要分别安排验证、预览、依赖检查或
报告字段。

## ready 表示什么

`ready` 表示所选 semantic objects 已经具备当前 scope 所需的技术认证证据。
`ready_with_warnings` 可能包含缺失或不匹配的 preview certification；此时
`analysis_ready_refs` 仍只列出依赖闭包中没有 blocker 的直接请求 refs。两种状态都不是 runtime
authorization token：不代表业务定义已经获批，不保证所有后续查询都能成功，也不会替用户
判断结论是否适合业务决策。

需要检查技术契约的读者，可以继续查看下面的报告和 API 参考。

## Scoped runtime evidence 之后的零查询收尾

`catalog.readiness(refs=[...])` 是一次**纯内存检查**。它从不打开 datasource 连接 —— 它只针对 declarations
本身进行推理：缺失定义、未知 refs、跨 datasource 的 fan-out、脆弱的 string references，以及
SQL parity 状态。它不检查 GitHub/GitLab 审批状态，也不做审计审批判断。

触及 backend 的校验是独立的，并按需运行：

| 检查 | 作用 | 是否触及 backend？ |
| --- | --- | --- |
| `catalog.verify_object(obj)` | 静态 object validation | 否 |
| `catalog.preview(obj, using=snapshot)` | 执行 scoped runtime smoke check，只记录 metadata，不持久化 rows | 是 |
| `catalog.preview(refs=[...], using=snapshot_or_mapping)` | 合并兼容的 runtime smoke checks，同时按 ref 分别记录 evidence | 是 |
| `catalog.readiness(refs=[revenue])` | 读取 matching verification/preview evidence | 否 |
| `ms.parity_check(ref)` | 可选且 potentially unbounded 的 metric/provenance SQL diagnostic | 是 |
| `ms.richness()` | 关于覆盖度/深度的建议性报告；从不 block | 否 |

:::caution[就绪检查不证明查询能跑通]
`ready` 表示 object 已通过 static validation，并且有 matching scoped preview evidence。
Snapshot 和 preview 时间戳只作为参考信息；仅仅时间较旧不会 block readiness，也不会触发
重新采样。它不证明每一种未来 analysis query 或 full-source truth。Parity 从不作为 readiness 要求。
:::

## 报告

`catalog.readiness(refs=[...])` 返回一个 frozen 的 `ReadinessReport`：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `status` | `"ready" \| "ready_with_warnings" \| "blocked"` | 总体结论（见下）。 |
| `blockers` | `tuple[ReadinessIssue, ...]` | 阻止所选 refs 完成认证的问题。 |
| `warnings` | `tuple[ReadinessIssue, ...]` | 不会 block 的建议性问题。 |
| `analysis_ready_refs` | `tuple[str, ...]` | 依赖闭包中没有 blocker 的直接请求 refs。 |
| `preview_required_refs` | `tuple[SemanticRef, ...]` | 缺少或没有 matching preview certification 的 typed executable refs。 |
| `input_summary` | `ReadinessInputSummary` | 检查了什么：datasources、refs、tables。 |
| `checked_at` | `str` | 检查的 ISO-8601 时间戳。 |

`status` 由严重程度推导：只要有 blocker ⇒ `blocked`；只有 warnings ⇒ `ready_with_warnings`；
其余 ⇒ `ready`。

每个 `ReadinessIssue` 都携带一个类型化的 `repair`，而不是自由文本的下一步描述，因此下一步
动作是机器可读的，并指向所属的 live help target：

| 属性 | 含义 |
| --- | --- |
| `kind` | 问题种类（见下表）。 |
| `severity` | `"blocker"` 或 `"warning"`。 |
| `refs` | 问题涉及的 semantic refs。 |
| `message` | 期望是什么、实际是什么。 |
| `repair` | 类型化的 `AuthoringRepair \| None`，包含 `action`、`help_target`、可选的 `snippet` 与 `candidates` —— 即具体的下一步及其契约的查阅位置。 |

`report.show()` 会把多个 `runtime_preview_missing` warning 合并成一条 batch repair 提示，
同时 `report.warnings` 仍保留逐 ref 细节；`report.to_dict()` 返回完整机器可读列表；
`report.contract()` 在缺失 preview evidence 时暴露一个 batch `preview` transition。
通过认证的 refs 直接保留在 `analysis_ready_refs` 中。

## 问题种类

| Kind | 严重程度 | 触发条件 | 下一步 |
| --- | --- | --- | --- |
| `load_error` | blocker | 某个 declaration 加载失败。 | 修复 semantic load 错误并重新加载。 |
| `unknown_ref` | blocker | 请求的 ref 不在 registry 中。 | 修正该 ref，或将其从 readiness 范围中移除。 |
| `missing_business_definition` | blocker | 被检查的对象没有 `ai_context.business_definition`。 | 补充 `business_definition`。 |
| `cross_datasource_unfederated` | blocker | 某个 metric 在没有 federation 的情况下跨越多个 datasource。 | 把集成移到上游、启用 federated backend，或拆分 metric。 |
| `missing_guardrails` | warning | 被检查的 metric、entity、dimension、measure 或 time dimension 没有 `ai_context.guardrails`。Guardrails 是建议性的 enrichment，不是认证必需项。 | 当显式使用约束有帮助时补充 `guardrails`。 |
| `sql_parity_unverified` | warning | 某个 metric 有 `provenance` SQL 但 parity 未确认。 | 运行 `ms.parity_check(ref)`。 |
| `fragile_string_ref` | warning | 某个对象使用了脆弱的 string reference。 | 用稳定的 object ref 替换它。 |
| `time_dimension_pushdown_advisory` | warning | 某个 time dimension 可能无法下推到 partition。 | 当业务时间轴与 partition 一致时，保留原始 partition 列并声明其 parse。 |
| `snapshot_missing` | blocker | executable ref 没有结构上匹配的 datasource snapshot metadata；仅仅过期不会触发。 | 获取 matching scoped snapshot。 |
| `runtime_preview_missing` | warning | 没有 preview check 匹配当前 semantic definition、dependencies、snapshot ids、backend 与 scopes；仅仅过期不会触发。 | 日常分析可以继续；在声明新建或修改的 object 已完成前运行 scoped preview。 |

## 限定检查范围

通过 `refs=` 传入 typed catalog objects，把 readiness 限定到正在 author、修改或明确认证的
objects；Marivo 会解析完整的依赖闭包（metric、它的 measures、dimensions、entities 与
datasources）：

```python
revenue = catalog.get("metric.sales.revenue")
report = catalog.readiness(refs=[revenue])
if report.preview_required_refs:  # authoring closeout 必须修复；日常分析可选
    catalog.preview(refs=report.preview_required_refs, using=orders_snapshot)
    report = catalog.readiness(refs=[revenue])
```

Batch preview 会在连接前校验全部 bindings，合并兼容的 entity/field/measure 与 metric
执行计划，并继续为每个 ref 记录独立、精确的 preview check。Entity preview 不会认证其 children。

使用该 report 完成 semantic changes 的 closeout，或满足更严格 workflow 的认证要求。日常分析
不会自动调用它。关于分析如何记录结论，参见 [证据链](/zh-cn/latest/concepts/evidence/)。

## 继续进入 analysis

当 readiness 未被 block 时，`analysis_ready_refs` 只包含依赖闭包中没有 blocker 的直接请求
refs。Warnings 继续显示在同一份 report 上，调用方必须在决定继续前明确披露；单独的
`runtime_preview_missing` warning 不会阻止日常分析。中间不再创建额外的传递对象。

```python
import marivo.semantic as ms

catalog = ms.load()
revenue = catalog.get("metric.sales.revenue")
report = catalog.readiness(refs=[revenue.ref])
if report.status == "blocked":
    report.show()
    raise SystemExit
ready_refs = report.analysis_ready_refs
```

Readiness 仍需显式调用；analysis operators 不会自动执行它。处理 warnings 后，只把列出的
refs 传给普通 analysis APIs。
