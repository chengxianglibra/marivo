## 智能体什么时候运行 readiness

就绪检查是针对所选 semantic objects 的显式技术认证和诊断。新建或修改 object 后，或者
workflow 要求 fresh technical certification 时运行它。它会识别不完整对象、未知引用，以及
缺少当前 validation 或 preview evidence 的情况。

Blocked report 不代表业务问题不能分析，而是说明所选 objects 尚未完成这次认证。先修复明确的
语义或技术缺口，再声明 semantic change 已完成。

Analysis APIs 不会自动调用 readiness。日常分析复用未变化的 objects 时，依赖 live catalog
loading、semantic input resolution、planning 和 runtime execution checks，不会在每次调用前刷新
authoring snapshots。

## 用户怎么处理

大部分就绪问题都由智能体通过 `marivo-semantic` 处理，例如修复声明、更新相关证据并重新
检查。只有修复过程涉及业务判断时，才需要用户参与，例如确认指标含义、纳入规则或正确的
时间轴。

确认或纠正这项业务选择后，让智能体继续即可。用户不需要分别安排验证、预览、依赖检查或
报告字段。

## ready 表示什么

`ready` 表示所选 semantic objects 已经具备当前 scope 所需的技术认证证据。它不是 runtime
authorization token：不代表业务定义已经获批，不保证所有后续查询都能成功，也不会替用户
判断结论是否适合业务决策。

需要检查技术契约的读者，可以继续查看下面的报告和 API 参考。

## Scoped runtime evidence 之后的零查询收尾

`catalog.readiness(refs=[...])` 是一次**纯内存检查**。它从不打开 datasource 连接 —— 它只针对 declarations
本身进行推理：缺失定义、未知 refs、跨 datasource 的 fan-out、脆弱的 string references，以及
SQL parity 状态。它不检查 GitHub/GitLab 审批状态，也不做审计审批判断。

触及 backend 的校验是独立的，并按需运行：

| 检查 | 作用 | 是否触及 backend？ |
| --- | --- | --- |
| `catalog.verify_object(obj)` | 静态 object validation | 否 |
| `catalog.preview(obj, using=snapshot)` | 执行 scoped runtime smoke check，只记录 metadata，不持久化 rows | 是 |
| `catalog.readiness(refs=[revenue])` | 读取 fresh verification/preview evidence | 否 |
| `ms.parity_check(ref)` | 可选且 potentially unbounded 的 metric/provenance SQL diagnostic | 是 |
| `ms.richness()` | 关于覆盖度/深度的建议性报告；从不 block | 否 |

:::caution[就绪检查不证明查询能跑通]
`ready` 表示 object 已通过 static validation，并且有 fresh matching scoped preview
evidence；它不证明每一种未来 analysis query 或 full-source truth。Parity 从不作为 readiness 要求。
:::

## 报告

`catalog.readiness(refs=[...])` 返回一个 frozen 的 `ReadinessReport`：

| 字段 | 类型 | 含义 |
| --- | --- | --- |
| `status` | `"ready" \| "ready_with_warnings" \| "blocked"` | 总体结论（见下）。 |
| `blockers` | `tuple[ReadinessIssue, ...]` | 阻止所选 refs 完成认证的问题。 |
| `warnings` | `tuple[ReadinessIssue, ...]` | 不会 block 的建议性问题。 |
| `analysis_ready_refs` | `tuple[str, ...]` | 通过当前认证契约的 refs。 |
| `input_summary` | `ReadinessInputSummary` | 检查了什么：datasources、refs、tables。 |
| `checked_at` | `str` | 检查的 ISO-8601 时间戳。 |

`status` 由严重程度推导：只要有 blocker ⇒ `blocked`；只有 warnings ⇒ `ready_with_warnings`；
其余 ⇒ `ready`。

每个 `ReadinessIssue` 都采用相同的教学式结构：

| 属性 | 含义 |
| --- | --- |
| `kind` | 问题种类（见下表）。 |
| `severity` | `"blocker"` 或 `"warning"`。 |
| `refs` | 问题涉及的 semantic refs。 |
| `message` | 期望是什么、实际是什么。 |
| `suggested_action` | 具体的下一步。 |

`report.show()` 打印一张有界卡片；`report.to_dict()` 返回完整的机器可读列表，便于程序化处理。

## 问题种类

| Kind | 严重程度 | 触发条件 | 下一步 |
| --- | --- | --- | --- |
| `load_error` | blocker | 某个 declaration 加载失败。 | 修复 semantic load 错误并重新加载。 |
| `unknown_ref` | blocker | 请求的 ref 不在 registry 中。 | 修正该 ref，或将其从 readiness 范围中移除。 |
| `missing_business_definition` | blocker | 被检查的对象没有 `ai_context.business_definition`。 | 补充 `business_definition`。 |
| `cross_datasource_unfederated` | blocker | 某个 metric 在没有 federation 的情况下跨越多个 datasource。 | 把集成移到上游、启用 federated backend，或拆分 metric。 |
| `missing_guardrails` | blocker（metric）/ warning（其他 analyzable） | 被检查的 metric 没有 `ai_context.guardrails` 时为 blocker；entity / dimension / measure / time_dimension 缺失时为 warning。 | 补充 `guardrails`。 |
| `sql_parity_unverified` | warning | 某个 metric 有 `provenance` SQL 但 parity 未确认。 | 运行 `ms.parity_check(ref)`。 |
| `fragile_string_ref` | warning | 某个对象使用了脆弱的 string reference。 | 用稳定的 object ref 替换它。 |
| `time_dimension_pushdown_advisory` | warning | 某个 time dimension 可能无法下推到 partition。 | 当业务时间轴与 partition 一致时，保留原始 partition 列并声明其 parse。 |

## 限定检查范围

通过 `refs=` 传入 typed catalog objects，把 readiness 限定到正在 author、修改或明确认证的
objects；Marivo 会解析完整的依赖闭包（metric、它的 measures、dimensions、entities 与
datasources）：

```python
revenue = catalog.get("metric.sales.revenue")
report = catalog.readiness(refs=[revenue])
for issue in report.blockers:
    print(issue.kind, issue.refs, "→", issue.suggested_action)
```

使用该 report 完成 semantic changes 的 closeout，或满足更严格 workflow 的认证要求。日常分析
不会自动调用它。关于分析如何记录结论，参见 [证据链](/zh-cn/latest/concepts/evidence/)。
