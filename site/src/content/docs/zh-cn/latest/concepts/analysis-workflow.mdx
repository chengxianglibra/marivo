## 告诉智能体问题，不必安排分析步骤

说明你想了解什么，并补充已经明确的指标、比较方式、范围或关注方向。智能体
会选择合适的类型化算子，在分析会话中推进调查，并记录证据。

例如：

> 分析上季度已完成订单收入相比去年同期下降的原因。先关注地区差异，最后说明主要结论、
> 支撑证据和限制。

你不需要选择算子、要求智能体调用 `show()`，也不需要提前设计检查点。

## 可能改变业务含义时再参与

只要下一步仍在已经确认的指标和问题范围内，智能体就可以自行推进。需要更换指标、调整分析
对象或比较方式，或者引入可能明显改变结论的业务假设时，智能体应该停下来确认。

用业务语言确认或纠正这个选择即可。如果缺少语义定义，或者定义本身有误，智能体应该把
问题交给 `marivo-semantic`，而不是在分析过程中悄悄修改。

## 查看最终交接

分析结束后，重点看三件事：结论是否回答了问题，重要判断是否有清楚的证据，已经说明的限制
是否影响结论用于当前决策。需要检查或集成运行时的读者，可以继续查看下面的技术参考。

## 操作与分析产物参考

每次 Marivo analysis 都从一个 **metric** 开始，并按**写-运行-读**循环推进：agent 写出一个
intent（一次 operator 调用及其参数），运行它，读取 typed result，再决定下一步。

Runtime 由以下部分组成：

- **session** 持有 guiding question、semantic catalog，以及每一步的持久化结果。
- **operator 调用** 是一次确定性的分析步骤 —— `observe`、`compare`、`attribute`、
  `discover.<objective>` 以及其他核心 operators —— 及其参数。这些参数*就是*分析规格。
- **artifact** 是 operator 的 typed result。Artifacts 是步骤之间的边界：每个 operator 消费
  特定 artifact family，并产出另一个。

```python
import marivo.analysis as mv
import marivo.semantic as ms
```

:::note[Agent discovery]
从命令行开始，先用 `marivo --help` 查看 CLI 入口。要获取完整的 analysis 能力索引与环境指纹，运行：

```bash
python -m marivo help analysis
```

使用安装了 `marivo` 的 Python 解释器。输出的版本号、Python 可执行路径和包路径构成环境指纹——从 `PATH` 解析到的裸 `marivo` 只有在指纹与分析解释器一致时才可信。然后深入具体能力，例如
`mv.help("observe")`、`mv.help("compare")` 或 `mv.help("MetricFrame")`，再编写调用。每个 focused help 主题都自包含一次正确的最小调用。
:::

analysis 需要新建或刚修改的 object 时，先完成该变更的 authoring closeout：
`md.inspect(...)` → explicit scope → 一次 `inspection.sample(...)` → snapshot
projections → 一个 Python object → `catalog.verify_object(obj)` →
`catalog.preview(obj, using=snapshot)` → 零查询 `catalog.readiness(refs=[obj])`。
日常分析复用未变化的 object 时，不重复这条 authoring evidence 流程。Uncommon formats 和
semantic judgments 仍由 agent 负责。

:::note[Artifacts 是不可变的]
Artifact 一旦产出便不再改变。用 `repr(artifact)` 获取廉价单行提示，再用
`artifact.show()` 做有界读取，用 `artifact.contract()` 查看接口上可执行的下一步。
只有需要可变本地副本时才调用 `artifact.to_pandas()`，且仅作为终端步骤。每个 artifact
都携带 lineage，并持久化在 project 本地 `.marivo/` 下，因此后续脚本可以用 ref 恢复它，
而不必重新查询 datasource。
:::

## 打开 session

```python
session = mv.session.get_or_create(name="revenue-investigation", question="Why did Q4 drop?")
catalog = session.catalog
revenue = catalog.get("metric.sales.revenue")           # a metric object
region = catalog.get("dimension.sales.orders.region")      # a dimension object

revenue.details().show()     # 读取定义、候选轴和 measure lineage
region.details().show()
```

`get_or_create` 是幂等的：它会附着到同名的已有 session，或创建一个新 session，并设为当前
session。
session 的精简 API 为 `get_or_create`、`current()`、`list()` 和 `delete(name)`。

用 `catalog.domains.show()` 和 scoped navigation 发现 refs，然后对本次分析会使用的每个
metric、dimension 和 time dimension 调用 `.details().show()`。`details` 输出会展示已编写
的 `ai_context`；metric details 还会展示 composition、effective entities、候选轴和 measure
lineage。候选轴只用于静态发现，relationship 与 fanout 的有效性仍以 `observe` 为准。
`.contract()` 只表示 verify/preview/readiness 等机械下一步；`.render()` 返回的 bounded card
与 `.show()` 打印的内容相同。如果所需 object 缺失或定义存在争议，首次 `observe` 前先回到
semantic authoring 处理。

:::note[什么时候运行 readiness]
新建或修改 semantic objects 后，或者某个 workflow 要求 technical certification 时，
运行 `catalog.readiness(refs=[...])`。Analysis APIs 不会自动调用它；日常分析也不需要在每次
`observe` 前刷新 authoring snapshots。Blocked report 表示所选 objects 尚未完成这次认证；
先修复它们，再声明 semantic change 已完成。
:::

## 一个 intent 如何被指定

Operators 接受 catalog 对象和若干共享的值对象。多数 intent 都会复用这些输入：

| 值 | 形态 | 含义 |
| --- | --- | --- |
| metric 输入 | `catalog.get("metric.sales.revenue")` | semantic-id 字符串、catalog metric 对象或 `MetricRef`。来自 `ms.aggregate(...)` 的 authoring ref 也可用；ref-only 约束适用于 semantic authoring 参数，不适用于 analysis 消费输入。 |
| dimension 输入 | `catalog.get("dimension.sales.orders.region")` | semantic-id 字符串、catalog dimension 对象，或其 `DimensionRef` / `TimeDimensionRef`（见 [Semantic refs](/zh-cn/latest/concepts/semantic-layer/#semantic-refs)）。用于 `dimensions`、`slice_by` 的 key 和 `axis`。 |
| `time_scope` | `{"start": "2026-10-01", "end": "2027-01-01"}` | 半开时间区间 —— `start` 含、`end` 不含。 |
| `grain` | `"day" \| "week" \| "month" \| "quarter" \| "year" \| "hour" \| …` | 时间桶大小。存在 ⇒ time series 或 panel。 |
| `dimensions` | `[region, country]` | 分段轴。v1 中全部必须解析到 metric 所属 entity。 |
| `slice_by` | `{region: "US"}` 或 `{amount: {"op": ">", "value": 100}}` | 聚合前的行过滤（见下方运算符）。 |
| `AlignmentPolicy` | `mv.window_bucket()` | `compare` / `correlate` 时两个窗口如何配对。 |

结果的 **semantic kind** 由 `grain` 和 `dimensions` 决定：`scalar`（两者都没有）、
`time_series`（仅 grain）、`segmented`（仅 dimensions）或 `panel`（两者皆有）。

### `slice_by` 谓词运算符

key 是 catalog dimension；value 为标量（`==`）、列表（`in`），或结构化的
`{"op": ..., "value": ...}` 形式：

| 形式 | 含义 |
| --- | --- |
| `"US"` | `==`（相等） |
| `["US", "CA"]` | `in`（成员） |
| `{"op": "!=", "value": "US"}` | 不等 |
| `{"op": ">", "value": 100}`（`>=`、`<`、`<=` 同理） | 数值比较 |
| `{"op": "between", "value": ["2026-07-01", "2026-09-30"]}` | 闭区间（恰好两个值） |

## 核心 operators

### `observe` → `MetricFrame`

任何分析都从这里开始：在某个时间范围和/或分段上物化一个 metric。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `metric` | metric 对象 / ref，或非空序列 | 是 | — | 要物化的 metric，或同一 scope 下多个 metric 的序列（multi-metric observation）。不支持裸字符串。 |
| `time_scope` | dict | 否 | `None` | 半开 `{"start", "end"}` 窗口。 |
| `grain` | grain | 否 | `None` | 时间桶；存在 ⇒ time series 或 panel。 |
| `dimensions` | `list[ref]` | 否 | `None` | 分段轴。 |
| `slice_by` | dict | 否 | `None` | 聚合前的行过滤。 |
| `time_dimension` | ref | 否 | entity 默认 | 当 entity 声明多个时间轴时选择其一。 |
| `expect_shape` | shape | 否 | `None` | 守卫；若预测形状不符，在访问 backend 前抛错。 |

```python
current = session.observe(
    revenue,
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
    dimensions=[region],
)
```

传入序列即可在一个 frame 中观测多个 same-scope metric。Marivo 会把同一
datasource 的 metric 合并为一次查询，跨 datasource 的 metric 按时间轴
outer-join。用 `frame.metric(id)` 投影出 arity-1 frame，对单个 metric 做
drill-down。

`observe` 返回的 frame 调用 `to_pandas()` 时，值列命名不受 arity 影响：每个
metric 各占一列，默认使用 metric 短名；名称冲突时使用限定后的 metric id。
因此 single 与 multi-metric 观测结果可以直接 merge，无需按 arity 单独 rename。
用 `frame.value_columns` 可读取实际导出的值列名。

```python
report = session.observe(
    [revenue, catalog.get("metric.sales.total_orders")],
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
)
revenue_only = report.metric("sales.revenue")
```

对于 single、derived、cumulative 和 multi-metric frame，观测时得到的 additivity、
aggregation 与 status-time 语义都会参与 artifact identity。升级后重新执行
`observe` 时，不会复用缺少当前语义门的旧缓存 frame。

### `compare` → `DeltaFrame`

量化两个 `observe` 结果之间的变化（current 减 baseline）。两个 frame 必须共享 metric 与
semantic kind。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `current` | `MetricFrame` | 是 | — | 当前期 frame。 |
| `baseline` | `MetricFrame` | 是 | — | 基线期 frame。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 桶/分段如何配对。 |

```python
baseline = session.observe(
    revenue,
    time_scope={"start": "2025-10-01", "end": "2026-01-01"},
    grain="month",
    dimensions=[region],
)
delta = session.compare(current, baseline)
```

`compare` 默认用 `window_bucket` 配对桶。传入 `alignment=` 可覆盖 ——
`mv.dow_aligned()`、`mv.holiday_aligned()` 或 `mv.holiday_and_dow_aligned()`；calendar
支持的几种还接受 `calendar=mv.CalendarRef(...)`。

只有两侧 source frame 的 additivity、aggregation 与 status-time 三项语义完全一致，
且 additivity 已知时，delta 才会携带这些语义。任一项缺失或不一致时，语义门保持
unknown，使后续归因 fail closed，而不是继承单侧语义。

### `attribute` → `AttributionFrame`

把 delta 的变动归因到显式 axes 上。这是“为什么变了？”分析的默认 public 入口。
component-aware 的 ratio 和 weighted-average delta 会使用 mix attribution。普通
非线性 sampled fold（例如 quantile、min、max、first 或 last）不能直接按 axis 求和；
除非它们属于已持久化 component-aware derived metric delta，否则仍不支持。

`DeltaFrame` 会持久化归因门所需的语义；执行归因时，Marivo 不会回查一个可能已经
变化的 catalog。若请求的 axis 尚未物化，Marivo 也会先校验原始 delta，再 replay
observe 来补齐 axis。`DeltaFrame.show()` 会直接显示归因是 supported、conditional
还是 blocked，`DeltaFrame.contract()` 则把同一边界暴露为类型化 attribution
precondition：unknown 与普通 non-additive delta 标记为失败；semi-additive delta
要求 axes 排除 status time dimension；已持久化的 ratio/weighted-average component
路径仍保持可用。

| 已持久化的 metric 语义 | Axis 归因 |
| --- | --- |
| `additive` | 支持 sum 与 hierarchy 归因。 |
| `semi_additive` | 支持非时间轴；在其 `status_time_dimension` 上拒绝。 |
| Component-aware `ratio` / `weighted_average` | 支持 ratio/weighted mix 归因。 |
| 基于 measure 的 Tier-1 `mean` | observe 时自动展开为 `sum(measure)` / `count_non_null(measure)`，并支持 weighted mix 归因。 |
| 其他 `non_additive` metric | 拒绝，包括 opaque/tier-2 mean、median、percentile、min、max、count-distinct、tier-2 non-additive metric，以及 non-additive linear composition。 |
| 缺少 additivity metadata | 拒绝；重新执行 `observe` 和 `compare` 后再归因。 |

mean 展开使用 measure 的非空计数，而不是 entity 行数。对于被拒绝的 metric，请建模
显式 ratio/weighted-average components，或分别归因可加的 numerator 与 denominator metric。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `frame` | `DeltaFrame` | 是 | — | 要归因的 delta。 |
| `axes` | `list[dimension]` | 是 | — | 进行归因的分段轴或时间轴。 |
| `mode` | `"joint" \| "hierarchy"` | 仅多轴必填 | — | `joint` 为每个完整轴组合输出一行；`hierarchy` 输出有序前缀层级行。 |

```python
attribution = session.attribute(delta, axes=[region, platform], mode="joint")
attribution.show()
```

`joint` 的 contribution 行可直接求和并与总变化对齐。`hierarchy` 的父级行会重复
子级总量，核对总变化时只对最深层求和。component-aware 的 ratio 和 weighted-average
在两种模式下都会保留 `value_effect`、`mix_effect` 与 `residual`。

### `correlate` → `AssociationResult`

在对齐后的桶上度量两个 metric 之间的关联。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `a`、`b` | `MetricFrame` | 是 | — | 要关联的两个 frame。 |
| `measure_a`、`measure_b` | `str` | 否 | frame measure | 各 frame 上的数值列。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 桶配对。 |
| `method` | `"pearson"` | 否 | `"pearson"` | 关联方法（v1：Pearson，零滞后）。 |

### `forecast` → `ForecastFrame`

将一个 time series 或 panel 向前投影。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `history` | `MetricFrame`（time_series/panel） | 是 | — | 连续历史，无 NaN。 |
| `horizon` | `int` | 是 | — | 要投影的桶数（≥ 1）。 |
| `model` | `"naive" \| "seasonal_naive" \| "drift"` | 否 | `"seasonal_naive"` | 预测策略。 |
| `seasonality_period` | `int` | 否 | 按 grain | 覆盖季节周期（day=7、week=52、month=12、quarter=4）。 |
| `interval_level` | `float` | 否 | `0.95` | 预测区间的置信水平。 |
| `measure_column` | `str` | 否 | frame measure | 要预测的列。 |

```python
history = session.observe(revenue, time_scope={"start": "2026-01-01", "end": "2026-04-01"}, grain="day")
projection = session.forecast(history, horizon=30)
```

### `assess_quality` → `QualityReport`

对一个 artifact 运行质量检查，返回一个独立的 `QualityReport`。
`artifact.quality_summary` 只是 cheap、persisted 的 metadata projection；它不等同于执行
`session.assess_quality(artifact)`。
检查后，源 artifact 的 `meta.blocking_issues` 会保存最新的质量 blocker。存在
`time_coverage` blocker 时，不得把该窗口作为完整周期事实报告。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `frame` | `MetricFrame` | 是 | — | 要检查的 frame。 |

### `hypothesis_test` → `HypothesisTestResult`

对一个 metric 在两期之间均值是否变化做配对检验。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `a`、`b` | `MetricFrame` | 是 | — | 当前期与基线期 frame。 |
| `hypothesis` | `"mean_changed"` | 否 | `"mean_changed"` | 检验类型（v1）。 |
| `value_a`、`value_b` | `str` | 否 | frame measure | 各 frame 上的数值列。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 检验的配对方式。 |
| `sampling` | `SamplingPolicy` | 否 | 推断 | 配对/最小样本规则。 |
| `alpha` | `float` | 否 | `0.05` | 显著性水平。 |

## Discovery —— `session.discover.*` → `CandidateSet`

Discovery operators 在一个 artifact 中搜索确定性候选行，并返回 `CandidateSet`。候选行顺序
是确定性 score order，不是 Marivo 的推荐。

| Helper | 源形状 | 必填 | 关键选项 |
| --- | --- | --- | --- |
| `point_anomalies` | `MetricFrame` time_series/panel | — | `value`、`threshold=3.0` |
| `period_shifts` | `DeltaFrame` time_series/panel | ≥ 7 个桶 | `value`、`threshold=2.0` |
| `driver_axes` | `DeltaFrame` | `search_space` | `value`、`limit` |
| `interesting_slices` | `MetricFrame` 或 `DeltaFrame` | — | `search_space`、`value`、`threshold=2.0`、`limit` |
| `interesting_windows` | time_series/panel frame | — | `value`、`threshold=2.0` |
| `cross_sectional_outliers` | `MetricFrame` segmented/panel | — | `peer_scope`、`value`、`threshold=3.0` |

```python
series = session.observe(revenue, time_scope={"start": "2026-01-01", "end": "2026-04-01"}, grain="day")
candidates = session.discover.point_anomalies(series, threshold=2.0)
candidates.show()
```

## Transform 参考 —— `frame.transform.*`

Transforms 在保持 frame 家族（`MetricFrame` → `MetricFrame`，`DeltaFrame` →
`DeltaFrame`）的前提下重塑 frame。每个 transform 是 frame 自身的方法——调用
`frame.transform.<op>(...)`，而非 session 级别的 helper。

| Transform | 关键参数 | 效果 |
| --- | --- | --- |
| `filter` | `predicate`（callable） | 保留谓词为真的行。 |
| `slice` | `slice_by`（轴 → 值/列表/区间） | 保留匹配精确轴值的行。 |
| `rollup` | `drop_axes` 和/或 `grain` | 删除轴并重新聚合，或将时间轴重新分桶到更粗的 `grain`。累计 frame 取每个周期的最后一个 bucket（`rollup_fold="last"`）。 |
| `topk` | `by`、`limit` | 按某个 measure 取前 N 行。 |
| `bottomk` | `by`、`limit` | 取后 N 行。 |
| `rank` | `by`、`method`、`rank_column` | 按某个 measure 排序并加一列 rank。 |
| `normalize` | `mode`、`baseline` | `index` / `share` / `pct_change` / `per_unit` / `z_score`（仅 MetricFrame）。 |
| `window` | `window` | 限制到一个时间窗口。 |

对于 delta frame，`bottomk(by="delta")` 返回最大下降项，因为它选择最负的 delta 值。

## 累计 frame 注意事项

累计 MetricFrame 存储累计值，其语义取决于累计锚点（`all_history`、`grain_to_date` 或 `trailing`）。

- `show()`、`contract()` 和 `transform.window(...)` 在累计 frame 上正常工作。`contract()` 和
  `show()` 会呈现锚点特定的注意事项，因此在你正在阅读的 frame 上即可看到允许的路径。
- `correlate`、`discover`、`assess_quality` 和 `hypothesis_test` 允许使用。Trailing frame 是独立的
  窗口聚合（不是累计值），因此这些意图会产生有意义的结果。单调趋势注意事项仅适用于
  `all_history` 和 `grain_to_date` 锚点——解读结果时需留意。
- `compare` 按锚点分派：
  - `all_history`：拒绝。请 observe base 流量 metric 再 compare——一个窗口内的累计差值等于
    该窗口内的 base 总量。
  - `trailing`：当两个 frame 共享相同的 trailing 锚点 payload（`count`、`unit`）时允许。
    滚动窗口值按序对齐。
  - `grain_to_date`：对单周期、边界锚定的窗口允许。窗口必须从重置边界开始，至多跨越一个
    重置周期，且两个 frame 必须共享重置粒度和查询粒度。结果 DeltaFrame 会在
    `alignment_dump["to_date"]` 下记录 to-date 对齐信息。
- 对 derived metric，仅当所有外层 component 都是 cumulative，且完全共享一个 `trailing` 或
  `grain_to_date` 锚点时，才复用上述 compare 路径。`all_history`、混合锚点、累计与非累计混合、
  非法 metadata，以及 current/baseline 锚点不一致都会 fail closed。
- `attribute`、`decompose` 和 `forecast` 无论锚点如何都拒绝累计 frame；即使 compare 已接受
  derived cumulative DeltaFrame 也不例外。请为这些意图重新 observe base 流量 metric。
- `transform.rollup(...)` 通过 `rollup_fold="last"` 重新聚合累计 frame：时间轴被重新分桶到目标
  `grain`，每个周期贡献其最后一个 bucket（周期末值）。既不可重新聚合、又未携带 `rollup_fold`
  的 frame 会被拒绝——请在目标粒度重新 observe。

```python
cum_frame = session.observe(
    cumulative_active_users,
    time_scope={"start": "2026-01-01", "end": "2026-04-01"},
    grain="day",
)
cum_frame.contract()  # 显示锚点特定的注意事项
windowed = cum_frame.transform.window(window={"start": "2026-02-01", "end": "2026-03-01"})
# 将累计 frame rollup 到更粗的粒度（周期末语义）：
monthly = cum_frame.transform.rollup(grain="month")
# 对于 attribute/forecast，请改为 observe base metric：
base_frame = session.observe(active_users, ...)
```

## 终端自定义分析

当计算无法通过 `session.observe(...)` 表达时，使用 `md.raw_sql(...)`——唯一的终端
raw SQL 执行路径。它强制 timeout、限制返回行数、只读执行。结果是一个
`RawSqlResult`，无法回到类型化分析；调用 `RawSqlResult.to_pandas()` 获取终端
pandas 导出。

```python
import marivo.datasource as md

result = md.raw_sql(
    md.ref("datasource.warehouse"),
    "SELECT region, SUM(amount) AS revenue FROM orders GROUP BY region",
    reason="自定义收入拆分",
)
result.show()
df = result.to_pandas()
```

需要从类型化 frame 做终端本地 pandas 工作时，使用 `artifact.to_pandas()`。
`md.raw_sql(...)` 结果和 `to_pandas()` 导出都无法回到类型化分析。如果缺少业务
语义，请回到语义层进行 authoring。

## 读取与恢复 artifacts

Agent 读取 artifact 时按层下钻，以降低跨 loop 成本：

```python
repr(delta)          # 廉价单行提示
delta.show()         # 有界当前状态读取
delta.contract()     # 接口上可执行的下一步
delta.to_pandas()    # 终端导出，用于自定义本地分析
```

当 operator 产生 evidence 时，`artifact.show()` 会在结果 preview 之前直接展示有界的
commit-time evidence 摘要。理解当前步骤只需一次 `show()`；回顾或综合多个步骤时使用
`session.knowledge()`，需要完整 typed records 与审计链时使用 `session.evidence`。

用 `artifact.contract().affordances` 查看接口兼容性。每个 affordance 携带
`capability_id`（稳定的注册表 id）、`public_entrypoint`、`help_target`、
`required_inputs`、`preconditions` 和 `expected_output_family`。它只说明哪些
public operator 可以通过接口消费该 artifact、缺哪些输入；它不排序、不推荐，也不决定下一步。`artifact.contract().boundary_ports` 列出类型化的终端出口端口
（例如 `boundary.to_pandas`），并附带 `preserves` 和 `does_not_preserve` 保证。

### 结构化错误

当 operator 失败时，它会抛出一个类型化的 `AnalysisError` 子类。每个错误都携带稳定的
类型化字段——`expected`、`received`、`location` 和 `repair`——而不是一个通用的
details 包。`repair` 字段是一个类型化的 `AnalysisRepair` 对象，包含 `kind`
（`retry`、`inspect`、`semantic_authoring`、`environment`）、`action`、`help_target`、
可选的 `snippet` 和从实时状态生成的可选 `candidates`。按照修复指引操作即可；它会指向
失败能力的确切 `mv.help(...)` 目标。

### 终端边界出口

Marivo 维护一条类型化分析链。终端出口是 `frame.to_pandas()` 和 `md.raw_sql(...)`。
`frame.to_pandas()` 返回防御性副本用于自定义本地分析；`md.raw_sql(...)` 返回
`RawSqlResult`，其 `RawSqlResult.to_pandas()` 是自身的终端 pandas 出口。两者都不
保留 lineage 或 evidence 连续性。终端结果无法回到类型化分析。如果缺少业务语义，
请回到语义层进行 authoring。

### 语义交接

分析消费稳定的 semantic refs，绝不猜测业务含义。如果缺少或有争议的 metric、dimension
或 time dimension，分析会停止并交给 `marivo-semantic`，而不是在分析过程中悄悄修改定义。
语义层拥有业务对象契约；分析层拥有基于这些对象的类型化计算。

后续脚本需要前序结果时，读取持久化事实，而不是重跑上游查询：

```python
summaries = session.frame_summaries()
jobs = session.recent_jobs(limit=5)
previous = session.get_frame(ref)
```

使用 `mv.help("recovery")` 查看这些真实的 Session 成员。`recovery` 与 `artifacts`
是 help topic，并不是 `session.recovery` / `session.artifacts` namespace。

## 证据链与 knowledge

每个 operator 都会把 evidence 写入 session，使结论保持可审计。

- `session.knowledge()` —— 整个 session 的观察摘要、既定事实、driver 事实、未决异常和建议的后续。
- `session.evidence.findings(...)`、`.propositions(...)`、`.assessments(...)`、
  `.proposition(id)`、`.latest_assessment(id)`、`.trace(id)` —— 查找 evidence 对象，
  并把一个 proposition 追溯回支持它的 findings。

完整模型见 [证据链](/zh-cn/latest/concepts/evidence/)。

## 端到端

```python
import marivo.analysis as mv

session = mv.session.get_or_create(name="revenue-check", question="Why did Q4 drop?")
catalog = session.catalog
revenue = catalog.get("metric.sales.revenue")
region = catalog.get("dimension.sales.orders.region")

current = session.observe(
    revenue,
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
    dimensions=[region],
)
baseline = session.observe(
    revenue,
    time_scope={"start": "2025-10-01", "end": "2026-01-01"},
    grain="month",
    dimensions=[region],
)
delta = session.compare(current, baseline)
attribution = session.attribute(delta, axes=[region])
attribution.show()
```

拿到 `delta` 后，可以继续分支：用 `session.discover.period_shifts(delta)` 找出变化发生在何时，
或用 `session.forecast(current, horizon=3)` 向前投影。
