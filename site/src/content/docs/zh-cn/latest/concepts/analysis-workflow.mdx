## 告诉智能体问题，不必安排分析步骤

说明你想了解什么，并补充已经明确的指标、比较方式、范围或关注方向。智能体
会选择合适的类型化算子，在分析会话中推进调查，并记录证据。

例如：

> 分析上季度已完成订单收入相比去年同期下降的原因。先关注地区差异，最后说明主要结论、
> 支撑证据和限制。

你不需要选择算子、要求智能体调用 `show()`，也不需要提前设计检查点。

## 可能改变业务含义时再参与

只要下一步仍在已经确认的指标和问题范围内，智能体就可以自行推进。需要更换指标、调整分析
对象或比较方式，或者引入可能明显改变结论的业务假设时，智能体应该停下来确认。

用业务语言确认或纠正这个选择即可。如果缺少语义定义，或者定义本身有误，智能体应该把
受影响的 typed branch 停下来。智能体可以使用显式披露临时推断口径的终端
`md.raw_sql(...)` 继续，但不得在分析阶段修改语义层，并且必须在 closeout 请求批准最小的
正式修订。

## 查看最终交接

分析结束后，重点看三件事：结论是否回答了问题，重要判断是否有清楚的证据，已经说明的限制
是否影响结论用于当前决策。需要检查或集成运行时的读者，可以继续查看下面的技术参考。

## 操作与分析产物参考

每次 Marivo analysis 都从受治理的 semantic inputs 开始，并按**写-运行-读**循环推进：
agent 写出一个 intent（一次 operator 调用及其参数），运行它，读取 typed result，再决定
下一步。Metric analysis 从 `session.observe(...)` 开始；Event journey analysis 从
`session.events.match(...)` 开始。

Runtime 由以下部分组成：

- **session** 持有 guiding question、semantic catalog，以及每一步的持久化结果。
- **operator 调用** 是一次确定性的分析步骤 —— `observe`、`events.match`、`compare`、
  `attribute`、`discover.<objective>` 以及其他核心 operators —— 及其参数。这些参数
  *就是*分析规格。
- **artifact** 是 operator 的 typed result。Artifacts 是步骤之间的边界：每个 operator 消费
  特定 artifact family，并产出另一个。

```python
import marivo.analysis as mv
import marivo.semantic as ms
```

:::note[Agent discovery]
从命令行开始，先用 `marivo --help` 查看 CLI 入口。要获取完整的 analysis 能力索引与环境指纹，运行：

```bash
python -m marivo help analysis
```

使用安装了 `marivo` 的 Python 解释器。输出的版本号、Python 可执行路径和包路径构成环境指纹——从 `PATH` 解析到的裸 `marivo` 只有在指纹与分析解释器一致时才可信。然后深入具体能力，例如
`mv.help("observe")`、`mv.help("compare")` 或 `mv.help("MetricFrame")`，再编写调用。每个 focused help 主题都自包含一次正确的最小调用。
:::

analysis 需要新建或刚修改的 object 时，先完成该变更的 authoring closeout：
`md.inspect(...)` → explicit scope → 一次 `inspection.sample(...)` → snapshot
projections → 一个 Python declaration → exact ref → `catalog.verify(ref)` →
`catalog.preview(ref, using=snapshot)` → 零查询 `catalog.readiness(refs=[ref])`。
日常分析复用未变化的 object 时，不重复这条 authoring evidence 流程。Uncommon formats 和
semantic judgments 仍由 agent 负责。

:::note[Artifacts 是不可变的]
Artifact 一旦产出便不再改变。用 `repr(artifact)` 获取廉价单行提示，再用
`artifact.show()` 做有界读取，用 `artifact.contract()` 查看接口上可执行的下一步。
只有需要可变本地副本时才调用 `artifact.to_pandas()`，且仅作为终端步骤。每个 artifact
都携带 lineage，并持久化在 project 本地 `.marivo/` 下，因此后续脚本可以用 ref 恢复它，
而不必重新查询 datasource。`artifact.id` 是 canonical `artifact.ref` 的只读别名。
:::

## 打开 session

```python
session = mv.session.get_or_create(name="revenue-investigation", question="Why did Q4 drop?")
catalog = session.catalog
revenue_object = catalog.require(ms.ref.metric("sales.revenue"))
region_object = catalog.require(ms.ref.dimension("sales.orders.region"))

revenue_object.details().show()     # 读取定义、候选轴和 measure lineage
region_object.details().show()
revenue = revenue_object.ref
region = region_object.ref
```

`get_or_create` 是幂等的：它会附着到同名的已有 session，或创建一个新 session，并设为当前
session。
session 的精简 API 为 `get_or_create`、`current()`、`list()` 和 `delete(name)`。

`repr(session)` 会引导调用 `session.show()`；该方法打印包含问题、读写状态、报告时区、
时间戳以及 catalog/job/frame 检查入口的 bounded state card。`session.render()` 返回同一文本，
但不写 stdout。

用 `catalog.domains.show()` 和 scoped navigation 发现 refs，然后对本次分析会使用的每个
metric、dimension 和 time dimension 调用 `.details().show()`。`details` 输出会展示已编写
的 `ai_context`；metric details 还会展示 composition、effective entities、候选轴和 measure
lineage。候选轴只用于静态发现，relationship 与 fanout 的有效性仍以 `observe` 为准。
`.contract()` 只表示 verify/preview/readiness 等机械下一步；`.render()` 返回的 bounded card
与 `.show()` 打印的内容相同。如果所需 object 缺失或定义存在争议，首次 `observe` 前先回到
semantic authoring 处理。

:::note[什么时候运行 readiness]
新建或修改 semantic objects 后，或者某个 workflow 要求 technical certification 时，
运行 `catalog.readiness(refs=[...])`。同一调用接受由 catalog refs 与封闭 `RuntimeMetricExpr`
roots 组成的有序 mixed list，并通过 `analysis_ready_inputs` 返回通过的 roots。Analysis APIs 不会自动调用它；日常分析也不需要在每次
`observe` 前刷新 authoring snapshots。Blocked report 表示所选 objects 尚未完成这次认证；
先修复它们，再声明 semantic change 已完成。
:::

## 匹配 Event journey

Event analysis 从 typed participant roles 和有序 EventPattern 开始。每个 step
使用稳定的 snake-case key；role 已携带 Event 和 subject Entity，而 subject Entity
的 primary key 提供 identity。

```python
cart_user = ms.participant_role(event=cart_created, name="user")
checkout_user = ms.participant_role(event=checkout_started, name="user")
payment_user = ms.participant_role(event=payment_succeeded, name="user")

cart_step = mv.step(participant=cart_user, key="cart")
checkout_step = mv.step(participant=checkout_user, key="checkout")
payment_step = mv.step(participant=payment_user, key="payment")

pattern = mv.sequence(cart_step, checkout_step, payment_step)
journeys = session.events.match(
    pattern=pattern,
    cohort_window=mv.TimeScope(start=start, end=end),
    completion_through=followup_end,
    matching=mv.first_per_subject(),
)
```

cohort window 是半开区间：只有 `[start, end)` 内的首 step occurrence 能建立
journey。`completion_through` 是包含端点的 follow-up bound，并且不得早于 `end`。
阶段一不接收 cohort/SubjectSet 参数，也不宣传 funnel、time-to-event、selection、
comparison 或 attribution continuation。

`EventFrame[journey]` 是 dense long table：每个 `journey × pattern step` 都有一行。
后续 step 缺失时，该行仍然存在，其 event、time 和 elapsed 值为空。只有全部必需
Event 都已知完整覆盖到 follow-up bound 时，缺失 step 才是 `incomplete`；否则为
`coverage_censored`。

backend 可以通过 `marivo_event_watermark(request)` 提供权威 Event completeness。
Marivo 不会从最大 observed Event time、query time 或 SLA 推导 watermark。没有权威
coverage 时，调用方可以携带精确且有界的显式假设：

```python
coverage = mv.declared_complete_through(
    inputs=(cart_created, checkout_started, payment_succeeded),
    through=followup_end,
    rationale="Warehouse reconciliation completed through followup_end.",
)
journeys = session.events.match(
    pattern=pattern,
    cohort_window=mv.TimeScope(start=start, end=end),
    completion_through=followup_end,
    matching=mv.first_per_subject(),
    completeness=(coverage,),
)
```

observed watermark 优先于 declaration。artifact 及其 quality/evidence records
会保留每个输入的 `observed_watermark | declared_complete | unknown` basis，以及
整体 `observed_watermark | declared_complete | mixed | unknown` basis。

## 一个 intent 如何被指定

Operators 接受精确 typed ref/expression 和若干共享值对象。多数 intent 都会复用这些输入：

| 值 | 形态 | 含义 |
| --- | --- | --- |
| metric 输入 | `catalog.require(ms.ref.metric("sales.revenue")).ref` | 精确 `Ref[metric]`，或 observe 使用的封闭 `RuntimeMetricExpr`；拒绝 loaded object、generic ref 与字符串。 |
| dimension 输入 | `catalog.require(ms.ref.dimension("sales.orders.region")).ref` | 精确 `Ref[dimension]` / `Ref[time_dimension]`（见 [Semantic refs](/zh-cn/latest/concepts/semantic-layer/#semantic-refs)）。用于 `dimensions`、`slice_by` key 与 `axis`。 |
| `time_scope` | `{"start": "2026-10-01", "end": "2027-01-01"}` | 半开时间区间 —— `start` 含、`end` 不含。 |
| `grain` | `"day" \| "week" \| "month" \| "quarter" \| "year" \| "hour" \| …` | 时间桶大小。存在 ⇒ time series 或 panel。 |
| `dimensions` | `[region, country]` | 分段轴。v1 中全部必须解析到 metric 所属 entity。 |
| `slice_by` | `{region: "US"}` 或 `{amount: {"op": ">", "value": 100}}` | 聚合前的行过滤（见下方运算符）。 |
| `AlignmentPolicy` | `mv.window_bucket()` | `compare` / `correlate` 时两个窗口如何配对。 |

结果的 **semantic kind** 由 `grain` 和 `dimensions` 决定：`scalar`（两者都没有）、
`time_series`（仅 grain）、`segmented`（仅 dimensions）或 `panel`（两者皆有）。

### `slice_by` 谓词运算符

key 是 catalog dimension；value 为标量（`==`）、列表（`in`），或结构化的
`{"op": ..., "value": ...}` 形式：

| 形式 | 含义 |
| --- | --- |
| `"US"` | `==`（相等） |
| `["US", "CA"]` | `in`（成员） |
| `{"op": "!=", "value": "US"}` | 不等 |
| `{"op": ">", "value": 100}`（`>=`、`<`、`<=` 同理） | 数值比较 |
| `{"op": "between", "value": ["2026-07-01", "2026-09-30"]}` | 闭区间（恰好两个值） |

## 核心 operators

### `observe` → `MetricFrame`

任何分析都从这里开始：在某个时间范围和/或分段上物化一个 metric。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `metric` | `Ref[metric] | RuntimeMetricExpr`，或非空序列 | 是 | — | 精确 catalog ref 或封闭 runtime expression；拒绝 loaded object、generic ref 与裸字符串。 |
| `time_scope` | dict | 否 | `None` | 半开 `{"start", "end"}` 窗口。 |
| `grain` | grain | 否 | `None` | 时间桶；存在 ⇒ time series 或 panel。 |
| `dimensions` | `list[Ref[dimension] | Ref[time_dimension]]` | 否 | `None` | 精确类型化分段轴。 |
| `slice_by` | `Mapping[AnalysisRef[dimension], SliceValue]` | 否 | `None` | 全局聚合前行过滤。 |
| `time_dimension` | `Ref[time_dimension]` | 否 | entity 默认 | 当 entity 声明多个时间轴时选择其一。 |
| `expect_shape` | shape | 否 | `None` | 守卫；若预测形状不符，在访问 backend 前抛错。 |

```python
current = session.observe(
    revenue,
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
    dimensions=[region],
)
```

传入序列即可在一个 frame 中观测多个 same-scope metric。Marivo 会把同一
datasource 的 metric 合并为一次查询，跨 datasource 的 metric 按时间轴
outer-join。用 `frame.metric(id)` 投影出 arity-1 frame，对单个 metric 做
drill-down。

单指标 `MetricFrame` 的所有公开读取路径都使用 metric 短名作为值列名，包括
`show()`、`columns`、`contract()`、索引和 `to_pandas()`；经过 transform 后列名
仍保持不变。若短名与轴列冲突，Marivo 会改用限定后的 metric id；若该名称仍
被占用，则追加确定性的 `#N` 后缀。multi-metric frame 仍保持每个 metric 一列，
因此不同 arity 的 frame 可以直接 merge，无需单独 rename。用
`frame.value_columns` 可读取准确的公开值列名。

typed transform 内部仍使用 canonical `value` 列。例如
`frame.transform.filter(predicate=lambda data: data["value"] > 0)` 仍然有效；
之后可通过 `filtered[filtered.value_columns[0]]` 或 `filtered.to_pandas()` 读取结果。

```python
report = session.observe(
    [revenue, catalog.require(ms.ref.metric("sales.total_orders")).ref],
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
)
revenue_only = report.metric("sales.revenue")
```

当 catalog 没有需要的临时分析口径时，可构造封闭 expression，并且仍只通过
`session.observe(...)` 物化：

```python
failed = mv.runtime_metric.aggregate(
    failed_requests_measure,
    agg="sum",
    slice_by={state: "failed"},
    label="Runtime failed requests",
)
weighted_latency = mv.runtime_metric.weighted_mean(
    latency_measure,
    request_count_measure,
    label="Runtime weighted latency",
)
failure_rate = mv.runtime_metric.ratio(
    failed,
    requests,
    zero_division="null",
    label="Runtime failure rate",
)
frame = session.observe(failure_rate, time_scope=window, grain="day")
```

runtime expression 可以递归包含 runtime expression 与 catalog `Ref[metric]`。
`runtime_metric.slice(..., by=...)` 是 branch-local，`observe(..., slice_by=...)`
是 global。catalog/runtime root 都会 lower 到同一 canonical graph（最大深度 10，
提交的 pre-CSE occurrence 最多 256）以及一个 model/source domain。label 只影响
展示，不会变成 value identity 或 catalog authority。每个 `runtime_metric` 构造器都
必须设置非空 `label`，包括嵌套 expression；当 expression 作为 observed root 物化时，
该 label 成为稳定的公开值列 handle。下游 operator 根据持久化 frame 状态准入，不根据
catalog/runtime 来源准入。

`runtime_metric.weighted_mean(value, weight)` 会自动完成行级乘权与成对聚合。
两个 measure 必须属于同一 entity 和物理行粒度，weight 必须为 additive；无需预先
定义 `SUM(value * weight)` measure。

每个 observed root 与 mixed forest 都会持久化递归 component graph。可通过
`frame.components()` 检查 node role、evaluator contract、quality、coverage ref
与 governed leaf lineage；只有同时支持数值分解的 root 才会另外带有 `component_ref`。

对于 single、derived、cumulative 和 multi-metric frame，观测时得到的 additivity、
aggregation 与 status-time 语义都会参与 artifact identity。升级后重新执行
`observe` 时，不会复用缺少当前语义门的旧缓存 frame。

### `compare` → `DeltaFrame`

量化两个 `observe` 结果之间的变化（current 减 baseline）。两个 frame 必须共享 semantic
kind 与持久化 comparable value semantics；catalog/runtime identity 可以不同。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `current` | `MetricFrame` | 是 | — | 当前期 frame。 |
| `baseline` | `MetricFrame` | 是 | — | 基线期 frame。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 桶/分段如何配对。 |

```python
baseline = session.observe(
    revenue,
    time_scope={"start": "2025-10-01", "end": "2026-01-01"},
    grain="month",
    dimensions=[region],
)
delta = session.compare(current, baseline)
```

`compare` 默认用 `window_bucket` 配对桶。传入 `alignment=` 可覆盖 ——
`mv.dow_aligned()`、`mv.holiday_aligned()` 或 `mv.holiday_and_dow_aligned()`；calendar
支持的几种还接受 `calendar=mv.CalendarRef(...)`。

只有两侧 source frame 的 additivity、aggregation 与 status-time 三项语义完全一致，
且 additivity 已知时，delta 才会携带这些语义。任一项缺失或不一致时，语义门保持
unknown，使后续归因 fail closed，而不是继承单侧语义。

### `attribute` → `AttributionFrame`

把 delta 的变动归因到显式 axes 上。这是“为什么变了？”分析的默认 public 入口。
component-aware 的 ratio 和 weighted-mean delta 会使用 mix attribution。普通
非线性 sampled fold（例如 quantile、min、max、first 或 last）不能直接按 axis 求和；
除非它们属于已持久化 component-aware derived metric delta，否则仍不支持。

`DeltaFrame` 会持久化归因门所需的语义；执行归因时，Marivo 不会回查一个可能已经
变化的 catalog。若请求的 axis 尚未物化，Marivo 也会先校验原始 delta，再 replay
observe 来补齐 axis。`DeltaFrame.show()` 会直接显示归因是 supported、conditional
还是 blocked，`DeltaFrame.contract()` 则把同一边界暴露为类型化 attribution
precondition：unknown 与普通 non-additive delta 标记为失败；semi-additive delta
要求 axes 排除 status time dimension；已持久化的 ratio/weighted-mean component
路径仍保持可用。

| 已持久化的 metric 语义 | Axis 归因 |
| --- | --- |
| `additive` | 支持 sum 与 hierarchy 归因。 |
| `semi_additive` | 支持非时间轴；在其 `status_time_dimension` 上拒绝。 |
| Component-aware `ratio` / `weighted_mean` | 支持 ratio/weighted mix 归因。 |
| 基于 measure 的 Tier-1 `mean` | observe 时自动展开为 `sum(measure)` / `count_non_null(measure)`，并支持 weighted mix 归因。 |
| 其他 `non_additive` metric | 拒绝，包括 opaque/tier-2 mean、median、percentile、min、max、count-distinct、tier-2 non-additive metric，以及 non-additive linear composition。 |
| 缺少 additivity metadata | 拒绝；重新执行 `observe` 和 `compare` 后再归因。 |

mean 展开使用 measure 的非空计数，而不是 entity 行数。对于被拒绝的 metric，请建模
显式 ratio/weighted-mean components，或分别归因可加的 numerator 与 denominator metric。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `frame` | `DeltaFrame` | 是 | — | 要归因的 delta。 |
| `axes` | `list[dimension]` | 是 | — | 进行归因的分段轴或时间轴。 |
| `mode` | `"joint" \| "hierarchy"` | 仅多轴必填 | — | `joint` 为每个完整轴组合输出一行；`hierarchy` 输出有序前缀层级行。 |

```python
attribution = session.attribute(delta, axes=[region, platform], mode="joint")
attribution.show()
```

`joint` 的 contribution 行可直接求和并与总变化对齐。`hierarchy` 的父级行会重复
子级总量，核对总变化时只对最深层求和。component-aware 的 ratio 和 weighted-mean
在两种模式下都会保留 `value_effect`、`mix_effect` 与 `residual`。
多轴调用没有默认 mode；单轴应省略 `mode`，即使传入也不会改变输出。
单轴输出会保留解析后的维度列名（例如 `cluster`），因此可以直接按该列与来源
`DeltaFrame` 连接。通用的 `level`、`axis`、`driver` 与 `path` 列仅用于多轴
hierarchy 输出；additive 单轴归因的 `attribution_shape` 为 `"sum"`。
保留的维度名不能与归因结果、value 或 panel bucket 列冲突；发生冲突时，Marivo
会 fail closed 并返回结构化 semantic-authoring repair，而不会生成重复列或含义
歧义的列。Evidence 协议字段通过 metadata 显式映射，不会占用用户维度名。
`attribution.attribution_mode` 表示行布局，而 `attribution.attribution_shape` / `meta.method`
表示归因数学，因此两种布局都可能显示 `method="weighted_mix"`。可调用
`mv.help("AttributionMode")` 查看这一 focused contract。

Contribution share 会显式说明分母：

- `share_of_total_delta` 保留符号；当某个 driver 抵消整体净变化时，它可以超过 100%；
- `share_of_positive_pool` 只为正 contribution 填值；
- `share_of_negative_pool` 只为负 contribution 填值，并返回正的池内占比。

Marivo 不会把任一符号池直接命名为“改善”或“恶化”，因为当前持久化 metric contract
不包含指标好坏方向；只有明确业务方向后，agent 才能加上这层解释。new/churned segment
会保留精确的单边 component contribution。`AttributionFrame.show()` 会显示对账行，包括
total delta、contribution 合计、one-sided contribution 合计、未归因合计和 residual；若任一
最深层 partition 无法在数值容差内对齐，归因会直接失败，不会输出表面合计为 100% 的结果。

### `correlate` → `AssociationResult`

在对齐后的桶上度量两个 metric 之间的关联。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `a`、`b` | `MetricFrame` | 是 | — | 要关联的两个 frame。 |
| `measure_a`、`measure_b` | `str` | 否 | frame measure | 各 frame 的 `value_columns` 返回的公开值列名。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 桶配对。 |
| `method` | `"pearson"`、`"spearman"`、`"kendall"` | 否 | `"pearson"` | 相关性计算方法。 |
| `lag_range` | `range` 或有符号整数序列 | 否 | lag 0 | 要计算的 lag；`k` 表示 `a[t]` 对 `b[t+k]`。正值表示 `a` 领先 `b`，负值表示 `b` 领先 `a`。 |

非零 lag 要求输入为 `time_series` 或 `panel`。Panel lag 只在各维度序列内偏移，
不会跨序列边界配对；空值会在偏移后按配对过滤，因此缺失桶不会压缩时间轴。
每个 lag 至少需要两个重叠且非常量的配对。结果为每个 lag 生成一行，并将绝对
相关性最强的 lag 记录为 `meta.best_lag`。

省略 `measure_a` / `measure_b` 时使用各 frame 唯一的 metric 值；显式指定时，
传入 `a.value_columns` 与 `b.value_columns` 返回的准确公开名称。

### `forecast` → `ForecastFrame`

将一个 time series 或 panel 向前投影。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `history` | `MetricFrame`（time_series/panel） | 是 | — | 连续历史，无 NaN。 |
| `horizon` | `int` | 是 | — | 要投影的桶数（≥ 1）。 |
| `model` | `"naive" \| "seasonal_naive" \| "drift"` | 否 | `"seasonal_naive"` | 预测策略。 |
| `seasonality_period` | `int` | 否 | 按 grain | 覆盖季节周期（day=7、week=52、month=12、quarter=4）。 |
| `interval_level` | `float` | 否 | `0.95` | 预测区间的置信水平。 |
| `measure_column` | `str` | 否 | frame measure | `history.value_columns` 返回的公开值列名。 |

```python
history = session.observe(revenue, time_scope={"start": "2026-01-01", "end": "2026-04-01"}, grain="day")
projection = session.forecast(history, horizon=30, measure_column=history.value_columns[0])
```

### `assess_quality` → `QualityReport`

对一个 artifact 运行质量检查，返回一个独立的 `QualityReport`。
`artifact.quality_summary` 只是 cheap、persisted 的 metadata projection；它不等同于执行
`session.assess_quality(artifact)`。
Report 的 `contract().issues` 包含类型化数据质量失败。存在 blocking 的
`time_coverage_incomplete` issue 时，不得把该窗口作为完整周期事实报告。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `frame` | `MetricFrame` | 是 | — | 要检查的 frame。 |

### `hypothesis_test` → `HypothesisTestResult`

对一个 metric 在两期之间均值是否变化做配对检验。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `a`、`b` | `MetricFrame` | 是 | — | 当前期与基线期 frame。 |
| `hypothesis` | `"mean_changed"` | 否 | `"mean_changed"` | 检验类型（v1）。 |
| `value_a`、`value_b` | `str` | 否 | frame measure | 各 frame 的 `value_columns` 返回的公开值列名。 |
| `alignment` | `AlignmentPolicy` | 否 | `window_bucket` | 检验的配对方式。 |
| `sampling` | `SamplingPolicy` | 否 | 推断 | 配对/最小样本规则。 |
| `alpha` | `float` | 否 | `0.05` | 显著性水平。 |

## Discovery —— `session.discover.*` → `CandidateSet`

Discovery operators 在一个 artifact 中搜索确定性候选行，并返回 `CandidateSet`。候选行顺序
是确定性 score order，不是 Marivo 的推荐。

| Helper | 源形状 | 必填 | 关键选项 |
| --- | --- | --- | --- |
| `point_anomalies` | `MetricFrame` time_series/panel | — | `value`、`threshold=3.0` |
| `period_shifts` | `DeltaFrame` time_series/panel | ≥ 7 个桶 | `value`、`threshold=2.0` |
| `driver_axes` | `DeltaFrame` | `search_space` | `value`、`limit` |
| `interesting_slices` | `MetricFrame` 或 `DeltaFrame` | — | `search_space`、`value`、`threshold=2.0`、`limit` |
| `interesting_windows` | time_series/panel frame | — | `value`、`threshold=2.0` |
| `cross_sectional_outliers` | `MetricFrame` segmented/panel | — | `peer_scope`、`value`、`threshold=3.0` |

```python
series = session.observe(revenue, time_scope={"start": "2026-01-01", "end": "2026-04-01"}, grain="day")
candidates = session.discover.point_anomalies(series, threshold=2.0)
candidates.show()
selection = candidates.select(rank=1)
print(selection.kind, selection.window, selection.keys)
```

`select(rank=1)` 返回与 candidate shape 对应的封闭、不可变 selection variant。
它不接受任意 attribute 名，也不会创建 job、artifact、lineage step 或 evidence 记录。

## Transform 参考 —— `frame.transform.*`

Transforms 在保持 frame 家族（`MetricFrame` → `MetricFrame`，`DeltaFrame` →
`DeltaFrame`）的前提下重塑 frame。每个 transform 是 frame 自身的方法——调用
`frame.transform.<op>(...)`，而非 session 级别的 helper。

| Transform | 关键参数 | 效果 |
| --- | --- | --- |
| `filter` | `predicate`（callable） | 保留谓词为真的行。 |
| `slice` | `slice_by`（轴 → 值/列表/区间） | 保留匹配精确轴值的行。 |
| `rollup` | `drop_axes` 和/或 `grain` | 删除轴并重新聚合，或将时间轴重新分桶到更粗的 `grain`。累计 frame 取每个周期的最后一个 bucket（`rollup_fold="last"`）。 |
| `topk` | `by`、`limit` | 按某个 measure 取前 N 行。 |
| `bottomk` | `by`、`limit` | 取后 N 行。 |
| `rank` | `by`、`method`、`rank_column` | 按某个 measure 排序并加一列 rank。 |
| `normalize` | `mode`、`baseline` | `index` / `share` / `pct_change` / `per_unit` / `z_score`（仅 MetricFrame）。 |
| `window` | `window` | 限制到一个时间窗口。 |

对于 delta frame，`bottomk(by="delta")` 返回最大下降项，因为它选择最负的 delta 值。

## 累计 frame 注意事项

累计 MetricFrame 存储累计值，其语义取决于累计锚点（`all_history`、`grain_to_date` 或 `trailing`）。

- `show()`、`contract()` 和 `transform.window(...)` 在累计 frame 上正常工作。`contract()` 和
  `show()` 会呈现锚点特定的注意事项，因此在你正在阅读的 frame 上即可看到允许的路径。
- `correlate`、`discover`、`assess_quality` 和 `hypothesis_test` 允许使用。Trailing frame 是独立的
  窗口聚合（不是累计值），因此这些意图会产生有意义的结果。单调趋势注意事项仅适用于
  `all_history` 和 `grain_to_date` 锚点——解读结果时需留意。
- `compare` 按锚点分派：
  - `all_history`：拒绝。请 observe base 流量 metric 再 compare——一个窗口内的累计差值等于
    该窗口内的 base 总量。
  - `trailing`：当两个 frame 共享相同的 trailing 锚点 payload（`count`、`unit`）时允许。
    滚动窗口值按序对齐。
  - `grain_to_date`：对单周期、边界锚定的窗口允许。窗口必须从重置边界开始，至多跨越一个
    重置周期，且两个 frame 必须共享重置粒度和查询粒度。结果 DeltaFrame 会在
    `alignment_dump["to_date"]` 下记录 to-date 对齐信息。
- 对 derived metric，仅当所有外层 component 都是 cumulative，且完全共享一个 `trailing` 或
  `grain_to_date` 锚点时，才复用上述 compare 路径。`all_history`、混合锚点、累计与非累计混合、
  非法 metadata，以及 current/baseline 锚点不一致都会 fail closed。
- `attribute`、`decompose` 和 `forecast` 无论锚点如何都拒绝累计 frame；即使 compare 已接受
  derived cumulative DeltaFrame 也不例外。请为这些意图重新 observe base 流量 metric。
- `transform.rollup(...)` 通过 `rollup_fold="last"` 重新聚合累计 frame：时间轴被重新分桶到目标
  `grain`，每个周期贡献其最后一个 bucket（周期末值）。既不可重新聚合、又未携带 `rollup_fold`
  的 frame 会被拒绝——请在目标粒度重新 observe。

```python
cum_frame = session.observe(
    cumulative_active_users,
    time_scope={"start": "2026-01-01", "end": "2026-04-01"},
    grain="day",
)
cum_frame.contract()  # 显示锚点特定的注意事项
windowed = cum_frame.transform.window(window={"start": "2026-02-01", "end": "2026-03-01"})
# 将累计 frame rollup 到更粗的粒度（周期末语义）：
monthly = cum_frame.transform.rollup(grain="month")
# 对于 attribute/forecast，请改为 observe base metric：
base_frame = session.observe(active_users, ...)
```

## 终端自定义分析

当 governed measure 或 catalog metric 足以表达临时口径时，应先使用封闭的
`mv.runtime_metric` expression。只有计算仍无法通过 `session.observe(...)` 表达时，
才使用 `md.raw_sql(...)`——唯一的终端
raw SQL 执行路径。它强制 timeout、限制返回行数、只读执行。结果是一个
`RawSqlResult`，无法回到类型化分析；调用 `RawSqlResult.to_pandas()` 获取终端
pandas 导出。

任何语义缺口也可以无需事前审批使用该出口。智能体可以临时推断 metric、dimension、
relationship 或过滤口径，但必须记录全部假设，并将 raw-SQL-supported claims 与 canonical
artifact evidence 分开。

```python
import marivo.datasource as md

result = md.raw_sql(
    ms.ref.datasource("warehouse"),
    "SELECT region, SUM(amount) AS revenue FROM orders GROUP BY region",
    reason="自定义收入拆分",
)
result.show()
df = result.to_pandas()
```

需要从类型化 frame 做终端本地 pandas 工作时，使用 `artifact.to_pandas()`。
`md.raw_sql(...)` 结果和 `to_pandas()` 导出都无法回到类型化分析。Raw SQL 不会修复缺失的
业务语义；closeout 仍然必须请求批准最小语义修订。

## 读取与恢复 artifacts

Agent 读取 artifact 时按层下钻，以降低跨 loop 成本：

```python
repr(delta)          # 廉价单行提示
delta.show()         # 有界当前状态读取
delta.contract()     # 接口兼容性与类型化 issues
delta.to_pandas()    # 终端导出，用于自定义本地分析
```

当 operator 产生 evidence 时，`artifact.show()` 会在结果 preview 之前展示有界的类型化
digest。同一个值可以从 `artifact.evidence_digest` 读取；通过
`artifact.evidence_status` 检查降级。精确 finding 与 derivation trace 使用
`session.evidence`。

用 `artifact.contract().affordances` 查看接口兼容性。每个 affordance 携带
`capability_id`（稳定的注册表 id）、`public_entrypoint`、`help_target`、
保留参数角色的 `inputs`、`preconditions` 和 `expected_output_family`。每个 input
记录准确参数名、接受的 artifact family，以及当前 artifact 能否绑定它。Contract
不排序、不推荐、不枚举调用，也不决定下一步。`artifact.contract().boundary_ports` 列出类型化的终端出口端口
（例如 `boundary.to_pandas`），并附带 `preserves` 和 `does_not_preserve` 保证。

### 结构化错误

当 operator 失败时，它会抛出一个类型化的 `AnalysisError` 子类。每个错误都携带稳定的
类型化字段——`expected`、`received`、`location` 和 `repair`——而不是一个通用的
details 包。`repair` 字段是一个类型化的 `AnalysisRepair` 对象，包含 `kind`
（`retry`、`inspect`、`semantic_authoring`、`environment`）、`action`、`help_target`、
可选的 `snippet` 和从实时状态生成的可选 `candidates`。按照修复指引操作即可；它会指向
失败能力的确切 `mv.help(...)` 目标。

### 终端边界出口

Marivo 维护一条类型化分析链。终端出口是 `frame.to_pandas()` 和 `md.raw_sql(...)`。
`frame.to_pandas()` 返回防御性副本用于自定义本地分析；`md.raw_sql(...)` 返回
`RawSqlResult`，其 `RawSqlResult.to_pandas()` 是自身的终端 pandas 出口。两者都不
保留 lineage 或 evidence 连续性。终端结果无法回到类型化分析。如果缺少业务语义，
只能用终端 raw SQL 提供临时结果，并在 closeout 保留缺口，获得批准后再 authoring。

### 语义交接

Typed analysis 消费稳定的 semantic refs，绝不猜测业务含义。如果缺少或有争议的 metric、
dimension 或 time dimension，受影响的 typed branch 会停止。智能体只可以在终端 `md.raw_sql(...)`
分支临时推断口径，并必须披露 datasource、目的、全部假设、有界结果，以及 canonical identity、
lineage 和 evidence continuity 的缺失。它在 closeout 提出最小正式语义修订，并且只在用户
明确批准后进入 `marivo-semantic`。语义层拥有业务对象契约；分析层拥有基于这些对象的
类型化计算。

后续脚本需要前序结果时，读取持久化事实，而不是重跑上游查询：

```python
summaries = session.frame_summaries(limit=20)
jobs = session.recent_jobs(limit=5)
previous = session.get_frame(ref)

# 仅在恢复任务、问题明显重复或同类错误再次出现时查看历史：
history = mv.session.recent(limit=10)
prior = mv.session.inspect(history.items[0].name) if history.items else None
```

`frame_summaries()` 返回 `FrameSummaryPage`。`has_more` 为 true 时把不透明的
`next_cursor` 传回同一个方法。分页采用普通 keyset 语义，不提供快照隔离。

使用 `mv.help("recovery")` 查看这些真实的 Session 成员。`recovery` 与 `artifacts`
是 help topic，并不是 `session.recovery` / `session.artifacts` namespace。

## 证据链与审计

每个已提交 operator 都会把确定性的类型化 evidence 写入 session。Marivo 不生成跨 artifact
判断，也不选择下一步操作。

- `session.evidence.digests(...)` —— 有界、最新优先的 digest page。
- `session.evidence.findings(...)` —— 有界、最新优先的 finding page。
- `session.evidence.digest(ref)`、`.finding(id)`、`.trace(id)` —— 精确读取与 finding
  derivation 审计。

完整模型见 [证据链](/zh-cn/latest/concepts/evidence/)。

## 端到端

```python
import marivo.analysis as mv

session = mv.session.get_or_create(name="revenue-check", question="Why did Q4 drop?")
catalog = session.catalog
revenue = catalog.require(ms.ref.metric("sales.revenue")).ref
region = catalog.require(ms.ref.dimension("sales.orders.region")).ref

current = session.observe(
    revenue,
    time_scope={"start": "2026-10-01", "end": "2027-01-01"},
    grain="month",
    dimensions=[region],
)
baseline = session.observe(
    revenue,
    time_scope={"start": "2025-10-01", "end": "2026-01-01"},
    grain="month",
    dimensions=[region],
)
delta = session.compare(current, baseline)
attribution = session.attribute(delta, axes=[region])
attribution.show()
```

拿到 `delta` 后，可以继续分支：用 `session.discover.period_shifts(delta)` 找出变化发生在何时，
或用 `session.forecast(current, horizon=3)` 向前投影。
