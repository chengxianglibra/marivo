## 指标是一份业务契约

指标不只是一个字段或查询结果。它是一份纳入版本管理的协议，规定智能体可以分析的业务数值：
它表示什么、排除什么、如何计算，以及可以用在什么场景。Marivo 可以检查声明和匹配的预览证据
是否已在技术上就绪；只有对该指标负责的业务负责人才能批准这份契约是否适用于预期决策。

## 语义层从哪里来

你可以直接使用团队已经维护的语义层，也可以从新项目开始建立。

### 克隆已有项目

项目中的 `models/` 已经包含语义对象时，先复用这些定义。安装依赖、配置本地环境变量，
再让智能体检查现有对象是否适用于当前问题。没有缺口时，直接开始分析；不要重新生成已有
对象。克隆后的模型发生变化，或者 workflow 要求 technical certification 时，再运行
scoped readiness。

### 从新项目开始

向智能体说明数据源、业务目标和已经确定的规则。智能体使用 `marivo-semantic` 检查证据并
起草对象；用户根据业务需要确认和调整口径。技术验证通过不等于业务定义已经获批。

## 通过 Git 维护和共享语义层

语义对象是 `models/` 下的 Python 源文件，应与项目代码一起纳入版本管理：

- 团队成员通过克隆项目复用同一组定义；
- 修改口径时使用分支和拉取请求，展示定义、负责人、`ai_context`、`guardrails` 和受影响指标；
- 审查不只检查 Python 是否可运行，还要检查业务含义是否得到负责人确认；
- 数据源只提交 `*_env` 引用，不提交凭据值；
- 不提交 `.marivo/` 运行状态、本地缓存或生成的密钥；
- semantic change 合并后，让智能体检查配置和受影响对象的 scoped readiness。

Git 保存可协作审查的语义声明，不会替代业务审批，也不会证明当前数据已经就绪。

## 和智能体一起确认业务含义

智能体可以检查数据源证据、起草语义对象、完成技术验证，并说明仍未解决的问题。用户需要
判断的是：这个指标是否真正代表了要分析的业务结果。

让智能体创建或修改指标时，说明对业务有影响的规则即可，例如重要的纳入或排除条件、采用
哪个业务事件时间、单位是什么，以及指标有哪些使用限制。你不需要把这些规则转换成 Marivo
字段或 Python 代码。

例如：

> 为这个项目增加已完成订单收入指标。排除已取消和全额退款的订单，使用订单完成时间；
> 如果项目中的信息不足以确定某项业务规则，再向我确认。

智能体会通过 `marivo-semantic` 处理证据检查和编写流程。你只需要从业务角度检查它提出的
定义：衡量的结果是否正确，重要边界是否已经说明。含义不对时先纠正，再批准使用。通过技术
验证或就绪检查，并不代表业务定义已经获得批准。

## 智能体什么时候应该停下来确认

现有证据无法确定业务规则，或者两种合理定义会明显改变结果时，智能体应该向用户确认。
声明、预览和就绪检查等技术步骤，不需要用户逐项安排。

## Marivo 如何表示这份契约

```text
Datasource → Domain → Entity → dimensions / time dimension / measures → metric
```

Python 声明就是契约。装饰器函数体返回 Ibis 表达式，而不是原始 SQL 字符串。
需要用 SQL 与既有查询做一致性检查时，SQL 只能作为
`provenance=ms.from_sql(sql=..., dialect=...)` 元数据存在。

这个简化表示包含一个 `orders` 实体、一个 `region` 维度、一个
`order_date` 时间维度、一个 `amount` 度量，以及一个 `revenue` 聚合指标。
以下简洁声明用于说明对象关系；生产环境中的对象还需要包含下文所述、经过用户批准的
`ai_context`：

```python
import marivo.datasource as md
import marivo.semantic as ms

sales = ms.domain(name="sales", owner="Mina Zhang")
warehouse = ms.Ref.datasource("warehouse")
orders = ms.entity(name="orders", datasource=warehouse, source=md.table("orders"), primary_key=["order_id"])
region = ms.dimension_column(name="region", entity=orders, column="region")
order_date = ms.time_dimension_column(name="order_date", entity=orders, column="order_date", granularity="day", is_default=True)
amount = ms.measure_column(name="amount", entity=orders, column="amount", additivity="additive", unit="CNY")
revenue = ms.aggregate(name="revenue", measure=amount, agg="sum")
```

用户批准并加载模型后，日常分析先读取指标，再调用 `observe`：

```python
revenue_entry = catalog.require(ms.Ref.metric("sales.revenue"))
revenue_entry.details().show()
frame = session.observe(revenue_entry.ref, time_scope={...}, grain="month")
```

如果指标刚刚新建或修改，先完成下文所述的 scoped authoring closeout，再进入这条日常分析流程。

## 建模步骤

### 1. 连接数据源

Datasources 在 `models/datasources/*.py` 中声明，每种 backend 对应一个 typed helper。
helper 会注册连接。Semantic 文件通过精确的 `ms.Ref.datasource("warehouse")` factory 引用
datasource。

```python
# models/datasources/warehouse.py
import marivo.datasource as md
import marivo.semantic as ms

md.duckdb(
    name="warehouse",
    path="warehouse.duckdb",
    ai_context=ms.ai_context(
        business_definition="Local DuckDB warehouse for sales analysis.",
        guardrails=["Use only for development or approved local analysis."],
    ),
)
```

每个 helper（`md.duckdb`、`md.sqlite`、`md.mysql`、`md.postgres`、`md.trino`、`md.clickhouse`）都共享以下参数：

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | 全局 lowercase snake_case datasource 名称。供 `ms.Ref.datasource(name)` 使用。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |
| `extra` | `dict` | 否 | `None` | typed helper 未建模的、罕见的 JSON 安全 ibis 关键字参数。 |

各 backend 特有的参数：

| Helper | 必填 | 可选 |
| --- | --- | --- |
| `md.duckdb` | — | `path`（默认 `":memory:"`）、`read_only`（默认 `False`） |
| `md.sqlite` | — | `path`（默认 `":memory:"`）、`read_only`（默认 `False`）、`type_map` |
| `md.mysql` | `host`、`database` | `port`（3306）、`autocommit`、`user_env`、`password_env` |
| `md.postgres` | `host`、`database` | `port`（5432）、`schema`、`autocommit`、`user_env`、`password_env` |
| `md.trino` | `host`、`catalog` | `port`（8080）、`schema`、`source`、`timezone`、`http_scheme`、`client_tags`、`session_properties`、`user_env`、`auth_env` |
| `md.clickhouse` | `host` | `port`（9000 / 9440 secure）、`database`、`secure`、`settings`、`user_env`、`password_env` |

ClickHouse datasource 默认关闭 clickhouse-connect 自动生成的 session id，这样 analysis
缓存 backend 时，重复查询或并发查询不会共享同一个 ClickHouse server session lock。
如果 datasource 需要临时表等 ClickHouse session state，可以显式传
`extra={"autogenerate_session_id": True}`。`settings={...}` 用于 server/query
settings；Ibis/clickhouse-connect client keyword arguments 放在 `extra={...}`。

SQLite datasource 通过 `md.table(...)` 使用表和视图。`read_only=True` 会启用 SQLite
连接级 query-only 模式；当类型亲和推断不足时，可用 `type_map={...}` 把声明的 SQLite
类型名映射为 Ibis 类型字符串。Parquet、CSV 和 JSON descriptor 仍只属于 DuckDB。
当前 Ibis SQLite backend 不支持 median、percentile metric 或字符串 `strptime` 表达式；Marivo
会将其报告为结构化的 unsupported-operation 错误。此时应改用受支持的聚合与原生 temporal
列。

:::caution[凭据是引用，绝不是明文]
敏感字段以 `*_env` 引用的形式声明，指向环境变量 —— `user_env="TRINO_USER"`、
`password_env="MYSQL_PASSWORD"`。把 secret 明文写进 datasource 字段会报错。Marivo 通过
`os.environ` → `~/.marivo/secrets.toml` 解析 `*_env` 值，并在一次通过验证的
`md.test(...)` 往返之后把解析到的 secret 缓存到那里。
:::

```python
# models/datasources/lake.py
import marivo.datasource as md

md.trino(
    name="lake",
    host="trino.example.internal",
    catalog="hive",
    user_env="TRINO_USER",
    auth_env="TRINO_AUTH",
)
```

### 2. 声明数据域和实体

#### 数据域

`ms.domain(...)` 打开一个 namespace。每个 `_domain.py` 调用一次。它返回一个
`Ref[domain]`，也可以作为 `domain=` 传入，用来覆盖同级文件中对象的活动 domain。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Domain namespace，例如 `"sales"`。对象成为 `<name>.<object>`。 |
| `owner` | `str` | 是 | — | 负责人姓名，负责该 domain 的语义正确性与质量，例如 `"Mina Zhang"`。 |
| `default` | `bool` | 否 | `True` | 为 `True` 时，本文件中的 decorators 在未传 `domain=` 时解析到该 domain。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |

```python
import marivo.semantic as ms

ms.domain(name="sales", owner="Mina Zhang")
```

#### 实体

Entity 表示一个物理源（一张表或一个文件）及其 primary key。dimensions、measures 和 metrics
都依附在 entity 上。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Entity 名称。成为 `<domain>.<name>`。 |
| `datasource` | `Ref[datasource]` | 是 | — | `ms.Ref.datasource("warehouse")`。 |
| `source` | source builder | 是 | — | Datasource-owned `md.table(...)`、`md.parquet(...)`、`md.csv(...)` 或 `md.json(...)`。 |
| `primary_key` | `list[str]` | 否 | `None` | 构成 primary key 的列名。 |
| `versioning` | `ms.snapshot \| ms.validity` | 否 | `None` | Snapshot 或 SCD2 validity 版本控制（见下）。 |
| `domain` | `Ref[domain]` | 否 | 文件默认 | 覆盖活动 domain。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |

```python
warehouse = ms.Ref.datasource("warehouse")

orders = ms.entity(
    name="orders",
    datasource=warehouse,
    source=md.table("orders"),
    primary_key=["order_id"],
    ai_context=ms.ai_context(business_definition="One row per order."),
)
```

##### Source builders

| Builder | 必填 | 可选 | 用于 |
| --- | --- | --- | --- |
| `md.table(name)` | `name` | `database` | datasource 中的一张表（Trino/MySQL 用 `database="schema"`）。 |
| `md.parquet(path)` | `path` | `hive_partitioning` | 通过 DuckDB 读取的 self-describing Parquet file source。 |
| `md.csv(path, schema=...)` | `path`、typed `schema` | `header`、`delimiter` | 通过 DuckDB 读取的 CSV file source。 |
| `md.json(path, schema=...)` | `path`、typed `schema` | `format` | JSON file source、glob 或 DuckDB httpfs URL。 |

### 3. 添加字段

#### 维度

维度是用于分组或过滤的分类属性。直接物理列使用 `ms.dimension_column(...)`；需要表达式
（例如 `table.region.upper()`）时，使用 `@ms.dimension` decorator，其 body 返回一个针对
entity 表的 ibis expression。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Dimension 名称。成为 `<domain>.<entity>.<name>`。 |
| `entity` | `Ref[entity]` | 是 | — | 所属 entity。 |
| `column` | `str` | 是 | — | Entity 表上的物理列名。 |
| `domain` | `Ref[domain]` | 否 | 文件默认 | 覆盖活动 domain。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |

```python
region = ms.dimension_column(
    name="region",
    entity=orders,
    column="region",
    ai_context=ms.ai_context(business_definition="Sales reporting region."),
)
```

#### 时间维度

时间维度是携带 grain 和解析元数据的特殊 dimension。**只有 time dimension 才能作为
`session.observe` 的时间轴。**

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Dimension 名称。 |
| `entity` | `Ref[entity]` | 是 | — | 所属 entity。 |
| `column` | `str` | 是 | — | Entity 表上的物理列名。 |
| `granularity` | grain 字面量 | 是 | — | `year`、`quarter`、`month`、`week`、`day`、`hour`、`minute` 或 `second` —— 查询有意义的最细粒度。 |
| `parse` | parse 变体 | 否 | `None` | 源列如何变成时间值（见下）。原生时间列可省略——分析时自动推断 parse 变体。 |
| `is_default` | `bool` | 否 | `False` | 当 entity 有多个时间轴时，标记默认时间轴。省略 `time_dimension=` 时 `observe` 会使用它。 |
| `domain` | `Ref[domain]` | 否 | 文件默认 | 覆盖活动 domain。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |

##### Parse 变体

`parse=` 值声明列的物理编码。省略时，分析会根据列的 ibis dtype 自动推断 parse 变体（原生
`date`、`datetime`、`timestamp` 列无需显式指定 parse）。对于 string 或 integer 列，需提供
`ms.strptime(format)` 或 `ms.hour_prefix(prefix)`。该变体必须与 `granularity` 兼容（例如
hour 粒度需要带时间的格式）。

| Builder | 源列是… | 关键参数 |
| --- | --- | --- |
| *（省略 `parse`）* | 原生时间列 | — |
| `ms.datetime()` | 原生 `datetime` | `timezone`（IANA）、`sample_interval` |
| `ms.timestamp()` | 原生 `timestamp` | `timezone`（IANA）、`sample_interval` |
| `ms.strptime(format)` | 需要解析的 string/integer | `timezone`、`sample_interval` |
| `ms.hour_prefix(prefix)` | 仅含小时的分区 | `sample_interval` —— `prefix` 是提供日期的 day 粒度 `Ref[time_dimension]` |

对于原生但 naive 的 `datetime` 或 `timestamp`，必须声明其 source timezone（例如
`ms.datetime(timezone="Asia/Shanghai")`）。显式 `ms.datetime()` 或
`ms.timestamp()` 未提供 `timezone=` 时，readiness 会返回
`undeclared_naive_time_axis` blocker：运行时会回退到 datasource read timezone，而
analysis window 使用 report timezone，可能在日或小时边界错位。零查询 gate 会报告这些
运行时解析的上下文，但绝不猜测业务 timezone。形如 `(5, "minute")` 的
`sample_interval` 标记一个被周期采样的时间轴，供 semi-additive fold 使用。

对于 time-series 和 panel frame，`bucket_start` 始终是 report timezone 下的 bucket
标签，并以 timezone-naive timestamp 或 date 输出。例如 `report_timezone="Asia/Shanghai"`
时，上海业务日的第一个小时桶显示为 `2026-06-20 00:00:00`，而不是等价的 UTC instant。

```python
# Day partition stored as the string "20260131"
log_date = ms.time_dimension_column(
    name="log_date",
    entity=orders,
    column="dt",
    granularity="day",
    parse=ms.strptime("%Y%m%d"),
    is_default=True,
)

# Native UTC timestamp, usable for sub-day buckets
event_ts = ms.time_dimension_column(
    name="event_ts",
    entity=orders,
    column="event_ts",
    granularity="minute",
    parse=ms.timestamp(timezone="UTC"),
)
```

#### 度量

Measure 是你打算聚合的行级数量事实（例如金额或数量）。直接物理列使用
`ms.measure_column(...)`；需要基于一个或多个列的表达式时使用 `@ms.measure`。Measure
携带 additivity 和可选的 unit。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Measure 名称。成为 `<domain>.<entity>.<name>`。 |
| `entity` | `Ref[entity]` | 是 | — | 所属 entity。 |
| `column` | `str` | 是 | — | Entity 表上的物理列名。 |
| `additivity` | additivity 值 | 是 | — | `"additive"`、`"non_additive"` 或 `ms.semi_additive(...)`。 |
| `unit` | `str` | 否 | `None` | UCUM unit token：`"USD"`、`"CNY"`、`"%"`、`"ms"`、`"{order}"`。 |
| `domain` | `Ref[domain]` | 否 | 文件默认 | 覆盖活动 domain。 |
| `ai_context` | `AiContextValue` | 否 | `None` | 面向智能体的上下文，通过 `ms.ai_context(...)` 构造。 |

```python
amount = ms.measure_column(
    name="amount",
    entity=orders,
    column="amount",
    additivity="additive",
    unit="CNY",
)
```

### 4. 定义指标


metric 不是一次 query 的结果，而是智能体可以安全 observe 的、可用于分析的业务数值。
它的定义和计算声明在版本控制中；entity、dimensions、时间行为、unit 和 additivity 说明它如何
被分析；`ai_context` 说明业务含义和 guardrails；readiness 用于在新建或修改后认证该定义，
再声明变更已经完成。

#### 选择适合任务的指标构建器

| 需要 | 使用 | 示例 |
| --- | --- | --- |
| 需要按行级数值汇总时 | `ms.aggregate` | 从 `amount` 得到 revenue |
| 需要计数时 | `ms.count` | 已完成订单数 |
| 需要按行级 measures 计算加权均值时 | `ms.weighted_mean` | 按请求数加权的延迟 |
| 需要由已有指标计算时 | `ms.ratio`、`ms.linear` | 转化率 |
| 需要累计时 | `ms.cumulative` | 年初至今收入 |
| 现有构建器无法表达时 | `@ms.metric` | 一个未被上述形式覆盖的计算 |

按需要选择对应的构建器。`@ms.metric` 仅用于现有构建器无法表达的计算。

#### 来自 measure 的简单 metric —— `ms.aggregate`

聚合一个 measure。无需 body；additivity 从 measure 继承。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Metric 名称。 |
| `measure` | `Ref[measure]` | 是 | — | 要聚合的 measure。 |
| `agg` | 聚合方式 | 是 | — | `"sum"`、`"mean"`、`"min"`、`"max"` 等。 |
| `fold` | fold | 否 | `None` | semi-additive measure 的 time-fold 覆盖。 |
| `filter` | `WhereFilter` | 否 | `None` | 由 `ms.where(col=value, ...)` 构建的 AND 等值过滤，仅聚合匹配行。 |
| `unit` | `str` | 否 | 继承 | 覆盖从 measure 推导的 unit。 |
| `domain` / `ai_context` | — | 否 | — | 同其他对象。 |

```python
revenue = ms.aggregate(name="revenue", measure=amount, agg="sum")
us_revenue = ms.aggregate(name="us_revenue", measure=amount, agg="sum", filter=ms.where(region="US"))
```

#### Count metric —— `ms.count`

需要统计 entity 行数时使用 `ms.count`，不要为了计行数额外声明冗余 measure。这个 helper
只接受 ref，并会从 entity ref 推断 metric 所属 domain。传入 `filter=ms.where(...)` 可对子集
计数（如失败数），无需手写 metric body。

```python
order_count = ms.count(name="order_count", entity=orders, ai_context=ms.ai_context(business_definition="订单总数。"))
failed_count = ms.count(name="failed_count", entity=orders, filter=ms.where(state="FAILED"))
```

`ms.where(**conditions)` 构建一个 AND 等值过滤（若干 `col=value`，value 为
`str` / `int` / `float` / `bool`）。作为 `ms.count` 或 `ms.aggregate` 的 `filter=`
入参，用于失败率、错误率、成功率等子集指标。

#### 加权均值 —— `ms.weighted_mean`

`ms.weighted_mean` 接受同一 entity 上的两个 measures，由 Marivo 自动完成逐行乘权和聚合：

```python
avg_latency = ms.weighted_mean(
    name="avg_latency",
    value=latency,
    weight=request_count,
)
```

它只对 value 与 weight 都非 null 的行计算
`SUM(value * weight) / NULLIF(SUM(weight), 0)`。weight 必须是 additive measure；
结果继承 value measure 的 unit，并持久化精确的 `numerator` / `weight` components，
供 `weighted_mix` 归因使用。它也接受 `filter`、`unit`、`domain` 和 `ai_context`。

#### 派生 metrics —— `ms.ratio` / `ms.linear`

派生 metric 由其他 metrics 组合而成，不需要 body；计算完全来自组成成分。

| Builder | 必填 | 计算 |
| --- | --- | --- |
| `ms.ratio(name, numerator, denominator)` | 两个 ref | `numerator / denominator`（如客单价、各类比率） |
| `ms.linear(name, add, subtract)` | `add`（共 ≥2 项） | `add` 之和减去 `subtract`（如 `net = gross - refunds`） |

它们也都接受 `unit`、`domain` 和 `ai_context`。

```python
net_revenue = ms.linear(name="net_revenue", add=[gross_revenue], subtract=[refunds])
aov = ms.ratio(name="aov", numerator=total_amount, denominator=orders_count)
```

观察（observe）时，除法按固定的 `zero_division="null"` 策略求值：某个 bucket 的分母
（`ms.ratio`）或成对权重和（`ms.weighted_mean`）存在但为零时，结果为 null——绝不会出现
`+/-inf`——受影响的行数记录在观察结果的 `meta.zero_denominator_rows` 与
`quality_summary.zero_denominator_rows` 中。

#### 累计 metrics —— `ms.cumulative`

当业务问题是"截至 bucket t 累计了多少"时使用 `ms.cumulative(...)`。
base 必须是使用 `sum`、`count`、`count_distinct` 或加权组件累计的 tier-1
`ms.aggregate(...)`、`ms.count(...)` 或 `ms.weighted_mean(...)` metric。
`anchor` 参数选择累计形态。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | 语义 metric 名称。 |
| `base` | `Ref[metric]` | 是 | — | 使用 `sum`、`count` 或 `count_distinct` 的 tier-1 简单聚合 metric。 |
| `over` | `Ref[time_dimension] \| None` | 否 | `None` | 累计时间轴。除非 base root entity 只有一个时间维度，否则需显式传入。 |
| `anchor` | `GrainToDate \| Trailing \| None` | 否 | `None` | 累计锚点。`None` = 全部历史累计；`ms.grain_to_date(...)` = MTD/QTD/YTD 重置；`ms.trailing(...)` = 滚动 N。 |
| `unit` / `domain` / `ai_context` | — | 否 | — | 同其他 metric。 |

```python
user_id = ms.measure_column(name="user_id", entity=events, column="user_id", additivity="non_additive")
active_users = ms.aggregate(name="active_users", measure=user_id, agg="count_distinct")
cumulative_active_users = ms.cumulative(
    name="cumulative_active_users",
    base=active_users,
    over=event_time,
)
```

对于 `count_distinct` base metric，累计值使用首次出现语义：每个去重实体在最早出现的 bucket
中计入，因此累计值单调非递减。累计 metric 可作为 ratio 组成成分——用 `ms.ratio(...)` 组合累计
分子和分母可得到累计比率。

##### 累计锚点

`anchor=None`（默认）是全部历史累计：observe 窗口裁剪展示行，但不会重置累计值。两个值对象
构造器开启额外的锚点类型。

`ms.grain_to_date(grain=...)` 在每个重置粒度边界重置累计值（MTD / QTD / YTD / WTD）。在一个
重置周期内值持续累计；到达边界时回落到该周期首个 bucket 的流量。`grain` 取值为 `week`、
`month`、`quarter`、`year` 之一。

```python
mtd_revenue = ms.cumulative(
    name="mtd_revenue",
    base=revenue,
    over=event_time,
    anchor=ms.grain_to_date(grain="month"),
)
```

`ms.trailing(count=..., unit=...)` 是固定大小的滚动窗口：每个 bucket 的值是该 bucket 结束边界
为止的 span 上的 base 聚合。空窗口为真零，不会向前 carry。部分窗口（span 触及数据起点之前）
展示实际的部分累计值，并在 coverage 中标记为 `partial`。`unit` 仅接受固定大小单位（`second`、
`minute`、`hour`、`day`、`week`）；日历可变单位（`month`、`quarter`、`year`）会被拒绝并给出
教学错误。

```python
rolling7_active = ms.cumulative(
    name="rolling7_active",
    base=active_users,
    over=event_time,
    anchor=ms.trailing(count=7, unit="day"),
)
```

两条跨锚点规则约束查询时的粒度选择：

- **粒度兼容规则**（`grain_to_date`）：每个展示 bucket 必须完整落入一个重置周期内。在
  `month` / `quarter` / `year` 重置下使用 `week` 查询粒度是非法的（week bucket 会跨越月边界）；
  `day` 和 `hour` 合法。
- **整数倍规则**（`trailing`）：窗口 span 必须是查询粒度的整数倍（`W_buckets = span / grain`）。
  Trailing 要求时间粒度。

Derived metric 可以组合 cumulative component。只有所有外层 component 都是 cumulative，且共享
完全相同的 `trailing` 或 `grain_to_date` 锚点时才支持 compare。`all_history`、混合锚点以及累计与
非累计 component 混合都会被拒绝；`attribute`、`decompose` 和 `forecast` 仍不支持。

## 使用并检查模型

### 智能体使用 metric 前先读取它

~~~python
import marivo.analysis as mv
import marivo.semantic as ms

session = mv.session.get_or_create(
    name="revenue-investigation",
    question="Why did Q4 revenue drop?",
)
catalog = session.catalog
revenue_entry = catalog.require(ms.Ref.metric("sales.revenue"))
region_entry = catalog.require(ms.Ref.dimension("sales.orders.region"))

revenue_entry.details().show()
current = session.observe(revenue_entry.ref, grain="month", dimensions=[region_entry.ref])
~~~

`details().show()` 展示人类编写的定义、guardrails、composition、effective entities、候选
普通/时间维度与 measure lineage。候选轴来自 metric 的 effective entities；跨实体 relationship
与 fanout 仍由 `session.observe(...)` 验证。只有在读取 verify/preview/readiness 等机械下一步时
才调用 `.contract().show()`；`.show()` 打印的内容与 `.render()` 返回的 bounded card 相同。
如果 object 缺失或定义存在争议，先回到 semantic authoring；否则该 observation 会成为 session
中有 evidence 支撑的调查的第一个 artifact。新建或修改的 object 把 scoped readiness 作为
authoring closeout 的一部分。

### 加载模型并认证变更

声明完成后，加载 catalog 并检查它：

```python
import marivo.semantic as ms

catalog = ms.load()                       # SemanticCatalog
catalog.domains.show()                    # top-level domains
catalog.metrics.show()                    # just metrics across all domains

sales = catalog.domains.get("sales")
orders = sales.entities.get("orders")
revenue = orders.metrics.get("revenue")   # one metric object
region = orders.dimensions.get("region")  # one dimension object
```

新建或修改 object 后，运行**就绪检查**，认证所选定义及其 scoped runtime evidence 仍然有效：

```python
report = catalog.readiness(refs=[revenue.ref, region.ref])
if report.status == "blocked":
    report.show()                         # blockers, with the next step for each
```

:::note[Readiness 是显式认证]
project 可以在 readiness 仍被阻塞时成功 `load()`，因为加载和认证回答不同问题。Analysis APIs
会加载、解析、规划并在运行时检查输入，但不会自动调用 readiness。Blocked report 需要在声明
semantic change 完成之前修复；日常分析复用未变化的 objects 时，不必在每次调用前重跑
readiness。缺失或不匹配的 runtime preview certification 会显示为 warning 而不是 blocker：
日常分析先披露再继续，authoring closeout 前完成修复。
:::

另外两个检查支持 authoring：

- `ms.richness()` —— 建议性的覆盖度/深度报告；永不阻塞。
- `ms.parity_check("sales.revenue")` —— 用 metric 的 `provenance` SQL 运行并比对结果。
  需要 `provenance=ms.from_sql(...)`。

就绪检查如何判断“是否就绪”，见 [就绪检查](/zh-cn/latest/concepts/readiness/)。
分析如何记录结论，见 [证据链](/zh-cn/latest/concepts/evidence/)。
关于 session 如何加载并验证 metric 输入，继续阅读
[分析流程](/zh-cn/latest/concepts/analysis-workflow/)；关于变更认证，见上面的 scoped readiness。
## 高级建模

### 语义契约参考


日常 authoring 会用到两个 namespace：

```python
import marivo.datasource as md   # connections 与 physical sources
import marivo.semantic as ms     # meaning (ms.entity, ms.metric, ...)
```

#### 命名规则

每个对象都归属于一个 **domain**，并通过限定 ref 引用：

- Domain 级对象：`<domain>.<object>` —— 例如 `sales.revenue`、`sales.orders`。
- Entity 级对象（dimensions 和 measures）：`<domain>.<entity>.<field>` —— 例如
  `sales.orders.region`。

一个 project 由一组声明文件组成。Datasources 在 `models/datasources/` 下声明一次；
semantics 放在 `models/semantic/<domain>/` 下，每个 domain 有一个 `_domain.py`：

```text
your-project/
  marivo.toml
  models/
    datasources/
      warehouse.py          # md.duckdb(name="warehouse", ...)
    semantic/
      sales/
        _domain.py          # ms.domain(name="sales", owner="Mina Zhang") + entities, metrics, ...
```

在多仓库语义层场景中，每个业务域仓库仍保持相同的 authored `models/` 布局，
中央分析 project 只需要引用这些 root：

```toml
[semantic]
layer_paths = ["../finance-domain/models"]
```

Marivo 会先加载中央 project 的 `models/`，再按配置顺序加载外部 `models/` root。
所有 datasource、domain 与 semantic object 会进入同一个 catalog，因此名称必须全局唯一。
`marivo doctor`、`catalog.preview`、`catalog.verify` 与 analysis session 都会按同一组
配置 roots 解析 datasource-backed semantic object。

#### Semantic refs

每个 semantic 对象都由一个 sealed、不可变的 `ms.Ref` 值标识。运行时只有一个
具体 `Ref` 类；泛型 kind（如 `Ref[metric]`、`Ref[dimension]`）负责静态角色检查。
只能通过 `ms.Ref.metric("sales.revenue")` 这类精确 factory，或返回同一值类型的
authoring declaration 创建 ref。

所有 ref 都有以下只读属性：

| 属性 | 类型 | 含义 |
| --- | --- | --- |
| `.path` | `str` | kind 内的限定路径，例如 `"sales.revenue"`。 |
| `.kind` | `SemanticKind` | 对象种类——以下八个值之一。 |
| `.key` | `str` | 规范 runtime key，例如 `"metric:sales.revenue"`。 |
| `.name` | `str` | 最后的本地名称 segment。 |

`str(ref)` 返回 `.key`，相等性和哈希基于 `(kind, path)`。不能直接调用
`Ref(...)`，也不能继承 `Ref`。公开边界不会把字符串或 catalog entry 当作 ref。

| Kind | Factory | 常见 authoring producer | 可调用? |
| --- | --- | --- | --- |
| `domain` | `ms.Ref.domain(path)` | `ms.domain(...)` | 否 |
| `datasource` | `ms.Ref.datasource(path)` | datasource declaration 的 `.ref` | 否 |
| `entity` | `ms.Ref.entity(path)` | `ms.entity(...)` | 否 |
| `dimension` | `ms.Ref.dimension(path)` | `ms.dimension_column(...)`、`@ms.dimension` | 仅在绑定的 expression body 内可调用 |
| `time_dimension` | `ms.Ref.time_dimension(path)` | `ms.time_dimension_column(...)`、`@ms.time_dimension` | 仅在绑定的 expression body 内可调用 |
| `measure` | `ms.Ref.measure(path)` | `ms.measure_column(...)`、`@ms.measure` | 仅在绑定的 expression body 内可调用 |
| `metric` | `ms.Ref.metric(path)` | `ms.aggregate(...)`、`@ms.metric`、`ms.ratio(...)` 等 | 否 |
| `relationship` | `ms.Ref.relationship(path)` | `ms.relationship(...)` | 否 |

在 expression body 内，field ref 必须显式传入已声明的 entity alias，例如
`amount(orders)`。Loader 会把该绑定记录到 compiled catalog。若在 body 外调用、使用错误
alias，或调用非 field ref，都会得到带修复提示的错误。

Authoring ref 可以直接传给分析。如果从字面身份开始，先做一次精确 catalog membership
校验，再使用 entry 的 ref：

```python
revenue = ms.aggregate(name="revenue", measure=amount, agg="sum")
frame = session.observe(revenue, time_scope={...})

revenue_entry = catalog.require(ms.Ref.metric("sales.revenue"))
frame = session.observe(revenue_entry.ref, time_scope={...})
```

`catalog.require(ref)` 是精确的全局查找；catalog collections 负责浏览和本地名称查找，
analysis 与 authoring API 始终只接受 `Ref` 作为 semantic identity。持久化 payload 使用带版本的
`{schema, kind, path}` 记录，裸字符串不是公开 interchange 格式。

#### `ai_context`：人与智能体之间的契约

每个 semantic object 和 datasource 都接受可选的 `ai_context` 参数，并通过 `ms.ai_context(...)`
构造。业务含义和 guardrails 放在这里；智能体使用对象前会读取这部分上下文。所有参数都是可选的，
但未知的关键字参数会在调用时被 Python 拒绝。

| 字段 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `business_definition` | `str` | 否 | `None` | 用一两句话说明对象的业务含义。 |
| `guardrails` | `list[str]` | 否 | `[]` | 智能体必须遵守的规则：必要的过滤、排除项、范围限制。 |

```python
ai_context=ms.ai_context(
    business_definition="Gross order amount before refunds.",
    guardrails=["Validate refund exclusions before using as net revenue."],
)
```

### Versioning（可选）

对于行会随时间变化的 entity，需要声明如何读取当前状态：

- `ms.snapshot(partition_field, grain="day", timezone=None, format=None)` ——
  按天分区的快照；`partition_field` 必须是 `Ref[dimension]` 或 `Ref[time_dimension]`。
- `ms.validity(valid_from, valid_to, interval, open_end, timezone=None)` ——
  SCD2 validity 区间；`valid_from` 和 `valid_to` 必须是 `Ref[dimension]` 或 `Ref[time_dimension]`。`interval` 为 `"closed_open"`（`[from, to)`）或
  `"closed_closed"`；`open_end` 列出表示“仍然有效”的哨兵值（例如 SQL `NULL` 用
  `(None,)`，或 `("9999-12-31",)`）。

### Custom Ibis metric —— `@ms.metric`

当 metric 无法用 `ms.aggregate(...)`、`ms.count(...)` 或派生 metric builder 表达时，
才使用这个 escape hatch。Body 返回一个 ibis 聚合；`additivity` 由你显式声明。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 否 | 函数名 | Metric 名称。 |
| `entities` | `list[Ref[entity]]` | 是 | — | Body 读取的 entities。 |
| `additivity` | additivity 值 | 是 | — | `"additive"`、`"non_additive"` 或 `ms.semi_additive(...)`。 |
| `root_entity` | `Ref[entity]` | 否 | 单个 entity | 当 `entities` 多于一个时必填。 |
| `fanout_policy` | `"block" \| "aggregate_then_join"` | 否 | `"block"` | 如何处理跨 entity 连接的 fan-out。 |
| `unit` | `str` | 否 | `None` | UCUM unit token。 |
| `provenance` | `SqlProvenance` | 否 | `None` | 用于 parity 校验的 `ms.from_sql(sql=..., dialect=...)`。 |
| `domain` / `ai_context` | — | 否 | — | 同其他对象。 |

```python
@ms.metric(
    entities=[orders],
    additivity="additive",
    name="revenue",
    provenance=ms.from_sql(
        sql="SELECT SUM(amount) AS revenue FROM orders",
        dialect="duckdb",
    ),
    ai_context=ms.ai_context(business_definition="Gross order amount before refunds."),
)
def revenue(table):
    return table.amount.sum()
```

### 分析正确性的约束

Additivity 与时间行为决定智能体更改分析粒度或增加切分时，数值是否仍有意义。SQL provenance
不会执行 metric；它为 `ms.parity_check(...)` 提供参考 query，以检查已声明的语义是否与既有实现一致。

- `ms.semi_additive(over, fold)` —— 用于在大多数轴上可加、但需要在某个时间轴上折叠的
  snapshot/status 事实。`over` 必须是 `@ms.time_dimension(...)` 返回的状态时间
  dimension ref；`fold` 为 `"last"`、`"first"`、`"mean"`、`"max"` 或
  `("percentile", 0.95)`。
- `ms.from_sql(sql, dialect)` —— 把 SQL 作为**仅 provenance** 附加，用于启用
  `ms.parity_check(...)`。它永远不会作为 metric body 执行。

```python
snapshot_date = ms.time_dimension_column(
    name="snapshot_date",
    entity=inventory_daily,
    column="snapshot_date",
    granularity="day",
)

on_hand_units = ms.measure_column(
    name="on_hand_units",
    entity=inventory_daily,
    column="on_hand_units",
    additivity=ms.semi_additive(over=snapshot_date, fold="last"),
)
```

### 关系

Relationship 声明两个 entity 如何 join，使 metrics 和 dimensions 能够跨越这两个 entity。
Keys 使用 dimension ref，而不是原始列名。

| 参数 | 类型 | 必填 | 默认 | 含义 |
| --- | --- | --- | --- | --- |
| `name` | `str` | 是 | — | Relationship 名称。 |
| `from_entity` | `Ref[entity]` | 是 | — | 源 entity。 |
| `to_entity` | `Ref[entity]` | 是 | — | 目标 entity。 |
| `keys` | `list[JoinKey]` | 是 | — | 一个或多个 `ms.join_on(from_key, to_key)` 配对。 |
| `domain` / `ai_context` | — | 否 | — | 同其他对象。 |

```python
ms.relationship(
    name="orders_to_customers",
    from_entity=orders,
    to_entity=customers,
    keys=[ms.join_on(order_customer_id, customer_id)],
)
```

### 先 inspect，只 sample 一次，再 author

对 datasource-backed 语义对象，使用一个显式 evidence snapshot。查阅
`md.help("authoring")` 了解 datasource inspection 与 scope，查阅
`ms.help("authoring")` 了解 semantic 依赖阶梯，查阅 `md.help("ai_context")` 了解共享的
`ms.ai_context(...)` 契约：

```python
warehouse = ms.Ref.datasource("warehouse")
orders = md.table("orders")

inspection = md.inspect(warehouse, orders)
inspection.show()
inspection.partitions().show()
scope = md.partition({"dt": "20260710"}, max_rows=1000, timeout_seconds=30)
snapshot = inspection.sample(
    scope=scope,
    columns=("order_id", "region", "created_at", "amount"),
)
snapshot.entity(columns=("order_id",)).show()
snapshot.dimensions(columns=("region",)).show()
snapshot.time_dimensions(columns=("created_at",)).show()
snapshot.measures(columns=("amount",)).show()
```

`md.duckdb(...)` 声明 datasource。`md.table(...)` 选择该 datasource 内部的
table 或 view。`md.parquet(...)`、`md.csv(...)` 和 `md.json(...)` 是配合
datasource ref 使用的 DuckDB file source descriptor；它们不是 datasource
declaration。

```python
events = md.json("data/events/*.json", schema={"event_id": "string"}, format="newline_delimited")
orders_parquet = md.parquet("data/orders/*.parquet")
orders_csv = md.csv("data/orders/*.csv", schema={"order_id": "string"})

md.inspect(warehouse, events).show()
```

Inspection 是 metadata-only，并在 sampling 前报告 schema、physical extent、partition
state 和 enforceable capabilities。Scope 必须有正整数 row/timeout guards 和明确 columns；
`LIMIT` 不保证 bytes scanned。Snapshot projections 不查询 datasource，values 默认只在内存。
只有接受有界 value evidence 明文写入 project-local cache 时才使用 `persist_values=True`。
Uncommon formats、keys、timezone、aggregation、unit、additivity、relationship cardinality
和 business meaning 仍由智能体负责。

写完一个 Python declaration 后，重新加载 catalog entry，取出 `entry.ref`，依次运行
`catalog.verify(ref)`、`catalog.preview(ref, using=snapshot)` 和零查询
`catalog.readiness(refs=[ref])`。

当 readiness 报告多个 runtime preview 缺失时，仍通过同一个入口修复：
`catalog.preview_many(report.preview_required_refs, using=snapshot_or_mapping)`。
它会合并兼容查询，但仍为每个 ref 保留独立 evidence；这不是 batch authoring 捷径。
