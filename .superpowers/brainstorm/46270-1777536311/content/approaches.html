Three Datasource Schema Approaches

How to structure the merged datasource object

A

Minimal Flat

Only fields with active runtime consumers. No nested objects except connection and policy. Capabilities fully derived.

Pros

  • Simplest to implement and reason about
  • No dead fields
  • Clear mental model

Cons

  • Less room for future growth
  • Policy may get crowded if more flags added
B

Grouped with Sections

Keep connection, sync, policy as nested objects. Each section has clear responsibility. allow_identity_reuse sits in policy.

Pros

  • Logical grouping aids readability
  • Follows current source contract shape
  • Easy to extend each section independently

Cons

  • Slightly more complex API
  • Still needs discipline to avoid section creep
C

Connection-Centric

Everything revolves around the connection. Identity, sync, and policy are all sub-properties of how we connect. allow_identity_reuse is part of connection identity config.

Pros

  • Intuitive for operators — one object = one connection
  • Identity reuse naturally lives near connection

Cons

  • Overloads connection with non-connection concerns
  • Sync policy is not a connection property
  • Harder to evolve independently
